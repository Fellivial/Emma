## 1. Gap Analysis Report

### 1.1 Executive Summary

This Gap Analysis compares Emma's current Brain Gateway architecture — a mature, structurally clean single-provider inference boundary surrounded by a fragmented Application Layer, per Phase 1's central finding — against the long-term architecture capabilities the roadmap and ADR-0003 name as the destination: a fully provider-agnostic system capable of evolving from OpenRouter-based development to hybrid and self-hosted inference, with capability-based routing, orchestration, and operational visibility, "without requiring major application refactoring" (roadmap, Objective).

The comparison surfaces two structurally different kinds of gap, and this document is organized to keep them distinct rather than blend them into one severity scale:

**Gaps of absence** — capabilities the target architecture requires that do not exist at all today, with no partial implementation to build on or be misled by. This category covers capability-based routing (§7), a Capability Registry or Model Registry beyond the static `models.ts` file, orchestration across multiple concurrently-available providers, and nearly the entirety of operational architecture — tracing, metrics, structured logging, diagnostics (§11). For these, Phase 1 already established there is nothing to extend; this phase's contribution is stating what the absence means measured against the roadmap's own named target capabilities, not re-discovering the absence itself.

**Gaps of unverified or fragmented capability** — mechanisms that exist, look structurally sound, and are confidently described in governing documents, but whose correctness under the target architecture's actual demands has never been exercised. The largest single gap in this category is the `BrainProvider` abstraction itself (§6): it is minimal, interface-based, and procedurally well-designed, but it has exactly one implementation, and Phase 1 confirmed no test, mock, or second implementation anywhere in the repository exercises its polymorphism. A target architecture whose central premise is "provider independence" is being judged, today, against an abstraction that has never actually abstracted anything in practice. The second-largest gap in this category is not a single missing thing but a repeating shape Phase 1 already named as the Application Layer's dominant pattern: one documented owner, one or more undocumented competitors, recurring across prompt construction, behavior derivation, context management, and rate/usage limiting (§3, §8, §10).

A third, smaller category is **gaps of governance**, not architecture: ADR-0003's own header still asserts "Implementation: None yet," a drift between an ADR and the code it governs that this same roadmap now treats as required input (§15).

No capability was found to be actively broken. Every gap identified in this document is either an absence (nothing exists to compare), an unverified assumption (something exists but has been proven only once), or a fragmentation (something exists in multiple, uncoordinated forms). This mirrors Phase 1's own overall finding — no architectural decision found to be actively wrong, only decisions incomplete, unenforced, or proven only once — extended here specifically against the roadmap's stated target capabilities rather than against general architectural quality.

### 1.2 Methodology

This analysis was performed by (1) reading the roadmap's Phase 3 scope list (`roadmaps/brain-gateway-roadmap-v1.md`, "Phase 3 — Architecture Discovery" section) as the most concrete available statement of desired long-term capabilities — system boundaries, Brain Gateway responsibilities, provider abstraction, Capability Registry, Model Registry, Routing Engine, Context/Memory/Behavior/Emotion/Prompt Pipelines, Response Validation, Retry, Fallback, Analytics, Cost Tracking, Configuration, Extension Model — since the roadmap does not separately enumerate "desired capabilities" outside of what Phase 3 is tasked to design; (2) cross-referencing ADR-0003's stated principles (Single Inference Entry Point, Provider Agnostic, Separation of Concerns, Stable Contracts, Extensibility, Incremental Evolution) and its "Out of Scope" / "Future Providers" sections as the second source of desired-capability language; (3) treating every current-state fact and quality judgment in Phase 0 and Phase 1 as established, cited by section reference rather than re-verified; and (4) performing direct repository inspection only for gap-specific questions Phase 0/1 were not scoped to answer — specifically, repo-wide searches for capability-registry or router code, mock/second `BrainProvider` implementations in `tests/`, and feature-flag or observability tooling in `package.json` (none of the four were found; see §7, §11, §14 for the searches and their results).

Per the phase's explicit non-goals, every gap below is stated as Current State → Desired Capability → Gap Description → Evidence. No gap description proposes a mechanism, technology, or implementation to close it.

### 1.3 Scope

Per the Phase 2 task brief: capability, responsibility, boundary, dependency, provider-abstraction, routing, context, memory, prompt, operational, error-handling, configuration, extension-readiness, and architectural-consistency gaps, plus a prioritized strategic summary. Addressed as §2–§16 below.

### 1.4 Current Baseline

The current architecture, as established by Phase 0 (facts) and Phase 1 (quality judgment), in brief: a 551-line, 3-file Brain Gateway (`src/core/brain/`) sits between 14 non-test source files (16 invocation sites) and a single OpenRouter provider implementation, with total, grep-confirmed provider-wire isolation (Phase 1 §1.4.4). It already carries streaming, tool-calling, vision, and embeddings through the same three exported functions (`brainChat`, `brainChatStream`, `brainEmbed`), and its two hardest-sounding future capabilities — tool execution and agent workflows — are not future work at all; they are shipped and already proven against the Gateway's real contract via `agent-loop.ts` (Phase 1 §1.4.13). Above the Gateway, the Application Layer exhibits the "one named owner, one undocumented competitor" pattern across prompt construction (6 owners), behavior derivation (3 call sites), context management (2 uncoordinated pipelines), and rate/usage limiting (2 independent implementations each) (Phase 1 §1.1, §7.1). Operationally, the system has no tracing, no metrics, and sparse (3-call-site) Sentry coverage across the entire AI pipeline (Phase 1 §1.4.14). This baseline is not re-derived here; it is the fixed input this document compares against the roadmap's stated destination.

## 4. Boundary Gap Assessment

Extends Phase 1 §3 (Boundary Assessment, which rated existing boundaries on well-defined/enforceable/consistent/future-proof) with the question this phase adds: which of those boundaries, as currently constituted, are sufficient for the boundaries the target architecture (roadmap Phase 3: "System boundaries," "Brain Gateway responsibilities") will need.

- **Application ↔ Brain Gateway ↔ Provider (the three-tier ADR-0003 split).** Phase 1 rated this boundary "genuinely future-proof" — the one boundary in the system that already absorbed streaming and tool-calling without a shape change (Phase 1 §3). **Gap: none identified.** This is the one boundary this document found requires no further architectural work to serve as the target architecture's foundation, based on the evidence available.
- **Gateway ↔ Provider, specifically for a _second_ provider.** Well-defined and enforceable for the one provider that exists (`providers/openrouter.ts` is importable only from `gateway.ts`, Phase 0 §7). **Gap:** future-proofness is explicitly rated "empirically unproven with a second provider" by Phase 1 (§3) — the boundary's _shape_ is sound, but a boundary that has only ever had one thing on one side of it has not yet been asked to enforce isolation _between_ two things on that side, which is a materially different (and currently untested) enforcement question.
- **Behavior ↔ Prompt Builder.** "Yes on paper... no mechanism — holds by convention only" (Phase 1 §3). **Gap:** a target Behavior Pipeline (roadmap Phase 3 scope) inherits a boundary that is real today only because no violation has yet occurred, not because anything would catch one. This is a boundary-_enforcement_ gap distinct from the boundary being wrongly drawn.
- **Prompt construction (single-owner claim).** "No — documented as one owner, actually six" (Phase 1 §3). **Gap:** a target Prompt Pipeline has no boundary to inherit at all in the sense the roadmap's Phase 3 scope implies (a single pipeline with a defined edge) — it would need to establish a boundary around a responsibility that currently has none.
- **Validation (sanitise / validator / request-validation).** "No single boundary — three independent mechanisms with three postures" (Phase 1 §3). **Gap:** same shape as prompt construction — no existing boundary to extend, three existing mechanisms with no shared edge between them.
- **TTS/STT vs. Brain Gateway.** Explicitly out of ADR-0003's scope and this phase's scope by the same reasoning Phase 1 applied (§4 of this document does not re-open a scope ADR-0003 explicitly closed). **Gap:** none evaluated — recorded here only to note the roadmap's own Phase 3 scope ("system boundaries") does not name TTS/STT, so this document treats their continued exclusion as consistent with, not a gap against, the stated target.
- **Boundary enforcement mechanism, system-wide.** Phase 1 found every boundary above the Gateway "holds by convention... not by a structural barrier (a lint rule, a module-boundary tool, a runtime assertion)" (§3). **Gap:** the target architecture's own Guiding Principle 7 ("incremental implementation must preserve architectural integrity and avoid introducing provider-specific coupling into higher application layers") presumes _some_ mechanism capable of detecting a violation. No such mechanism exists anywhere in the reviewed system for any boundary, Gateway boundary included — the Gateway boundary's cleanliness today is verified by grep sweep performed by this initiative's own review phases, not by anything the running system itself checks.

## 5. Dependency Gap Assessment

Extends Phase 1 §4 (Dependency Assessment, quality-rated) with the question of whether current dependency structure supports long-term maintainability as the roadmap defines it (Guiding Principle 6).

- **Excessive dependencies:** none identified in the Gateway/Provider files (Phase 1 §4). One near-miss: `providers/openrouter.ts` imports all three model-array constants from `models.ts` though a single request only needs one — architecturally harmless today (Phase 1 §4), but a **gap relative to a future Model Registry** (roadmap Phase 3 deliverable): a registry serving many providers would need import granularity finer than "import everything, use one," a pattern that does not currently exist to demonstrate it works at that finer grain.
- **Hidden dependencies:** two confirmed by Phase 1 (§4, §1.4.3) — the task-tier→model mapping's true location inside the provider rather than the nominal model registry, and `env-validation.ts`'s unconditional `OPENROUTER_API_KEY` requirement as a hidden dependency from deployment/boot on a specific provider the Gateway's own contract claims not to have. **Gap:** a target Model Registry and a target Configuration capability (both named Phase 3 deliverables) each inherit one of these hidden dependencies as a starting condition to reconcile, not a clean slate.
- **Coupling risks:** the dominant risk this phase identifies, extending Phase 1's coupling findings (§2, §4) forward against the target architecture, is that **dependency inversion is applied inconsistently** — present exactly where two different needs independently required it (`gateway.ts` → `BrainProvider` interface for substitutability; `cost-gate.ts` → `CostGateDependencies` for testability) and absent everywhere else in the Application Layer (`personas.ts`, `behavior-flags.ts`, `memory-db.ts` all depend on concrete Supabase/encryption modules directly, per Phase 1 §4). **Gap:** a target architecture whose Phase 3 scope names Context/Memory/Behavior/Emotion "Pipelines" as first-class, presumably substitutable/testable components inherits an Application Layer where the one pattern (dependency inversion) that would make components substitutable is applied in exactly two places, not as a systemic property.
- **Layering gaps:** dependency direction itself is clean and violation-free everywhere Phase 1 checked (§4) — **no gap identified** in layering direction. The one open question Phase 1 could not fully resolve from code alone — whether TTS/STT's direct-provider pattern is a deliberately separate layer or a boundary violation — is treated by this document the same way Phase 1 treated it: a scope boundary per ADR-0003's explicit text, not a layering violation, and therefore not a gap against the target architecture this document is comparing against (which is scoped identically to ADR-0003).

## 6. Provider Abstraction Gap Analysis

The single most consequential gap in this document, because the roadmap's stated Objective ("provider-agnostic... capable of evolving... to hybrid and fully self-hosted inference") depends entirely on this one abstraction holding under conditions it has never faced.

- **Current state:** `BrainProvider` (`types.ts:163-169`) is a four-method interface with no OpenRouter-shaped parameters, structurally minimal in a way that is a positive signal for portability (Phase 1 §1.4.4). One implementation exists (`providers/openrouter.ts`). This phase's own grep confirms zero mock or second implementations anywhere in `tests/`, and `providers/` (`src/core/brain/providers/`) contains exactly one file (`openrouter.ts`).
- **Desired capability:** "OpenRouter (today) · Ollama · vLLM · LM Studio (future) · Future Emma-owned model (future)" as interchangeable Provider Layer implementations (ADR-0003, High-Level Architecture diagram, "Future Providers" section); cloud, local-inference, multimodal, and specialized providers named explicitly by the Phase 2 brief's own examples for this analysis dimension.
- **Gap — cloud providers (a second OpenRouter-shaped or REST-based provider):** the interface's minimalism suggests low risk, but "suggests" is the operative word — no evidence exists either way, because n=1.
- **Gap — local inference (Ollama/vLLM/LM Studio):** Phase 1 identified a gap beyond the interface itself: these providers are typically single-model (no `models: [...]` fallback-array concept) and may not support SSE in the same shape OpenRouter does. The `BrainProvider` interface's four methods have never been implemented against a transport that lacks these OpenRouter-specific capabilities, so whether the interface's abstraction level is actually sufficient — as opposed to accidentally OpenRouter-shaped in ways not yet visible because nothing has stressed it — is unknown. Compounding this: `env-validation.ts` unconditionally requires `OPENROUTER_API_KEY` in production (Phase 1 §1.4.12), meaning a hybrid deployment (some requests local, some cloud) could not even boot under current validation rules regardless of whether the Provider Layer itself were ready.
- **Gap — multimodal providers:** the current provider already serves vision (via the `"vision"` task tier) through the same `BrainProvider` contract as chat, which is a positive precedent (Phase 1 §1.4.13 treats vision as already-proven, not aspirational). The gap here is narrower than for local inference: it is specifically whether a provider whose _native_ multimodal interface differs structurally from "vision-as-a-chat-task-tier" (e.g., a provider requiring a distinct multimodal endpoint rather than accepting image content within a chat-shaped request) would fit the current contract without a shape change — untested, because the one provider that exists happens to support vision through the same request shape as chat.
- **Gap — specialized providers** (the Phase 2 brief's fourth named example, interpreted per the roadmap's own language as a fine-tuned or purpose-built Emma model, ADR-0003 "Future Providers"): no gap-relevant evidence exists at all — this is the least-evidenced sub-question in the entire Gap Analysis, because it is furthest from anything the current single-provider system has ever had reason to consider.
- **Gap — provider selection mechanism itself:** the Gateway holds exactly one module-level provider instance, chosen once at import time, not per-request (Phase 1 §1.4.13). A target architecture with multiple concurrently-available providers requires _some_ per-request or per-capability selection point; none exists today, at any granularity — this is distinct from, and sits below, the routing gap in §7 (routing is _how_ a provider is chosen; this gap is that there is currently no mechanism to hold more than one provider instance at all, which routing would need to select between).

## 7. Routing Gap Analysis

- **Current state:** confirmed, both by Phase 1's direct code reading (§1.4.5: `TASK_MODELS` is a static compile-time map, no runtime selection, scoring, or capability matching) and by this phase's independent repo-wide grep for `CapabilityRegistry`, `capability_registry`, and `ModelRegistry` (zero matches outside `models.ts`'s own static config, and zero router-shaped class or module found anywhere in `src/core` or `src/app`).
- **Desired capability:** task routing, capability routing, policy routing, fallback routing, hybrid routing (Phase 2 brief, named examples); "Routing Engine" as a named Phase 3 deliverable (roadmap); "Capability Registry (task → capability → provider matching)" and "Cost-aware or latency-aware routing" explicitly named as deferred future work by ADR-0003 itself ("Out of Scope").
- **Task routing:** the closest thing to routing that exists is the `BrainTask` closed union (`"brain" | "vision" | "utility"`), which selects a _tier_, not a provider or model directly — Phase 1 rates this as "exactly the right abstraction level" for future task routing to extend (§1.4.5, Strength #3), meaning this specific sub-capability has a genuine, evidence-based precondition already in place, distinct from the other four routing types below, which have none.
- **Capability routing** (matching a request's required capability — e.g., "needs tool-calling," "needs 128k context" — to a provider that offers it): no precedent exists. The current system's capabilities (tool-calling, vision, embeddings) are each hardcoded to specific task tiers rather than queried as provider capabilities, because only one provider exists to query.
- **Policy routing** (routing by cost, latency, compliance, or data-sovereignty policy): no precedent exists; `cost-gate.ts` enforces budget _after_ a provider has already been selected (there being only one), not as an input to selection.
- **Fallback routing** (retry against a different provider on failure, as opposed to OpenRouter's own internal `models: [...]` fallback-array, which is a single-provider capability already normalized inside the provider file per Phase 1 §1.4.5): the existing fallback mechanism is provider-internal, not Gateway-level cross-provider fallback. Phase 1 explicitly notes the current code offers a precedent for provider-internal fallback "without having ever exercised" cross-provider fallback even once (§1.4.5).
- **Hybrid routing** (splitting traffic between local and cloud providers by request characteristics): no precedent exists; depends on both the provider-selection-per-request mechanism gap (§6) and the `env-validation.ts` configuration gap (§13) being addressed as preconditions, neither of which this document proposes addressing.
- **Ambiguous ownership for routing's future home:** ADR-0003's own architecture diagram places "routing/registry logic" inside the Gateway box, but the one routing-adjacent mechanism that exists today (task-tier fallback) lives inside the _provider_ file, not the gateway itself (Phase 1 §1.4.5). This is a gap the target architecture will need to resolve as a precondition to designing routing at all, not a routing capability gap in itself — recorded here because it is evidence bearing directly on routing readiness.

## 8. Context Architecture Gap Analysis

- **Lifecycle:** two independent pipelines with different lifecycles — client-side summaries live only in the browser's in-memory array (never persisted); server-side summaries persist encrypted to `conversations.summary` (Phase 1 §1.4.6). **Desired:** a "Context Pipeline" (roadmap Phase 3 scope) implies one lifecycle a caller can reason about. **Gap:** no single lifecycle exists to formalize into a pipeline — two lifecycles would need to be reconciled as a precondition, not extended from a shared base.
- **Summarization:** client-side is ratio-triggered against a 100k-token budget with 4-char-per-token approximation; server-side is a flat 20-message hard cap with no token accounting at all (Phase 1 §1.4.6). **Gap:** these are not two implementations of one concept but two different concepts (token-budget-aware vs. message-count-aware) that share a name. A target Context Pipeline's "summarization" capability has two non-interoperable existing definitions to reconcile.
- **Ownership:** context _composition_ (what goes into a request) is cleanly owned end-to-end by `route.ts` as a defensible single composition root (Phase 1 §1.4.6). **No gap** in composition ownership specifically — this is one of the few sub-areas in context architecture Phase 1 found genuinely sound, and this document does not manufacture a gap where the evidence does not support one.
- **Scalability:** the client-side pipeline's 4-char-per-token approximation is a heuristic, not an exact accounting — Phase 1 does not flag this as presently broken, but a target architecture with more providers (each with potentially different tokenization) inherits an approximation calibrated against one provider's tokenizer, with no evidence it holds for others.
- **Composition (cross-pipeline):** a single request can carry both a persisted "Previous Session Context" prompt block and an in-band synthetic `[SUMMARY]` message simultaneously (Phase 0 §1.4, confirmed by Phase 1 §1.4.6). **Gap:** no single component can currently answer "what does the model already know from earlier in this conversation" — a question a target Context Pipeline would need to be able to answer authoritatively, and currently cannot, from any one place.

## 9. Memory Architecture Gap Analysis

- **Retrieval:** `getRelevantMemoriesForUser()` fetches all active rows (no DB-side `LIMIT`), scores only if the row count exceeds the requested limit, using keyword-overlap × confidence — not embedding similarity (Phase 1 §1.4.7). **Desired:** the roadmap Phase 3 scope names a "Memory Pipeline" without specifying retrieval mechanism, but the Phase 2 brief's own named example for this dimension is explicitly "retrieval" and "ranking" as distinct sub-capabilities. **Gap:** no ranking infrastructure exists beyond in-process keyword scoring; no embedding column exists on the memory schema (Phase 0, confirmed absent).
- **Ranking:** the same absence — no database index or column supports relevance ranking. **Gap, coupled to retrieval:** raising the current 200-active-memory-per-user cap and improving ranking quality are architecturally coupled changes, not independent ones, because full in-process scoring is only viable below that cap (Phase 1 §1.4.7).
- **Persistence:** fully owned by one module (`memory-db.ts`), AES-256-GCM encrypted, with capacity enforcement and supersede-on-change semantics (Phase 1 §1.4.7; Phase 0 §8 — the one responsibility Phase 0 found with zero ownership fragmentation). **No gap** in persistence ownership or lifecycle mechanics specifically.
- **Lifecycle:** 200-row cap with soft-deletion of the oldest/lowest-confidence entry on overflow (Phase 0 §1.3). **No gap** identified in lifecycle policy itself, though it is the mechanism the retrieval/ranking gap above is structurally bound to.
- **Scalability:** "architecturally flat past" the current cap (Phase 1 §2) — this phase does not treat "flat past a cap" as itself a defect (per Phase 1's own framing, it is sound _given_ the cap), but records it as a gap against a target architecture whose Objective explicitly names scaling "without requiring major application refactoring," since raising the cap today would require exactly that: a coupled schema and retrieval-mechanism change, not an independent scaling decision.
- **Injection architecture:** a single, correctly centralized point (`personas.ts`'s `serializeMemories()`), in contrast to the fragmented pattern found in prompt construction generally (Phase 1 §1.4.7). **No gap** in injection-point centralization.

## 10. Prompt Architecture Gap Analysis

- **Ownership:** nominally `personas.ts`; actually six competing owners, five identified by Phase 0 (`vision/route.ts`, `summarize/route.ts`, `ingest/whatsapp/route.ts`, two in `history/route.ts`) plus internal mixing within `personas.ts` itself (persona voice and `[EMMA_ROUTINE]`/`[emotion:]` protocol-tag instructions interleaved as undifferentiated string literals, Phase 1 §1.4.1, §1.4.10). **Desired:** a "Prompt Pipeline" (roadmap Phase 3 scope) with the Phase 2 brief naming "ownership," "reuse," "composition," "consistency," "maintainability" as the specific sub-dimensions to evaluate.
  - **Ownership gap:** six owners where the target architecture's naming ("Prompt Pipeline," singular) implies one.
  - **Reuse gap:** near-zero across channels — standard chat, vision, summarization, and WhatsApp each construct prompt text independently with no shared template or shared tag-guard helper; the `[EXTERNAL DATA]` prompt-injection guard is implemented three separate times (Phase 1 §1.4.10, Phase 0 §1.7).
  - **Composition gap:** `buildSystemPromptBlocks()`'s stable/dynamic split is a genuinely useful seam that anticipates prompt-caching, but `buildSystemPrompt()` — the only function any caller actually invokes — flattens the seam back into one string, meaning the seam provides zero runtime value today; it is dormant infrastructure for a capability (prompt caching, named explicitly as a future extension point by the Phase 7B report) the Gateway itself would also need to expose before the seam could be used (Phase 1 §1.4.10).
  - **Consistency gap:** a tone-rule change (e.g., core vocabulary in `personas.ts:41-57`) has no mechanism to propagate to the five other hardcoded prompts — tone consistency across channels is maintained by remembering to update six files, not by architecture (Phase 1 §1.4.10).
  - **Maintainability gap:** directly follows from the above — six independent edit points for one conceptual change.
  - **Isolation:** the one sub-dimension with **no gap** — prompt text never crosses into the Gateway or provider layer as anything but opaque message content; ADR-0003's boundary holds completely here (Phase 1 §1.4.10).

## 11. Operational Gap Assessment

The domain with the fewest existing artifacts to compare against a target, confirmed independently by this phase's own inspection (in addition to Phase 1's direct code reads):

- **Tracing:** zero request-correlation mechanism anywhere in the AI pipeline; a repo-wide grep for `correlationId|traceId|requestId|OpenTelemetry|opentelemetry` returns matches only in an unrelated subsystem (`src/core/account-deletion/`), independently re-confirmed by Phase 1 (§1.4.14) and consistent with this phase's own `package.json` inspection showing no `@opentelemetry/*` dependency present. **Desired:** "tracing," "diagnostics" named explicitly by the Phase 2 brief; "Analytics" named by the roadmap's Phase 3 scope. **Gap: total absence**, not a partial or misconfigured implementation.
- **Metrics:** zero counters, histograms, or gauges for gateway call volume, latency, retry rate, or per-provider error rate anywhere in the reviewed code (Phase 1 §1.4.14); this phase's `package.json` search confirms no `statsd` or `posthog` dependency exists. The only quantitative signal captured anywhere is `cost-gate.ts`'s token/request accounting, which serves billing, not operational health (Phase 1 §1.4.14). **Gap: total absence.**
- **Structured logging:** no shared structured-logging utility exists; every module prefixes its own `console.warn`/`console.error` calls with its own ad hoc tag (`[EMMA]`, `[EMMA API]`, `[CostGate]`, `[Memory DB]`), already named as accepted debt in the Phase 7B report (Phase 1 §1.4.14); this phase's `package.json` search confirms no `pino` or `winston` dependency exists. **Gap: total absence of structured logging infrastructure**; heterogeneous unstructured logging exists in its place.
- **Diagnostics:** the closest existing artifact is `admin-diagnostics.ts` (referenced in Phase 0's grep results as the label-only provider-token reference — the other such incidental reference, in `memory-extraction-parser.ts`, being comment-only), but this document found no evidence it is Gateway-specific diagnostic tooling as opposed to general admin tooling — insufficient evidence exists to characterize this as anything more than a name Phase 0/1 encountered in passing.
- **Monitoring:** no dashboard, alert configuration, or monitoring-as-code artifact specific to the Brain Gateway was found in `docs/` (this phase's directory listing of `docs/` shows general-purpose runbooks — `runbook-incident-response.md`, `runbook-staging-environment-setup.md`, `runbook-restore-drill.md`, `runbook-encryption-key-escrow.md` — none scoped to the Gateway specifically).
- **Observability (as a synthesizing capability):** Sentry is wired but sparse — exactly 3 call sites in the entire AI pipeline (`route.ts:509,637,665`), with no Sentry instrumentation inside the Gateway or provider files themselves; the Gateway's own header comment explicitly lists "Sentry capture policy" as _not_ its responsibility, and no other component has consistently assumed it (Phase 1 §1.4.14). **Gap:** a designed cross-cutting concern with an explicit non-owner and no compensating owner — distinct from, and worse than, simply "under-instrumented," because the current architecture has already made an explicit decision (in the Gateway's header comment) not to own this, without anything else picking it up.

## 12. Error Handling Gap Analysis

- **Representation:** three independent, structurally similar-but-disconnected error shapes — `BrainRequestError` (Gateway/Provider boundary, normalized, single construction point), `EmmaError` hierarchy (`src/lib/errors.ts`, thrown, separate type family), and `CostGateDecision` (a third ad hoc discriminated union overlapping in purpose with `RateLimitError`/`RateLimitUnavailableError` without reusing them) — plus a fourth, disconnected user-facing translation layer (`getPersonaErrorMessage()`) that switches on raw HTTP status rather than consulting the `BrainRequestError.code` taxonomy the Gateway already computed for the same failure (Phase 1 §1.4.15). **Desired:** "error representation," "consistency" named explicitly by the Phase 2 brief. **Gap:** four shapes for one concept (a failed request), with the most-normalized of the four (`BrainRequestError`) bypassed by the translation layer that most needs normalized input.
- **Ownership:** each of the four shapes above has a clear, single owner within its own file — the gap is not ownership-within-a-shape but the absence of any owner for reconciling the four shapes into one taxonomy, a gap this document records as structurally identical to the "one named owner, one undocumented competitor" pattern found in prompt/behavior/context architecture (§3, §8, §10), here applied to error representation.
- **Propagation:** the error contract's dual mode (values for HTTP failures, throws for transport failures) is a deliberate, documented design choice mirroring pre-existing call-site patterns (Phase 1 §6, Strength #4) — **no gap** identified in propagation mechanism itself.
- **Retry strategy:** the retry _mechanism_ is properly centralized (`fetchWithRetry()`, single implementation, no duplicate retry loop found anywhere, Phase 1 §1.4.16). Retry _policy_ is set per call site with no documented rationale distinguishing choices — the shared default is `maxRetries: 3`, but the Gateway's own request translation overrides it to `0` unless a caller explicitly opts in, meaning each of the 16 invocation sites makes its own silent retry-count decision (Phase 1 §1.4.16). **Gap:** no per-task-tier or system-wide retry _policy_ exists, only a centralized retry _mechanism_ individual call sites configure independently.
- **Retry-eligibility boundary:** `DEFAULT_RETRY.retryOn` (`429, 500, 502, 503, 529`) is fixed and itself contains provider-specific vocabulary (529, an OpenRouter/Anthropic-specific status) defined independently in both `errors.ts`'s shared retry list and the provider's own normalization (Phase 1 §1.4.16). **Gap relative to provider independence (§6):** a status code ADR-0003 treats as something "the Gateway normalizes so callers never see it" is simultaneously hardcoded one layer up, in a utility that sits outside the Gateway's boundary — a second provider without this exact status code would either silently fail to retry-eligible-classify one of its own failure modes, or the shared list would need to already anticipate a status code that does not yet have a second provider to be defined against.
- **Timeout coupling:** `connectionTimeoutMs` aborts only when no response headers arrive within the window; a request that starts streaming and then stalls mid-body is not covered (Phase 1 §1.4.16, citing the Phase 7B report's own acceptance of this as a known behavior delta). **Gap:** a target architecture's "Retry" and "Fallback" (both named Phase 3 deliverables) inherit a timeout mechanism with a known, accepted blind spot for exactly the failure mode (mid-stream stall) that a hybrid or local-inference provider — potentially slower or less reliable than a managed API — would be more likely to exhibit.

## 13. Configuration Gap Analysis

- **Configuration ownership:** split along a defensible line — model IDs in `models.ts`, per-client product config in `client-config.ts`, environment shape in `env-validation.ts` — with one confirmed leak (embedding model ID in the provider file, justified in-file as "provider configuration, not caller configuration," a defensible position but one that means "the model registry" is not, in fact, one file (Phase 1 §1.4.12). **Desired:** a "Model Registry" as a distinct, named Phase 3 deliverable, implying a single, complete, consultable source of every model ID Emma uses. **Gap:** no single file currently satisfies that description; a reader consulting only `models.ts` would not find every model ID in use.
- **Feature flags:** none exist as a distinct mechanism anywhere in the reviewed AI pipeline (Phase 1 §1.4.12), independently re-confirmed by this phase's `package.json` search (no `flagsmith`, `launchdarkly`, `unleash`, `posthog`, or similarly-purposed dependency present). Phase 1 characterized this as a neutral observation (the system does not currently need feature flags for what it does today) rather than a gap; this document extends that judgment forward: a target architecture introducing multiple concurrently-available providers, capability routing, and staged provider rollouts (the roadmap's own stated destination) is a materially different situation from today's single-provider system, and **has no existing feature-flag-shaped mechanism to build a staged-rollout capability on top of**, should the target architecture require one — recorded as a gap of absence relative to that future need, not a defect in the current system.
- **Runtime configuration:** task-tier→model mapping is compile-time static (`TASK_MODELS`), with no runtime override — no env var, no database row, no admin toggle (Phase 1 §1.4.12). **Gap:** a target Configuration capability implies at least the possibility of changing provider/model/routing behavior without a code change and redeploy; no such mechanism exists today at any granularity.
- **Provider configuration — the unconditional `OPENROUTER_API_KEY` requirement:** `env-validation.ts`'s `PRODUCTION_REQUIRED_ENV` unconditionally requires `OPENROUTER_API_KEY` in production (Phase 1 §1.4.12, independently re-confirmed by Phase 0 §1.7 as already-documented, deliberately-deferred debt). **Gap, and the most structurally consequential single item in this section:** this assumption sits one layer below the Gateway itself — production _boot_ fails without OpenRouter specifically, not merely "without a configured provider." A target architecture where OpenRouter is one interchangeable provider among several inherits a validation layer that currently treats it as the _only_ valid provider, meaning multi-provider readiness is blocked at configuration before a second provider's code could even be reached in a production deployment.

## 15. Architectural Consistency Analysis

Whether recurring architectural patterns are applied consistently across the system — per the Phase 2 brief: ownership, dependency direction, abstraction, error handling, configuration, routing.

- **Ownership:** **inconsistent**, and this is the single most recurring finding across this entire document. The same shape — one documented owner, one or more undocumented competitors — recurs independently in prompt construction (§10), behavior derivation (§3), context management (§8), rate/usage limiting (§3, Phase 1 §1.4.1), and error representation (§12). This is not four or five unrelated gaps; it is one architectural pattern applied inconsistently across five different subsystems, each of which independently arrived at the same failure mode.
- **Dependency direction:** **consistent.** Every dependency this document's baseline (Phase 1 §4) checked flows strictly downward with no cycles and no upward violations — the one property found consistently applied everywhere, including at the Gateway boundary specifically. No gap.
- **Abstraction:** **inconsistent.** Dependency inversion — the specific abstraction pattern that would make components substitutable — is applied in exactly two places (`gateway.ts`'s `BrainProvider` interface, `cost-gate.ts`'s `CostGateDependencies`) and absent everywhere else in the Application Layer (§5 above). The Gateway itself demonstrates the codebase knows how to apply this pattern; it is not applied as a systemic convention.
- **Error handling:** **inconsistent** — four disconnected shapes for one concept (§12).
- **Configuration:** **partially consistent** — ownership is coherent along a defensible split with one confirmed leak (§13); the inconsistency is less severe here than in ownership/abstraction/error-handling generally, because the split, while imperfect, follows an articulable rule (provider-internal vs. caller-facing) rather than being purely accidental.
- **Routing:** not evaluable for consistency — no routing exists anywhere in the system to be consistent or inconsistent (§7).
- **Cross-cutting observation:** the one place this document finds the architecture _is_ consistent — dependency direction — is also the one property that requires no cross-team or cross-module coordination to maintain (each file simply does not import something upstream of it). Every place the architecture is inconsistent — ownership, abstraction application, error representation — is a property that requires active coordination across multiple independently-evolving files to hold. This document does not draw a causal conclusion from that correlation (doing so would begin to propose a solution, out of scope for this phase), but records it as the clearest single pattern connecting the otherwise-separate findings above.

## 17. Phase 2 Conclusion

**Overall architectural maturity relative to the target architecture:** the current system's bimodal maturity, already established by Phase 1 (mature at the Gateway boundary, fragmented above it), maps onto the target architecture's gaps in a specific and non-uniform way. The Gateway boundary itself — where Phase 1 found the strongest existing architecture — is also where this Gap Analysis finds the single highest-severity gap (GAP-01, provider-abstraction unverified), because that boundary is precisely where the roadmap's stated destination places the most weight. The Application Layer's fragmentation, already Phase 1's dominant finding, recurs here as GAP-05 in gap-specific form: not "this subsystem is imperfect" but "the target architecture's own Phase 3 deliverable list (Context/Memory/Behavior/Prompt Pipelines) presumes single, coherent owners this document could not find."

**Overall readiness for Architecture Discovery (Phase 3):** the gaps identified in this document are, without exception, gaps this document can describe with evidence — none required speculation about what might be true, because Phase 0 and Phase 1's prior work already supplied direct, cited evidence for nearly every finding above; the handful of gaps requiring this phase's own direct inspection (routing/registry code absence, mock-provider absence, observability-tooling absence in `package.json`) were resolved by simple, unambiguous repository searches with clean negative results. This is itself evidence of readiness: a Gap Analysis that had to guess at current-state facts would not be a sound basis for Architecture Discovery; this one did not have to guess.

**Key capability gaps a future Architecture Discovery phase would need to treat as design inputs, not incremental extensions:** capability-based routing and the Capability/Model Registry (GAP-02) have no existing precedent to extend — Phase 3 would be designing these from a clean slate, informed by the one positive precondition Phase 1 already identified (the `BrainTask` abstraction's tier-shaped, not model-ID-shaped, design). Operational architecture (GAP-04) has the same clean-slate character. The provider-abstraction's unverified status (GAP-01) is different in kind: it is not a clean slate but an existing artifact whose correctness under real multi-provider conditions Phase 3 would need to treat as an open question, not a settled premise, when designing anything that depends on it.

**Recommendation: Ready for Phase 3, with Minor Concerns.**

Justification: no gap identified in this document rises to a severity that would block Architecture Discovery from proceeding — every Critical-rated item (GAP-01, GAP-02) is a gap of absence or unverified assumption, not a defect requiring remediation before design work can begin; in both cases, Phase 3's own design work is precisely where a Capability Registry, Routing Engine, and multi-provider validation would be defined, meaning these are not blockers to Phase 3 so much as they are Phase 3's core subject matter, correctly surfaced here as gaps rather than already (incorrectly) solved. The "Minor Concerns" qualifier reflects two things specifically: (1) GAP-03's configuration-layer blocker (`OPENROUTER_API_KEY` unconditional requirement) sits structurally _below_ where Phase 3's architecture work would normally focus, meaning Phase 3's design should account for it as a known precondition rather than discover it mid-design; and (2) the ownership-fragmentation pattern (GAP-05) recurs widely enough across the Application Layer that Phase 3's five named Pipeline deliverables (Context, Memory, Behavior, Emotion, Prompt) should each expect to begin from a reconciliation problem, not a clean extension point — a scoping expectation worth setting explicitly before Phase 3 begins, rather than letting each pipeline's design work discover its own version of the same pattern independently. The "Major Analysis Required" threshold is not met: this document found no gap whose severity or evidentiary uncertainty would call the soundness of proceeding into question, and every gap identified traces to specific, already-cited evidence rather than to a suspected but unconfirmed problem.
