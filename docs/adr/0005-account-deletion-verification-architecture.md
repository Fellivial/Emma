# ADR 0005: Account Deletion Verification Architecture — extending the existing Registry, adapter, and checkpoint substrate

- **Status:** Proposed (Revised) — revised following Independent Review; awaiting Independent Re-review and approval per the Roadmap's Engineering Workflow step 6 ("ADR Approval") before Phase 4B (Technical Design) begins.
- **Date:** 2026-07-18 (original); revised 2026-07-18
- **Phase:** 4 ("Verification") — see [Account Deletion Roadmap v1.0 (Frozen)](../roadmaps/account-deletion-roadmap-v1.md)
- **Implementation:** None yet. This ADR records the chosen architecture for Phase 4B to design against; it does not itself change `src/core/account-deletion/*`, `src/app/api/emma/gdpr/route.ts`, or any migration.
- **Companion:** [Phase 4A Architecture Discovery Report](../plans/2026-07-18-account-deletion-phase4a-architecture-discovery.md) — the full gap analysis, dependency analysis, reuse validation, assumption register, alternatives evaluation, and requirements traceability matrix this ADR's decision is drawn from. This ADR states the decision and its rationale at ADR altitude; the discovery report carries the supporting analysis this document does not repeat.
- **Review history:** [Independent ADR-0005 Review Report](../plans/2026-07-18-account-deletion-adr-0005-independent-review.md) — verdict MAJOR REVISIONS REQUIRED, three scoped findings. [ADR-0005 Revision Report](../plans/2026-07-18-account-deletion-adr-0005-revision.md) — resolution matrix for every finding, applied in this revision.

## Context

Phase 3 and Phase 3.1 (ADR-0004) built a registry-driven, transactional deletion execution foundation and a durable, concurrency-safe workflow orchestrator on top of it. Both explicitly stopped short of verification: ADR-0004's own "Future evolution constraints" section named "real per-table database verification" as work "deferred to whichever future phase implements it," and the Phase 3.1 Production Readiness Report's closing section ("What Phase 4 inherits") states plainly that Phase 4 "should be able to build ... real per-table verification on top of what exists here without revisiting this phase's fix."

The roadmap now frozen for Phases 4–7 names Phase 4's objective directly: _"Add the capability to prove that all targeted resources have been successfully deleted. The completion workflow must not be considered proof of successful deletion without a verification process."_ Today, the workflow's own success report (`status: "completed"`) is the only signal that a deletion succeeded — it is generated by the same code path that performed the deletion, not independently confirmed.

## Problem Statement

The account-deletion workflow reports `completed` based on: an atomic database transaction returning without error, and Storage adapter `delete()` calls returning `success: true`. Neither is independently re-checked. The Registry already reserves a `verificationAdapter` field (populated `null` on all 37 entries) and the workflow already reserves three verification states (`verify_database`, `verify_storage`, `verify_external`) in its state machine — both provisioned by Phase 1/3 specifically for this future need, and both currently no-ops. Phase 4 must decide how to populate them without duplicating or destabilizing the Phase 1–3.1 foundation that Phase 3.1's Production Readiness Report explicitly said should not need to be revisited.

## Design Goals

1. Every resource the Registry lists as deleted must be independently re-checked, not inferred from the deletion step's own report — mirroring the pattern `StorageBucketAdapter.verify()` already established (re-list, don't trust `delete()`'s return value).
2. Verification evidence must survive the deletion it documents — the whole point of "auditable evidence" is that it outlives the data it attests to.
3. No new persistence model, registry, or adapter contract is introduced unless the existing ones (Registry, `DeletionAdapter`, `deletion_requests`/`checkpoint`) are demonstrably insufficient.
4. Deletion success and verification success must become distinguishable, not conflated, in the workflow's final observable status and in whatever the caller (`POST /api/emma/gdpr`, and therefore its clients) is ultimately told — a verification failure must be capable of changing that outcome, not just being logged alongside an unaffected `"completed"` result.
5. Zero behavior change to the untouched foundation: the Registry's existing entries, the transactional delete function, and the `DeletionAdapter` interface are not modified in shape — only extended (new field values, one new function, real bodies for already-declared step functions).

## Constraints

Carried directly from the roadmap's own Phase 4 scope and the roadmap's governance rules:

- Out of scope: grace period, delayed deletion, reconciliation automation, operator dashboard, monitoring dashboard, scheduling, production metrics (roadmap, Phase 4 "Out of Scope").
- Must not modify the Roadmap, ADR-0001 through ADR-0004, or any existing TDD (Phase 4A's own constraints).
- Must not introduce parallel registries, duplicate workflow states, redundant adapters, or persistence models that duplicate existing responsibilities (roadmap Constraints).
- Extension is prioritized over replacement; every "Create New" decision requires technical justification (roadmap Constraints; Phase 4A Reuse Validation, discovery report §7).
- No implementation, SQL, migrations, or TDD-level interface definitions may be produced at this stage — this ADR records the decision, not the design.

## Alternatives Considered

Full evaluation (complexity/maintainability/extensibility/operational cost/compatibility/production risk for each) is in the discovery report §9. Summarized here:

**A. Registry-and-Workflow-integrated verification** — populate the Registry's existing `verificationAdapter` field, add one new read-only Registry-parameterized SQL function for database resources (mirroring `delete_user_owned_data_ordered`'s safety pattern), implement real logic in the already-reserved `stepVerifyDatabase`/`stepVerifyExternal` workflow steps, and write evidence into the existing `checkpoint jsonb` array. **Selected.**

**B. New, separate `verification_results` table and standalone verification module**, decoupled from `deletion_requests`/`checkpoint`. **Rejected** — duplicates a persistence responsibility the Registry and `deletion_requests` were already deliberately built to hold (the `verify_*` states were reserved in the _same_ status enum as the deletion states from Phase 1 onward), and directly conflicts with the roadmap's own anti-duplication constraint and its declared dependency on `deletion_requests`.

**C. Fully decoupled, out-of-band verification**, independent of `runDeletionWorkflow`'s execution model, triggered separately per user or on a schedule. **Rejected as a full alternative** — scheduling triggers are explicitly Phase 5 scope, not Phase 4, and a second, less-exercised invocation path would add review surface without adding capability Phase 4 actually needs. **Its core principle is preserved inside Alternative A**: verification must independently re-check real state rather than trust the deletion step's own report. This is not architectural decoupling from the workflow — it is behavioral independence from the deletion step's self-reported success, exactly what `StorageBucketAdapter.verify()` already proves works.

## Chosen Architecture

Extend, in place, already-existing extension points — introduce nothing that isn't either a value change to an existing field, one narrowly-scoped new function, or a binding requirement on how already-declared functions must behave:

1. **Registry (`registry.ts`):** populate `verificationAdapter` on database resource entries (a label, mirroring how `deletionAdapter: "legacy-table-delete"` is a label rather than a literal adapter object, per ADR-0004's rationale for why database resources don't use the `DeletionAdapter` object pattern). No new Registry fields.
2. **A new, read-only, Registry-parameterized SQL function** — the verification counterpart to `delete_user_owned_data_ordered`: given a user and the Registry's table/column list, returns remaining-row counts per table, with the same identifier-validation and per-column-type-casting discipline Phase 2.1's live validation already proved necessary for this exact pattern. This is the one genuinely new component in this architecture, and it is justified in the discovery report §7 specifically because no existing read-only, multi-table, Registry-driven query mechanism exists, and mutating the existing delete function to add a "dry run" mode would conflate two different risk profiles under one function. **This function must be callable as a single batched call across all database resources, not one call per table** — see item 6 below.
3. **Workflow (`workflow.ts`):** `stepVerifyDatabase` and `stepVerifyExternal` — already-declared, currently-placeholder functions — get real bodies. No new states are added to `STATE_ORDER`; `verify_database`, `verify_storage`, `verify_external` are already correctly positioned there. `stepVerifyStorage`'s existing call to real `DeletionAdapter.verify()` logic needs no change to _what_ it calls — but see items 5 and 7 below for binding changes to how its _result_ is handled.
4. **Evidence (`deletion_requests.checkpoint`):** verification results are written as real `CheckpointEntry` rows in the same array deletion progress already uses, replacing today's `"skipped"` placeholders. This is viable specifically because `deletion_requests` is not itself one of the Registry's 32 deletion targets and `auth.users` is never deleted by this flow (discovery report §4, §8 — Assumptions 1–2) — a `deletion_requests` row, and its checkpoint evidence, structurally outlives the user data it attests to.
5. **Workflow outcome authority (added on revision — resolves Independent Review finding, see the ADR-0005 Revision Report §"Finding 1"):** a verification failure **must** be capable of changing the workflow's final `status`, and therefore `route.ts`'s `success` field and every client that reads it (confirmed: `settings/privacy/page.tsx` branches directly on `success`). Today, `workflow.ts`'s `CRITICAL_STEPS` (`workflow.ts:342`) contains only `"deleting_database"`, so a failure in `verify_database`/`verify_storage`/`verify_external` falls through to the log-only `best_effort_step_failed` branch and the workflow still reaches `"completed"` — this is the exact gap Phase 4 exists to close, and it does not close itself merely by giving the verify steps real bodies (item 3). Phase 4B **must** extend the workflow's status-derivation mechanism so this is no longer true. The **exact mechanism is deliberately left to Phase 4B** (a TDD-level, HOW-altitude decision, not an ADR-level, WHY-altitude one, per the Roadmap's Architecture Authority table) — candidates include adding the verify phases to `CRITICAL_STEPS` outright (reusing the existing `retry_pending`/`failed` semantics as-is) or introducing a distinct, less drastic outcome (e.g., a status distinguishing "deleted but not independently verified" from "deletion itself failed") — but **that a decision must be made and must be load-bearing for the final status is binding at this ADR level**, not optional or deferred indefinitely.
6. **Runtime duration (added on revision — resolves Independent Review finding, see the ADR-0005 Revision Report §"Finding 2"):** verification adds real round-trips to the already-synchronous `POST /api/emma/gdpr` request (Alternative C, which would have moved this out of the request path, was rejected — see Alternatives Considered). To keep this bounded, database verification **must** execute as one additional batched call across all 32 database resources (mirroring how `delete_user_owned_data_ordered` itself is one call, not 32), not a per-table loop. Additionally: `vercel.json` currently has **no `maxDuration` override for `src/app/api/emma/gdpr/route.ts`**, unlike every comparable route (`emma/route.ts`: 60s, `agent/route.ts`: 120s, `vision/route.ts`: 30s, and others) — confirmed directly against `vercel.json`'s `functions` block. Phase 4B **must** explicitly evaluate whether the platform default is sufficient for the added, batched verification work, and set an explicit override if not. This is architectural (it constrains the shape of item 2's new function to "one batched call") rather than purely an implementation detail, which is why it is named here rather than left to Phase 4B to discover independently.
7. **Resume safety (added on revision — resolves Independent Review finding, see the ADR-0005 Revision Report §"Finding 3"):** `stepDeletingDatabase` and `stepDeletingStorage` both guard against re-running already-completed work on a resumed workflow via `isPhaseCompleted()` (`workflow.ts:223`, `:243`). The verify steps **must** follow the same pattern: `stepVerifyDatabase`, `stepVerifyStorage`, and `stepVerifyExternal` (including `stepVerifyStorage`'s current implementation, which does not yet have this guard either) must each check `isPhaseCompleted()` before re-executing, so a workflow interrupted mid-verification and re-invoked resumes from the last completed resource rather than re-running the whole step and appending a duplicate batch of checkpoint entries. This reuses an existing, already-proven pattern — it does not introduce a new one.

Whether `CheckpointResourceStatus`'s three-value vocabulary (`completed`/`failed`/`skipped`) needs a verification-specific extension, or whether disambiguating via the existing `phase` field (`verify_*` vs. `deleting_*`) is sufficient, is left to Phase 4B — a type-shape decision, not an architectural one (discovery report §13, Open Questions). This is unchanged by this revision; it is a separate question from item 5's requirement that verification results be load-bearing for the final status, whatever vocabulary ultimately expresses them.

## Trade-offs

- The one new SQL function inherits the same class of risk Phase 2.1 found real, mock-invisible bugs in for the delete function (type/casting edge cases across non-`uuid` ownership columns). This argues for live-database validation of the verification function before production reliance, following the same discipline Phase 2.1 and Phase 3.1 already established for this codebase, not against choosing this pattern.
- Extending `checkpoint` rather than introducing a dedicated verification table means verification results are queryable only by reading and parsing a `jsonb` array per user, not by SQL aggregation across users. This is an accepted limitation for Phase 4: the roadmap explicitly excludes operator/monitoring dashboards from this phase's scope, so cross-user queryability is not a Phase 4 requirement. If Phase 7 (Production Operations) later needs it, that is a genuine future consideration (below), not a gap this ADR needs to close now.
- Verification adds real DB/Storage round-trips to every deletion request's execution time (today's placeholder steps are near-instant). This revision makes batching (one additional RPC call, not one per table) and a `vercel.json` timeout evaluation **binding requirements**, not discretionary recommendations — see Chosen Architecture item 6. The exact timeout value is still a Phase 4B decision; that batching must happen, and that the route's timeout headroom must be checked, are not.
- OAuth and background-job resources remain unverifiable under this architecture until they have real deletion adapters — which no phase on the current frozen roadmap is explicitly assigned to build (see Open Questions).

## Consequences

**Positive:**

- Zero new persistence models, registries, or adapter contracts — every extension point this ADR uses was deliberately provisioned by ADR-0004 for exactly this purpose, so Phase 4 closes the gap ADR-0004 named rather than opening a new one.
- Verification evidence is durable by construction (survives the deletion it documents) without any new table or lifecycle question about when/how it's created relative to `deletion_requests` rows — it's the same row, same array.
- The re-verify-don't-re-trust principle (Alternative C, folded into A) means verification actually catches a real class of bug the deletion step's own self-report cannot: a resource that appears deleted according to the code that deleted it, but isn't, according to an independent check — exactly the gap `StorageBucketAdapter.verify()` was built to close for Storage, now extended to database resources.

**Trade-offs (see above, restated for completeness):** cross-user queryability of verification results is deferred past Phase 4; the new SQL function needs the same live-validation discipline the delete function required; OAuth/background-job verification stays blocked on deletion-adapter work with no assigned owner phase.

**Process consequence:** this is the first ADR in the account-deletion effort written _before_ any implementation exists for it, rather than retroactively (ADR-0004's own "Process consequence" section named writing the next phase's design record before implementation begins as "the strongest argument" its own retroactive origin could make). This ADR is that argument acted on.

## Future Considerations

- If Phase 6 (Reconciliation) or Phase 7 (Production Operations) later need cross-user or cross-time queryability of verification results that `checkpoint jsonb` cannot efficiently provide, that would be a new ADR's decision to make, not a reason to preemptively build that capability into Phase 4, per the roadmap's explicit instruction to prioritize extension over replacement and avoid speculative architecture.
- OAuth (`oauth.client_integrations`) and background-job (`background.document_process`) deletion adapters remain unbuilt and unassigned to any named phase on the current roadmap. Whoever eventually builds them should extend the same `DeletionAdapter` interface Storage already uses, and, per this ADR, should also populate their Registry `verificationAdapter` values the same way this ADR populates them for database resources, rather than inventing a third verification pattern.
- The evidentiary standard question (existence-check vs. something stronger) is a product/legal decision, not an engineering one, and is not resolved by this ADR (see Open Questions in the companion discovery report, §13).

## Open Questions

See the companion discovery report §13 for full detail (evidentiary standard for "auditable evidence"; which phase, if any, implements OAuth/background-job deletion adapters; whether production shares the linked validation project's schema-tracking gap; the `CheckpointResourceStatus` vocabulary fork). None of these block Phase 4B from beginning — each is scoped to an owner outside this ADR's decision (product/legal, the roadmap owner, ops, or the Phase 4B TDD author respectively). This set is unchanged by this revision: the fifth question the Independent Review flagged as conspicuously absent ("whether/how the workflow's status-derivation mechanism needs to change") is not added here as an open question, because it is no longer open — it is now a binding requirement (Chosen Architecture item 5, Decision Log above). What remains open about it (the exact status-transition _mechanism_) is scoped to Phase 4B in the Decision Log row above, consistent with how the other four questions are scoped.
