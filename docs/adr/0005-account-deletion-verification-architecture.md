## Context

Phase 3 and Phase 3.1 (ADR-0004) built a registry-driven, transactional deletion execution foundation and a durable, concurrency-safe workflow orchestrator on top of it. Both explicitly stopped short of verification: ADR-0004's own "Future evolution constraints" section named "real per-table database verification" as work "deferred to whichever future phase implements it," and the Phase 3.1 Production Readiness Report's closing section ("What Phase 4 inherits") states plainly that Phase 4 "should be able to build ... real per-table verification on top of what exists here without revisiting this phase's fix."

The roadmap now frozen for Phases 4–7 names Phase 4's objective directly: _"Add the capability to prove that all targeted resources have been successfully deleted. The completion workflow must not be considered proof of successful deletion without a verification process."_ Today, the workflow's own success report (`status: "completed"`) is the only signal that a deletion succeeded — it is generated by the same code path that performed the deletion, not independently confirmed.

## Problem Statement

The account-deletion workflow reports `completed` based on: an atomic database transaction returning without error, and Storage adapter `delete()` calls returning `success: true`. Neither is independently re-checked. The Registry already reserves a `verificationAdapter` field (populated `null` on all 37 entries, `registry.ts`) and the workflow already reserves three verification states (`verify_database`, `verify_storage`, `verify_external`, `workflow.ts`) in its state machine — both provisioned by Phase 1/3 specifically for this future need, and both currently no-ops. Phase 4 must decide how to populate them without duplicating or destabilizing the Phase 1–3.1 foundation that Phase 3.1's Production Readiness Report explicitly said should not need to be revisited.

## Design Goals

1. Every resource the Registry lists as deleted must be independently re-checked, not inferred from the deletion step's own report — mirroring the pattern `StorageBucketAdapter.verify()` already established (re-list, don't trust `delete()`'s return value).
2. Verification evidence must survive the deletion it documents — the whole point of "auditable evidence" is that it outlives the data it attests to.
3. No new persistence model, registry, or adapter contract is introduced unless the existing ones (Registry, `DeletionAdapter`, `deletion_requests`/`checkpoint`) are demonstrably insufficient.
4. Deletion success and verification success must become distinguishable, not conflated, in the workflow's final observable status and in whatever the caller (`POST /api/emma/gdpr`, and therefore its clients) is ultimately told — a verification failure must be capable of changing that outcome, not just being logged alongside an unaffected `"completed"` result.
5. Zero behavior change to the untouched foundation: the Registry's existing entries, the transactional delete function, and the `DeletionAdapter` interface are not modified in shape — only extended (new field values, one new function, real bodies for already-declared step functions).

## Constraints

Carried directly from the roadmap's own Phase 4 scope and the roadmap's governance rules:

- Out of scope: grace period, delayed deletion, reconciliation automation, operator dashboard, monitoring dashboard, scheduling, production metrics (roadmap, Phase 4 "Out of Scope").
- Must not modify the Roadmap, ADR-0001 through ADR-0004, or any existing TDD (Phase 4A's own constraints).
- Must not introduce parallel registries, duplicate workflow states, redundant adapters, or persistence models that duplicate existing responsibilities (roadmap Constraints).
- Extension is prioritized over replacement; every "Create New" decision requires technical justification (roadmap Constraints; Phase 4A Reuse Validation, discovery report §7).
- No implementation, SQL, migrations, or TDD-level interface definitions may be produced at this stage — this ADR records the decision, not the design.

## Alternatives Considered

Full evaluation (complexity/maintainability/extensibility/operational cost/compatibility/production risk for each) is in the discovery report §9. Summarized here:

**A. Registry-and-Workflow-integrated verification** — populate the Registry's existing `verificationAdapter` field, add one new read-only Registry-parameterized SQL function for database resources (mirroring `delete_user_owned_data_ordered`'s safety pattern), implement real logic in the already-reserved `stepVerifyDatabase`/`stepVerifyExternal` workflow steps, and write evidence into the existing `checkpoint jsonb` array. **Selected.** Confirmed correct by Independent Review (Phase 4A) — not reopened by either subsequent revision.

**B. New, separate `verification_results` table and standalone verification module**, decoupled from `deletion_requests`/`checkpoint`. **Rejected** — duplicates a persistence responsibility the Registry and `deletion_requests` were already deliberately built to hold (the `verify_*` states were reserved in the _same_ status enum as the deletion states from Phase 1 onward), and directly conflicts with the roadmap's own anti-duplication constraint and its declared dependency on `deletion_requests`.

**C. Fully decoupled, out-of-band verification**, independent of `runDeletionWorkflow`'s execution model, triggered separately per user or on a schedule. **Rejected as a full alternative** — scheduling triggers are explicitly Phase 5 scope, not Phase 4, and a second, less-exercised invocation path would add review surface without adding capability Phase 4 actually needs. **Its core principle is preserved inside Alternative A**: verification must independently re-check real state rather than trust the deletion step's own report — behavioral independence from the deletion step's self-reported success, not architectural decoupling from the workflow.

## Chosen Architecture

Extend, in place, already-existing extension points. Items 1–4 are unchanged since the original Architecture Discovery; items 5–7 were added during the Phase 4A.1 revision to close Independent Review findings (see the Revision Report for the finding-by-finding rationale — not repeated here).

1. **Registry (`registry.ts`):** populate `verificationAdapter` on database resource entries (a label, mirroring how `deletionAdapter: "legacy-table-delete"` is a label rather than a literal adapter object, per ADR-0004's rationale for why database resources don't use the `DeletionAdapter` object pattern). No new Registry fields.
2. **A new, read-only, Registry-parameterized SQL function** — the verification counterpart to `delete_user_owned_data_ordered`: given a user and the Registry's table/column list, returns remaining-row counts per table, with the same identifier-validation and per-column-type-casting discipline Phase 2.1's live validation already proved necessary. The one genuinely new component in this architecture — justified in the discovery report §7 (no existing read-only, multi-table, Registry-driven query mechanism exists; mutating the delete function to add a "dry run" mode would conflate two risk profiles). **Must be callable as a single batched call across all database resources** — see the Production Impact Summary.
3. **Workflow (`workflow.ts`):** `stepVerifyDatabase` and `stepVerifyExternal` — already-declared, currently-placeholder functions — get real bodies. No new states are added to `STATE_ORDER`; `verify_database`, `verify_storage`, `verify_external` are already correctly positioned there.
4. **Evidence (`deletion_requests.checkpoint`):** verification results are written as real `CheckpointEntry` rows in the same array deletion progress already uses, replacing today's `"skipped"` placeholders. Viable specifically because `deletion_requests` is not itself one of the Registry's 32 deletion targets and `auth.users` is never deleted by this flow (discovery report §4, §8 — Assumptions 1–2) — a `deletion_requests` row, and its checkpoint evidence, structurally outlives the user data it attests to.
5. **Workflow outcome authority:** a verification failure **must** be capable of changing the workflow's final `status`, and therefore `route.ts`'s `success` field (`route.ts:111`) and every client that reads it. `workflow.ts`'s `CRITICAL_STEPS` (`workflow.ts:342`) does not include the verify phases today, so this does not happen automatically merely by giving the verify steps real bodies (item 3). Phase 4B **must** extend the workflow's status-derivation mechanism so a verification failure is load-bearing for the final outcome. The exact mechanism is a Phase 4B (TDD, HOW-altitude) decision — the Roadmap's Governance section states technical details must reside in the ADR and TDD, not the roadmap itself; that one must exist and be binding is decided here (WHY-altitude).
6. **Runtime duration:** database verification **must** execute as one batched call across all resources (mirroring how `delete_user_owned_data_ordered` itself is one call, not 32), not a per-table loop — this constrains the shape of item 2's function. Phase 4B **must** also evaluate the route's serverless timeout headroom before implementation — see the Production Impact Summary for current state.
7. **Resume safety:** the verify steps **must** use the same `isPhaseCompleted()` skip-guard pattern already used by `stepDeletingDatabase`/`stepDeletingStorage` (`workflow.ts:223`, `:243`), so a workflow interrupted mid-verification and resumed does not re-run completed work or duplicate checkpoint entries. Reuses an existing, already-proven pattern.

Whether `CheckpointResourceStatus`'s three-value vocabulary (`completed`/`failed`/`skipped`) needs a verification-specific extension, or whether disambiguating via the existing `phase` field (`verify_*` vs. `deleting_*`) is sufficient, is a Phase 4B type-shape decision, separate from item 5's requirement that verification results be load-bearing for the final status.

## Trade-offs

- The one new SQL function inherits the same class of risk Phase 2.1 found real, mock-invisible bugs in for the delete function (type/casting edge cases across non-`uuid` ownership columns) — argues for live-database validation before production reliance, following the same discipline Phase 2.1 and Phase 3.1 already established.
- Extending `checkpoint` rather than introducing a dedicated verification table means verification results are queryable only per-user, not by SQL aggregation across users. Accepted for Phase 4 — the roadmap excludes operator/monitoring dashboards from this phase's scope. See Future Considerations if Phase 7 later needs this.
- Verification adds real DB/Storage round-trips to an already-synchronous request. Batching and timeout evaluation are binding requirements (Chosen Architecture item 6), not discretionary — see the Production Impact Summary for the current evidence this is grounded in.
- OAuth and background-job resources remain unverifiable under this architecture until they have real deletion adapters — no phase on the current frozen roadmap is assigned to build these (see Open Questions).

## Consequences

**Positive:**

- Zero new persistence models, registries, or adapter contracts — every extension point was deliberately provisioned by ADR-0004 for this purpose.
- Verification evidence is durable by construction — no new table or lifecycle question about when it's created relative to `deletion_requests` rows.
- The re-verify-don't-re-trust principle (Alternative C, folded into A) catches a real class of bug the deletion step's own self-report cannot.

**Trade-offs:** see above — cross-user queryability deferred past Phase 4; the new SQL function needs the same live-validation discipline the delete function required; OAuth/background-job verification stays blocked with no assigned owner phase.

**Process consequence:** this is the first ADR in the account-deletion effort written _before_ any implementation exists for it, rather than retroactively — the outcome ADR-0004's own "Process consequence" section argued for.

## Future Considerations

Future work and dependencies only — no implementation guidance:

- If Phase 6 (Reconciliation) or Phase 7 (Production Operations) need cross-user or cross-time queryability of verification results beyond what per-user `checkpoint` evidence provides, that is a future ADR's decision, not a reason to build it into Phase 4.
- OAuth (`oauth.client_integrations`) and background-job (`background.document_process`) deletion adapters remain unbuilt and unassigned to any named phase on the current roadmap. How verification eventually extends to them is a future ADR's decision.
- The evidentiary standard question (existence-check vs. something stronger) is a product/legal decision, not resolved by this ADR (see Open Questions).
