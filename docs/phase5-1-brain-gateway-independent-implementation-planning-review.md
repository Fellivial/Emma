## 1. Independent Review Report

### 1.1 Executive Summary

Phase 5's Implementation Planning is a substantively sound, evidence-traced, low-risk execution plan that faithfully translates the nine Accepted ADRs and the Phase 4 Technical Design into six independently reviewable implementation waves. Every planned task independently re-traces to a Technical Design section, an ADR Decision, and (through it) a Phase 2 gap, with no orphaned task and no task inventing an interface, schema, or behavior beyond what Technical Design already specified (§2, §3, §9). Repository state was independently re-verified against every load-bearing factual claim Phase 5 relies on — the 65-file test count, the 551-line/three-file Gateway, the still-present `529` entry in `errors.ts`'s shared retry list, the still-unconditional `OPENROUTER_API_KEY` requirement, the absence of `registry.ts`/`routing.ts`/`context-pipeline.ts`/the Governance artifact, and the `document_chunks`/`match_document_chunks` precedent Wave 6E's schema design mirrors — and all matched exactly (§1.5), confirming Phase 5's planning assumptions still hold as of this review.

**One Major finding was identified and corrected directly in this review.** Independently reconstructing the wave dependency graph from the ADR texts alone (§9, per this review's mandate not to reuse Phase 5's own graph) surfaced that the Provider-boundary `no-restricted-imports` lint rule — planned by Phase 5 as Wave 6F, Task 4 — has no technical dependency on any of Waves 6A through 6E, a fact Phase 5's own text already half-discloses ("exists independently of 6A but is grouped here per the Roadmap's own wave naming"). Scheduling it last means the exact CI-enforced protection ADR-0013 was selected specifically to provide — closing GAP-10's own stated failure mode, "nothing would catch a violation introduced by this very roadmap's own later implementation phases" — is itself absent for four of the five waves (6B–6E) it exists to guard. This is a self-referential architecture-drift exposure precisely inside the domain this review was tasked to assess (§12), not a hypothetical one. It has been corrected directly: the task is relocated to Wave 6A (§5.1 of the Phase 5 document), where it has zero dependency and can ship as soon as the Registry it is conceptually paired with exists.

Five Minor findings and three Editorial findings were also identified, none of which change any wave's scope, any ADR's decision, or the plan's overall readiness — they are dependency-labeling precision issues, a testing-strategy coverage gap, and a rollback-strategy disclosure gap (§8). No Critical finding was identified. **Recommendation: Approved with Minor Revisions — the implementation plan, as corrected, is Ready for Phase 6 (§14).**

## 8. Review Summary

### 8.1 Findings, classified

**Critical:** None found.

**Major:**

**M-1. Provider-boundary lint rule scheduled in the wrong wave, defeating its own governing ADR's rationale for four of five intervening waves.** Phase 5 originally planned the `no-restricted-imports` Provider-boundary rule (Technical Design §11.1) as Wave 6F, Task 4 — the very last wave. Independent reconstruction of the wave dependency graph (§9) confirms this task has **zero** technical dependency on Waves 6A through 6E: the `providers/` directory it protects already exists in the repository today (confirmed by direct directory listing), and Phase 5's own prose already conceded as much ("this task has no hard dependency on 6A/6B/6C's code... and may be sequenced first within this wave") without acting on that observation. ADR-0013's own stated rationale for selecting this exact mechanism is to close GAP-10 — "nothing would catch a violation introduced by this very roadmap's own later implementation phases" — and scheduling the rule itself as the _last_ of those later phases leaves that protection absent for Waves 6B, 6C, 6D, and 6E, the four waves most likely to introduce exactly the kind of import-boundary violation the rule exists to catch. Evidence: Phase 5 §5.6 (original Task 4), §7 (Dependency Validation row already conceding no hard dependency), ADR-0013 Context/Decision text. **Corrected in this review:** the task is relocated to Wave 6A (Phase 5 §5.1, new Task 6), where it ships alongside the Registry it is conceptually paired with and protects Waves 6B–6E as they are built, not only after they are already merged. Cross-references updated in Phase 5 §3 (Wave 6A/6F overviews), §4 (dependency graph), §7 (dependency validation table), §10 (testing strategy table), §12 (Traceability Matrix, GAP-10 row), and §13 (Milestone Checklist).

**Minor:**

**Min-1. Wave 6F's dependency on 6D is real but overstated to the whole wave rather than scoped to one task.** Of 6F's (post-correction) seven tasks, only the DI-lint-rule-extension task genuinely requires Wave 6D to have shipped (per Technical Design §11.1's "not retroactively invented now for modules that do not exist yet"); the remaining six tasks (correlationId, instrumentation wrapper, boot validation, Governance-artifact content, and both test tasks) have no dependency on 6D at all. Phase 5 §3/§4 label the whole wave as depending on 6D. This does not change wave order (6F is scheduled last regardless, satisfying the one real sub-dependency), but the graph's precision could mislead a reader into thinking more of 6F is blocked on 6D than actually is. Evidence: Phase 5 §5.6 (task list), §4 (dependency graph). Recommended correction: note in §4 that the 6D dependency attaches to one specific task, not the wave as a whole (informational; not applied in this review, since it does not change any conclusion or the mandated sequential order).

**Min-2. Rollback Strategy does not disclose the cross-wave file-overlap risk for `gateway.ts`/`types.ts`.** Both files are independently modified by three separate waves (6A, 6C, 6F). Phase 5 §9's "Revert PR" framing for 6A and 6C implies a clean, automatic, single-command revert regardless of when the defect is discovered. If a defect in 6A or 6C is found only after 6F has already merged (which also touches both files), a `git revert` of the earlier wave's PR may require manual conflict resolution rather than being fully automatic. This does not invalidate wave-independence as a property (the underlying interface changes remain additive and the revert remains conceptually straightforward), but it is an undisclosed mechanical caveat. Evidence: `src/core/brain/gateway.ts`/`types.ts` file-touch lists in Phase 5 §5.1/§5.3/§5.6. Recommended correction: add a one-line caveat to §9.1 noting this.

**Min-3. Testing Strategy (§10) has no dedicated Migration Validation axis.** Wave 6E's backfill plan (§5.5) already specifies real migration-safety controls, but §10's table has no column or row cross-referencing them as a testing-strategy checkpoint — a reader consulting only §10 would not see migration validation is covered. Evidence: Phase 5 §10 table structure vs. §5.5 content. Recommended correction: add a note to §10 (or a dedicated row) cross-referencing §5.5's cutover criteria as Wave 6E's migration-validation gate.

**Min-4. No boundary/DI lint-rule task is planned anywhere in Phase 5 for `src/core/brain/routing.ts`.** ADR-0013's own principle — "every new component this Freeze introduces will need its own enforcement rule authored" — arguably extends to the Routing Engine Wave 6C introduces, but Phase 5 names only the Provider boundary (now 6A) and, later, Context/Prompt Pipeline boundaries (6F) as lint-rule targets. Practical exposure is lower than the Provider-boundary gap (Finding M-1) since `routing.ts` has exactly one caller (`gateway.ts`) today, but the omission is real. Evidence: Phase 5 §5.6 (lint-rule tasks), ADR-0013 Consequences ("coverage only as complete as the rules authored"). Recommended correction (not applied, recorded for a future editorial pass): add a `no-restricted-imports` entry for `routing.ts` to Wave 6F's lint-rule-extension task.

**Min-5. ## 13. New Mandatory Rule — Architecture Compliance Checklist

Per this phase's brief, starting with Phase 6A and continuing through Phase 6F, every implementation wave must end with the Architecture Compliance Checklist below, in addition to its own Definition of Done. This checklist has been added directly to the Phase 5 Implementation Planning document (new §6.1, `docs/phase5-brain-gateway-implementation-planning.md`) so that Phase 6 contributors find it in the same place as the rest of each wave's completion criteria, rather than only in this review document.

```
□ Implementation follows all governing ADRs
□ Technical Design remains unchanged
□ No architectural shortcuts introduced
□ No permanent TODO/FIXME left behind
□ No temporary implementation committed as final behavior
□ No provider-specific logic leaked outside the Provider Layer
□ No Gateway boundary violations
□ No undocumented design changes introduced
□ All public interfaces conform to the approved Technical Design
□ All new components preserve dependency direction
□ Rollback remains possible without affecting previous waves
□ Wave Definition of Done satisfied
```

If any checklist item fails, the implementation wave must not be marked complete until the issue is resolved. Several items above map directly to specific exposures this review's Implementation Drift Risk Assessment (§12) identified as live during Phase 6 execution — provider-specific-logic leakage, Gateway boundary violations, and temporary-implementation permanence in particular — so this checklist functions as the wave-local enforcement mechanism for risks this review found were otherwise only detectable through ad hoc code review.
