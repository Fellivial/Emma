## Critical (must fix)

### C-1 — `approvals` INSERT violates NOT NULL constraints — ✅ Fixed

**File:** `src/core/agent-loop.ts` → `createApproval()` (~line 874)
**Schema:** `supabase/schema.sql` table `approvals`

`approvals.client_id` is `NOT NULL` and `approvals.tool_name` is `NOT NULL`, but `createApproval()` inserts without either field. Every call to `createApproval()` fails with a DB constraint violation, silently returning `null`. This means:

- No approval record is ever created in the DB
- The Realtime broadcast and push notification are never sent (guarded by `if (approvalId)`)
- The agent task gets stuck with `status: 'awaiting_approval'` but no record exists for the user to act on

**Fix:** Added `client_id: task.clientId`, `tool_name: toolName`, and `tool_input: toolInput` to the `createApproval()` insert payload in `agent-loop.ts`.

### C-2 — `action_log` status constraint violated by three agent-loop values — ✅ Fixed

**File:** `src/core/agent-loop.ts` (~lines 499–703)
**Schema:** `supabase/schema.sql` table `action_log`

The `action_log.status` CHECK constraint only allows `('pending', 'running', 'completed', 'failed', 'awaiting_approval', 'approved', 'rejected')`. Agent-loop.ts inserts three values not in this list:

- `'injection_detected'` (~line 703)
- `'skipped_low_autonomy'` (~line 507)
- `'moderate_executed'` (~line 511)

Each insert fails with a constraint violation (silently swallowed via `void supabase...`).

**Fix:** Extended the `action_log` CREATE TABLE constraint and added an idempotent `ALTER TABLE DROP CONSTRAINT / ADD CONSTRAINT` block at the bottom of `schema.sql` covering all three new values.

### C-3 — `google_drive` missing from `client_integrations` service constraint — ✅ Fixed

**File:** `src/core/integrations/adapter.ts`, `src/core/integrations/google.ts`, OAuth routes
**Schema:** `supabase/schema.sql` (line 796–800)

The final `client_integrations_service_check` constraint (applied by ALTER TABLE at the bottom of schema.sql) includes `gmail`, `google_calendar`, `slack`, `notion`, `hubspot`, `elevenlabs`, and `mcp_%` — but NOT `google_drive`. Any insert of a `google_drive` integration row will be rejected.

**Fix:** Added `'google_drive'` to both the CREATE TABLE inline constraint and the idempotent ALTER TABLE constraint block in `schema.sql`.

## High (should fix soon)

### H-1 — `tasks.step_transcript` and `tasks.started_at` never persisted — ✅ Fixed

**File:** `src/core/agent-loop.ts` (lines 292, 563, 627)
**Schema:** `supabase/schema.sql` table `tasks`

The agent loop writes `.update({ started_at: ..., step_transcript: messages })` at multiple points, but neither column existed in the `tasks` table. PostgREST silently ignores unknown columns. The conversation transcript needed for task resume after an approval pause was never stored — resumed tasks started with an empty message history.

**Fix:** Added `alter table public.tasks add column if not exists started_at timestamptz` and `alter table public.tasks add column if not exists step_transcript jsonb` to `schema.sql`.

---

### H-2 — STT route: no audio file size validation — ✅ Fixed

**File:** `src/app/api/emma/stt/route.ts` (line 55)

Only checks `audio.size === 0`. OpenAI Whisper rejects files larger than 25 MB with a 413 error, propagating as a generic "Transcription failed" 502. A large upload is forwarded blindly to OpenAI.

**Fix:** Added `MAX_AUDIO_BYTES = 25 * 1024 * 1024` guard after the empty-check, returning a clear 413 "Audio file too large" before the OpenAI call.

---

### H-3 — Push notification: non-410/404 errors silently swallowed — ✅ Fixed

**File:** `src/lib/push-notify.ts` (lines 51–55)

The `catch` block only handles status 410 and 404. Any other error (429 rate limit, 400 bad request, network timeout) is silently swallowed with no logging. Failed pushes are invisible in production.

**Fix:** Added `else { console.error(...) }` branch for unexpected errors. Also wrapped `webpush.setVapidDetails()` in `initVapid()` with try/catch to prevent an uncaught exception from a malformed VAPID key crashing the notification path (L-2 fix combined here).

---

### H-4 — Memory DB: corrupt decryption sentinel values silently injected into system prompt — ✅ Fixed

**File:** `src/core/memory-db.ts` → `rowToMemoryEntry()` (line 399)

`decrypt()` returns `"[decryption failed]"` or `"[encrypted — key missing]"` on failure rather than throwing. `rowToMemoryEntry` passes these sentinel strings directly into `MemoryEntry.value` without filtering. They are then injected verbatim into the system prompt via `personas.ts`, potentially confusing the model.

**Fix:** Added a `DECRYPT_SENTINELS` set in `memory-db.ts`. `rowToMemoryEntry` now returns `null` for any matching value and logs a warning with the entry ID. `getMemoriesForUser` filters nulls via `.filter((e): e is MemoryEntry => e !== null)`.

---

### H-5 — Memory DB: no per-user memory count limit — ✅ Fixed

**File:** `src/core/memory-db.ts` → `addMemoryForUser()`

No cap on `active` memory entries per user. Over time, unbounded growth causes increasingly large system prompts and higher per-request token costs — eventually exceeding the model's context window.

**Fix:** Added `MAX_ACTIVE_MEMORIES = 200` cap at the top of `addMemoryForUser()`. When the user is at capacity, the oldest lowest-confidence active entry is soft-deleted (status → `'superseded'`) before the new insert proceeds.

## Medium (worth fixing)

### M-1 — OAuth adapter: most tool handlers bypass `callWithTokenRefresh` — ⚠️ Deferred

**File:** `src/core/integrations/adapter.ts` → `getIntegrationTokens()` (line 87–89)

When a token is expired and a `refresh_token` exists, `getIntegrationTokens()` throws `IntegrationAuthExpiredError` without attempting a refresh. Tool handlers that call `getIntegrationTokens()` directly (not via `callWithTokenRefresh()`) surface this as a generic tool error rather than a user-friendly "please reconnect" message.

**Deferred:** Requires auditing and updating every tool handler in `tool-registry.ts` individually. Scope is broad; no data loss risk — users simply see a tool error and can reconnect manually. Tracked for a dedicated integrations polish pass.

---

### M-2 — Agent loop: no code-level infinite-tool-call guard — ℹ️ Acceptable

**File:** `src/core/agent-loop.ts`

The system prompt says "if stuck after 2 attempts, complete with an error summary" — a soft instruction to the model, not a code guard. If a tool always succeeds but doesn't advance the goal, the loop runs all `maxSteps` iterations. The `maxSteps` cap (default 10) is the only real guard.

**Acceptable:** `maxSteps` (default 10) provides a hard ceiling. A code-level identical-call detector would require serialising tool inputs for comparison and could false-positive on legitimately repeated calls (e.g. paginated search). The model instruction is sufficient for normal operation; the rate-limiter prevents runaway cost even in degenerate cases.

---

### M-3 — Agent loop: approval resume does not inject the approval decision as a tool result — ⚠️ Deferred

**File:** `src/core/agent-loop.ts` → resume via `task.resumeMessages`

When a task resumes after approval, the saved transcript is replayed but the approval decision (approved/rejected) is never fed back into the tool result slot for the paused tool call. The model resumes without knowing the outcome of the pending action.

**Deferred:** Requires changes to the approval API route (`/api/emma/agent/route.ts`) to pass the decision back and to the loop resume logic to inject a synthetic tool result. Non-trivial multi-file change; no data loss. Tracked for the next agent-loop iteration.

---

### M-4 — `getTokenRow` decryption failure produces a misleading auth error — ℹ️ Acceptable

**File:** `src/lib/oauth-refresh.ts` → `getTokenRow()` (lines 102–103)

If `EMMA_ENCRYPTION_KEY` is not set, `decrypt()` returns `"[encrypted — key missing]"`. The guard catches this and throws `IntegrationExpiredError`, surfacing "requires re-authorization" to the user — the real cause is a missing server env var, not an expired token.

**Acceptable:** A missing `EMMA_ENCRYPTION_KEY` in production is an operator error that should be caught at deploy time, not at runtime. The misleading message only appears in that misconfigured state. Adding a specific sentinel check would complicate the hot path for a scenario that should never occur in a correctly deployed instance.

---

### M-5 — `usage-enforcer.ts`: read-then-write race condition — ℹ️ Acceptable

**File:** `src/core/usage-enforcer.ts`

`checkUsage()` reads the window, then `recordUsage()` increments it separately. Two simultaneous requests can both pass the check and both exceed the limit. This is an intentional fail-open design trade-off, but worth noting as a known gap.

**Acceptable:** Documented behaviour — the module header explicitly states "Enforcement MUST fail open." A race allows at most 1–2 extra messages at the boundary, which is preferable to blocking legitimate requests due to a DB timing issue. Eliminating the race would require a compare-and-swap RPC, adding latency to every brain call.

**File:** `src/lib/push-notify.ts` → `initVapid()` (line 22)
