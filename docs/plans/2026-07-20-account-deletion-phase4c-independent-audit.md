# Account Deletion — Phase 4C Independent Production Audit

**Status:** Complete. **Verdict: GO WITH MINOR FOLLOW-UPS.**
**Written:** 2026-07-20.
**Auditor independence:** Fresh-context review with no authorship stake in the roadmap, ADR-0004, ADR-0005, the Phase 4B TDD (or any of its three revisions), the four-round independent review chain, or the [Phase 4C Production Readiness Review](2026-07-20-account-deletion-phase4c-production-readiness.md) (the "PRR") this document was commissioned to challenge. Nothing in the PRR, the TDD, or either ADR was taken at face value — every load-bearing claim cited below was independently re-verified against current repository state this session (source reads, a live test run, and direct greps against `vercel.json`, `.github/workflows/ci.yml`, and `package.json`), not merely re-read from a prior document's prose.

---

## 1. Mandate

This is a production-approval-board audit of whether **implementation** of the Phase 4 (Verification) design — ADR-0005 (Accepted) plus the Phase 4B Technical Design Document (Accepted, Revision 3, 2026-07-20) — should be allowed to **begin**. It is explicitly not a review of running production code: no Phase 4 verification code exists anywhere in this repository. The question is whether the design package (roadmap + ADR-0004 + ADR-0005 + the Phase 4B TDD + its four-round review chain + the PRR synthesizing all of it) is honest, complete, and safe to act on — not whether the TDD's internal state-machine logic is correct. That question was already tested adversarially, four times, by reviewers with no memory of each other's rounds, and I did not attempt to re-derive it from scratch. Where I re-checked their conclusions, it was to confirm the conclusions still hold against current `HEAD`, not to re-litigate the mechanism.

## 3. Findings

### MAJOR-1 — The PRR's own Risk Register omits two implementation-fidelity risks the TDD explicitly names as capable of reintroducing the CRITICAL/Major defects the review chain just spent four rounds closing

The Phase 4B TDD's own §14.1 ("Implementation risks") table names two risks in these exact words:

> "The `db.verification-batch`/`external.verification-batch` synthetic marker convention is a new pattern; a Phase 4C implementer could plausibly miss it and use per-resource guards instead (the 'naturally symmetric' but wrong choice, §4.4)"

> "(Revision 3) §7.1's retry-deduplication algorithm is a new pattern with a non-obvious failure mode: a naive implementation that applies 'keep latest `recordedAt`' uniformly, without §4.2's corresponding guard-body fix, silently reintroduces the exact Storage undercount defect this revision closes"

Both risks are structurally different from anything already in the design: they name specific, plausible implementation choices that would silently regress the codebase back to the exact CRITICAL and Major defects Revisions 1 and 3 fixed — not "an ambiguity," but "a correct, unambiguous spec that is easy to accidentally revert because the codebase's own existing precedent (`stepDeletingStorage`'s push-a-placeholder guard, `stepDeletingDatabase`'s per-resource-shaped guard) points the other way." Yet the PRR's Deliverable 3 Risk Register (9 items) contains no risk matching either of these. The closest entries — Risk 6 ("An `'inconclusive'` `verify_storage` entry permanently satisfies its own resume guard") and Risk 9 ("No live-database validation has occurred") — are both different risks (Risk 6 is about a disclosed, accepted design trade-off; Risk 9 is about the SQL function specifically, not the checkpoint/dedup logic).

This is a completeness gap in the PRR's own synthesis, not a fabrication: the underlying mitigation already exists (TDD §14.2 mandates the exact regression tests that would catch both — the "resume with marker present and a prior failed entry" test, and the scenario-16/scenario-17 retry-then-resolve tests). But a Risk Register whose entire purpose is to surface risk for tracking should not silently drop risks its own source document names as capable of reintroducing a data-privacy-relevant defect. **Not blocking implementation start** — the mitigation (mandatory regression tests) is already a binding TDD requirement independent of whether the Risk Register lists it — but should be added to the Risk Register (or an equivalent tracked item) before Phase 4C's PR is reviewed, specifically so a PR reviewer checking the Risk Register alone would still be prompted to verify these two behaviors, not just the ones currently listed.

### MAJOR-2 — No deployment-time enforcement of the TDD's own "binding" migration-before-code sequencing requirement, and no kill switch for a compliance-critical path

TDD §14.5 states migration-before-code sequencing is "a binding requirement, not a preference," and §9 scenario 13 states its inversion is a **silent** failure mode (the RPC-not-found error degrades to an `"inconclusive"` checkpoint entry, not a hard error, per §5.4's deliberate design). I independently verified the actual deployment mechanics this requirement depends on, which none of the prior documents checked:

- `.github/workflows/ci.yml:79-93` — the `deploy-production` job runs on every push to `main` (`if: github.ref == 'refs/heads/main' && github.event_name == 'push'`), with no manual approval gate and no step anywhere in the pipeline that applies a Supabase migration. `package.json:5-16`'s `scripts` block has no migration-related script (`db:push`, or similar) at all.
- A repo-wide grep for feature-flag infrastructure (`featureFlag`, `feature_flag`, `growthbook`, `LaunchDarkly`) returns nothing. There is no mechanism in this codebase to disable a shipped code path short of a full redeploy.

Put together: the one binding sequencing requirement this design depends on for correctness is enforced by **human discipline alone**, checked by nothing in CI or the deploy pipeline; the failure mode if that discipline lapses is **silent** (no alert exists — Review Area 7 of the PRR itself confirms "Alerting: None added — explicitly out of scope"); and there is no fast way to disable verification if it misbehaves in production short of a full code revert-and-redeploy. This is a materially different risk profile from Phase 2's migration (cited by the TDD as "same risk class") — Phase 2's migration gated the _delete_ RPC, whose absence throws loudly (`deleting_database` is already `CRITICAL_STEPS`, so a missing function hard-fails the request in a way an engineer would notice within one support ticket). Phase 4C's migration gates a verify RPC whose absence is invisible unless someone is actively reading `verification.database.inconclusive` counts that nothing currently surfaces or alerts on.

A second, related point: widening `CRITICAL_STEPS` to include `verify_storage` (TDD §5.1, correctly closing the Independent Review's finding that a genuine Storage leftover was previously silently ignored for outcome purposes) means that, for the first time, a Storage-side problem can drive a `deletion_requests` row to permanent `status: "failed"` — a state with no recovery path today (`workflow.ts:406-423`'s explicit "not auto-restarting" terminal branch, and no retry/cancel endpoint exists anywhere in this codebase). Combined with the PRR's own honest admission (Review Area 7) that operational tooling to investigate such a stuck row is "Per-user only... insufficient for a fleet-wide operational incident" and correctly scoped to Phase 7 — this means Phase 4C is the point at which real users can, for the first time, hit a permanent, un-self-serviceable "contact support" dead end for a GDPR erasure request, with no support runbook described anywhere in the roadmap, ADRs, TDD, or PRR for what "support" is supposed to do when that ticket arrives.

**Not blocking implementation start** (writing the Phase 4C code, SQL, and tests against the accepted TDD does not require these resolved) — but this should be resolved or explicitly, consciously accepted **before Phase 4C's PR is allowed to merge to `main`**, given `main`'s push triggers an unconditional production deploy. At minimum: add a migration-applied verification step to the deploy pipeline (or a manual pre-merge checklist item enforced by the PR template, not merely documentation prose), and decide explicitly whether a support runbook is needed before this ships, rather than discovering the gap from a real user's support ticket.

### MINOR-1 — The PRR's Review Area 8 conclusion overstates what "no ambiguity" establishes

Review Area 8 concludes: "No missing specification, ambiguous contract, undefined interface, or unstated implementation assumption was found that the accepted TDD does not already resolve." Taken narrowly (is every decision point resolved in the document?), this is true and I did not find a counterexample. But it elides the distinction MAJOR-1 above turns on: a specification can be complete and unambiguous on the page while still being easy for a competent implementer to get wrong in practice, precisely because the "correct" choice diverges from this same file's own established convention at two adjacent, structurally similar call sites (`stepDeletingStorage`'s guard vs. `stepVerifyStorage`'s corrected guard; `stepDeletingDatabase`'s per-resource-shaped skip guard vs. `stepVerifyDatabase`'s required aggregate-marker guard). The TDD itself is honest about this risk (§14.1, quoted in MAJOR-1); the PRR's synthesis of "Implementation Readiness" does not carry that nuance forward into its own conclusion. This is a phrasing/completeness issue in the PRR, not a design defect.

### MINOR-2 / OBSERVATION — The append-only-`jsonb`-plus-read-time-dedup pattern is the root cause of three of the four review rounds' findings, and will remain a live hazard for Phase 6

Reading the four-round history end to end: the CRITICAL finding (unconditional marker status), the first Major finding (DB rollup summing across retry history), and the second Major finding (Storage rollup losing evidence to a placeholder) are three different surface manifestations of the same underlying shape: `checkpoint` is an **immutable, append-only array with no natural key**, so "what is the current, authoritative state of resource X" is not a stored fact — it has to be re-derived, correctly, at every read site, by hand, in TypeScript, via a "keep the latest entry per key" reduction that has no structural enforcement. A table with a real `UNIQUE (request_id, phase, resource_id)` constraint and `UPSERT` semantics would have made this entire defect class structurally unreachable — there would be nothing to deduplicate because there would only ever be one row per key. This is **not** a reason to reopen Alternative B (rejected at ADR-0005 altitude for a legitimate reason — avoiding a second persistence model that duplicates `deletion_requests`'s responsibility — and I am not second-guessing that trade-off here). But it is worth recording plainly for whoever builds Phase 6 (Reconciliation), which the roadmap describes as needing to "distinguish between permanent and temporary failures" and "provide auditable reconciliation results" against this exact evidence store: the fragility this phase's design fought through three review rounds to close is a property of the storage shape, not something Phase 4C's implementation can retire once it ships. Carried forward as an observation for Phase 6/7 planning, consistent with how the PRR's own Risk 7 (checkpoint's unbounded growth) is already framed the same way.

### OBSERVATION-1 — No assigned human owner behind the "Product/Legal" and "Roadmap owner" placeholders

ADR-0005's Open Questions table names "Product/Legal" and "Roadmap owner (Emma Engineering)" as owners for two non-blocking-but-real open questions (the evidentiary-standard question; OAuth/background-job verification's future phase assignment). Neither has a linked person, ticket, or process for actually getting an answer — both are correctly marked non-blocking for Phase 4C, and I am not asking that they be resolved before implementation starts. Recorded only so the distinction between "this question is formally deferred" and "someone is actually going to answer it" is not lost, the same way ADR-0004's own "Process consequence" section already warned this subsystem's documentation has drifted from reality before.

### OBSERVATION-2 — The entire governance chain is AI-authored with no evidenced human checkpoint

The four independent reviews, the ADR, the TDD, and the PRR are all, by their own text, produced by "a fresh review agent with no prior context on this initiative" (direct quote, Third Final Independent Re-Review, line 5) — there is no evidence anywhere in this chain of a human engineer, architect, or Product/Legal stakeholder actually reading and signing off on any of it. This is consistent with, not worse than, how Phases 1–3.1 of this same feature were run (ADR-0004's own "Process consequence" section already discloses this pattern), and I independently confirmed several of the chain's specific code-level claims hold against current `HEAD` — the rigor is real, not performative. But "independently reviewed four times" should be read by this board, and by whoever approves Phase 4C's eventual PR, as "reviewed by four independently-seeded AI review passes," not "reviewed by four different human engineers." I recommend this distinction be stated plainly wherever this chain's rigor is cited as justification for reduced human review at PR time — it is a reason to trust the _design's_ internal consistency, not a substitute for a human owning the _production_ decision to ship it.

## 4. What I looked for and did not find

- **No new correctness defect** in the workflow/checkpoint/rollup mechanism beyond what the four-round chain already found and closed. I traced the `verify_storage` empty-`entries` edge case, the shared cross-phase `retry_count` semantics, and the rollback-compatibility string-comparison claim independently and none breaks.
- **No evidence the PRR misdescribes the code it cites.** Every specific claim I checked against source (registry cardinality, `CRITICAL_STEPS`'s current value, `vercel.json`'s 16-entries-none-for-gdpr claim, `CostOperation`'s exclusions, `privacy/page.tsx`'s branch structure, the 15-value status constraint) was accurate.
- **No dishonest scope-narrowing.** The PRR's central framing move — "this is an Implementation Readiness Review, not the roadmap's own Production Readiness Review, because no code exists yet to assess" — is not a dodge. I independently confirmed against the roadmap's own Engineering Workflow (step 10, "Production Readiness Review," sequenced explicitly after step 8 "Implementation" and step 9 "Hardening & Validation") that this is exactly the same sequencing precedent Phase 2→2.1 and Phase 3→3.1 both followed. The PRR's naming correction is accurate, not self-serving.
- **No evidence the review chain's convergence claim is manufactured.** I independently re-derived the Storage guard-fix's inductive argument in the Third Final Re-Review and found it holds; I did not find a retry-count interleaving or resume scenario the chain's scenario table (17 entries) misses.

## 5. Recommendation

**GO WITH MINOR FOLLOW-UPS.**

The design itself — ADR-0005 plus the accepted Phase 4B TDD — is genuinely sound. Four independent adversarial rounds tested it against the property that actually matters (`success`/`status` cannot be made to lie), converged to zero Critical/Major findings, and I did not find a defect they missed. Nothing found in this audit rises to a CRITICAL or unresolved MAJOR finding **against beginning implementation** — both MAJOR items above are about what happens between "implementation is written" and "this ships to real users," not about whether the design is safe to start building against.

**Track as follow-up work, not blockers to starting Phase 4C:**

1. Add the two TDD §14.1-named implementation-fidelity risks (synthetic-marker-convention reversion; naive-dedup-without-guard-fix reversion) to the Risk Register or equivalent tracked list, so a PR reviewer checking the Risk Register alone is prompted to verify TDD §14.2's corresponding regression tests exist and pass (MAJOR-1).
2. Before Phase 4C's PR is allowed to merge to `main` (not before implementation begins): resolve or consciously accept the lack of a deployment-pipeline check for migration-before-code sequencing, and decide whether a support runbook is needed for the newly-reachable permanent-`"failed"`-verification dead end (MAJOR-2).
3. Carry MINOR-1 and MINOR-2/OBSERVATION-1/OBSERVATION-2 forward as documentation-quality and future-phase-planning notes; none require action before Phase 4C starts.
