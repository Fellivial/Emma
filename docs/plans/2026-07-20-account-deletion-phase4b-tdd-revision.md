## 1. Why This Report Exists

The 2026-07-18 Phase 4B TDD was committed to `main` in the same commit as its own Independent Review (`1183a50`, PR #134). That review's verdict was explicit: **"not approved... Phase 4C must remain blocked."** The merge proceeded anyway — a process gap, not a decision that the finding was acceptable or that the roadmap's own validation checklist ("The design should not be considered complete until all CRITICAL findings are resolved") had been satisfied. This report, and the accompanying revision to the TDD itself, close that gap before any Phase 4C implementation work begins.

## 2. Revision Summary

The Independent Review confirmed the TDD's overall shape (every ADR-0005 Chosen Architecture item correctly translated; scope, runtime, traceability, and requirements-traceability sections all passed cleanly) and found one CRITICAL defect plus several MAJOR/MINOR gaps, all clustered around one root cause: the design's new synthetic "aggregate marker" pattern (introduced to make the resume-skip guard work for a batched, non-transactional verification call) never accounted for the marker needing to carry two different pieces of information — "did the batch execute" and "did the batch execute and find everything clean" — and collapsed them into one unconditional status value. That collapse is what let a confirmed database-verification defect become invisible on retry.

This revision does not change what the TDD builds — it corrects how three already-specified mechanisms (the database marker, the external marker, the Storage failure mapping) compute their status values, and fills in three specification gaps the review found (the external marker's write step, the required function-signature changes, the API rollup's exclusion of synthetic entries) that the original text left implicit or unstated.

## 7. Revision 2 (2026-07-20) — Findings from the Final Independent Re-Review

Revision 1 (§§1-6 above) was submitted for a [Final Independent Re-Review](2026-07-20-account-deletion-phase4b-tdd-final-review.md), performed by a fresh reviewer with no authorship stake in either the original TDD, the original review, or Revision 1 — including no trust extended to this report's own §6 Self Verification Report, which was re-derived independently rather than accepted. That review's central task was adversarial: assume Revision 1's CRITICAL fix "looks" correct and try to break it by composition, the same way the original review broke the unrevised TDD by tracing a mechanism across two calls instead of reading one section in isolation.

**Result: the CRITICAL fix holds.** The reviewer independently re-traced the full first-call/second-call retry sequence against the actual, unmodified `isPhaseCompleted()`, `resumeStartStatus()`, and `CRITICAL_STEPS`-branch code in `workflow.ts` and confirmed a confirmed database-verification defect no longer silently converges to `success: true` on retry. Three of the original review's four blocking findings were also independently reconfirmed resolved.

**But the same adversarial composition-check found a new Major defect one section over:** §7.1's `verification` API rollup, also touched by Revision 1 to exclude synthetic markers, was never adjusted for the fact that the CRITICAL fix's own retry mechanism — once it can legitimately re-run — causes `stepVerifyDatabase` to re-append a full new batch of checkpoint entries on every retry (it has no per-table skip guard, only the one aggregate marker guard, by §4.4's own deliberate design). Summing `resourceStatus` occurrences over the entire historical `checkpoint` array, as Revision 1 specified, produces counts that scale with retry count and can report the same resource as both `failed` and `verified` if it failed early and passed later. Two further Minor, purely mechanical gaps were found alongside it (§4.5 audited the step-function layer for "signature change never stated" but missed two adjacent call sites: `DeletionWorkflowResult`, which had no field for §7.1's algorithm to read from, and `checkpointEntry()`, whose factory was never extended for the `remainingCount` field §6.2 added to the type).

## 8. Revision 3 (2026-07-20) — Findings from the Second Final Independent Re-Review

Revision 2 (§7 above) was submitted for a [Second Final Independent Re-Review](2026-07-20-account-deletion-phase4b-tdd-final-review-2.md), performed by a third fresh reviewer with no authorship stake in any prior document, explicitly tasked with checking whether Revision 2's own dedup fix (for round 3's Finding A) generalizes correctly, and with assessing whether this review chain is converging or diverging.

**Result: the CRITICAL finding was reconfirmed closed for a third consecutive round**, independently re-derived rather than accepted from any prior round's trace. Round 3's Finding C (`checkpointEntry()`'s missing `remainingCount`) was reconfirmed fully resolved with no residue. Round 3's Finding A (the rollup summing across retry history) was reconfirmed genuinely fixed **for the exact case it was diagnosed against** — a phase guarded by a single aggregate marker (database, external).

**But the same adversarial composition-check found the identical dedup mechanism does not correctly handle `verify_storage`, which uses a different, per-resource skip guard.** Concretely: if one Storage bucket verifies clean on an early call and a different bucket's failure triggers a retry, the per-resource guard for the already-clean bucket fires on the retry and — under Revision 2's guard body, unchanged from the pre-existing `stepDeletingStorage` convention — pushes a fresh, later-timestamped `"skipped"` placeholder entry. Revision 2's "keep latest `recordedAt`" rule then keeps that placeholder over the genuine, earlier `"completed"` entry for the same bucket, silently downgrading a resource that was directly and successfully re-verified empty to indistinguishable-from-never-checked. This is the identical defect _shape_ Finding A itself was (a rollup losing fidelity across a retry), recurring in the one place §7.1's own Revision-2 text names as a second motivating mechanism but whose fix did not actually cover. One further Minor, purely mechanical gap was found alongside it: §4.6's `DeletionWorkflowResult.checkpoint` population rule cited 4 lines but `runDeletionWorkflow()` has 7 return statements — 3 real, reachable branches (including the everyday "already permanently failed" response) were left unaddressed.

The reviewer explicitly assessed the three-round chain's trajectory as **converging**: Round 1 found 1 CRITICAL + 3 Major + 4 Minor/Low; Round 3 (Revision 1's re-review) found 0 Critical + 1 Major + 2 Minor + 2 Observations; this round (Revision 2's re-review) found 0 Critical + 1 Major (narrower in scope than round 3's) + 1 Minor + 2 Observations. Every finding since the original CRITICAL one has been confined to the `verification` field's descriptive accuracy — never to `success`/`status`, which is the property ADR-0005 and the roadmap actually require and which has now survived three independent, adversarial attempts to break it by composition.
