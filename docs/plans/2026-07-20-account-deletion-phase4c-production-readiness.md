## A note on scope and naming, stated up front rather than left implicit

The roadmap's Engineering Workflow (13 mandatory steps) places "Production Readiness Review" (step 10) and "Independent Phase Gate Review" (step 11) **after** Implementation (step 8) and Hardening & Validation (step 9) — matching precedent: Phase 2's PRR and Phase 3's PRR were both written after code existed to assess. **No account-deletion verification code exists yet.** This review is therefore, more precisely, an **Implementation Readiness Review** for Phase 4's Technical Design — it validates that the design package (ADR-0005 + the accepted Phase 4B TDD) is complete and internally consistent enough for implementation (step 8) to begin, not that a built system is ready to serve production traffic. That later assessment — the one the roadmap's step 10 actually describes — is still owed, after implementation and live-database validation (mirroring the Phase 2→2.1 and Phase 3→3.1 precedent), before this feature can be relied on for real users. This review's task brief titles it "Production Readiness Review" and asks for a GO/NO-GO on beginning "Phase 5" (implementation); this document honors that brief's deliverables and decision framework exactly, while flagging this naming precisely so the distinction isn't lost the way ADR-0004's "Process consequence" section already warned this subsystem's documentation has drifted from reality before. Wherever "production readiness" appears below, read it as "readiness for implementation to safely begin," not "readiness to serve production traffic today."

**On "Phase 5":** the task brief's "Phase 5" means _this feature's next step_ (Phase 4's own Implementation), not the roadmap's own **Phase 5 — Grace Period & Scheduling**, which has its own explicit prerequisite ("Depends on: Phase 4") meaning Phase 4 fully _complete_ (implemented and hardened), not merely designed. This review's GO/NO-GO decision gates implementation of the accepted Phase 4B design, consistent with the roadmap's step 7→8 transition.

## Review Area 2 — Workflow Correctness

| Property                     | Finding                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Idempotency**              | The new verify RPC is a pure read (`SELECT count(*)`), idempotent by construction — no state to duplicate across repeated calls. The existing delete RPC's idempotency was already proven via two real double-execution tests (Phase 2.1 live validation) and is unmodified by this design.                                                                                                                                                                                                                             |
| **Resume safety**            | Reuses the exact `isPhaseCompleted()` skip-guard pattern already proven for `stepDeletingDatabase`/`stepDeletingStorage`. Two distinct guard shapes are used correctly for two distinct data shapes: an aggregate marker for `verify_database`/`verify_external` (one batched call needs one guard, not 32 — TDD §4.4), and a per-resource guard for `verify_storage` that, after Revision 3, writes no placeholder entry when it fires (closing the one resume-safety defect the review chain found).                  |
| **Retry safety**             | `CRITICAL_STEPS` widened to include all three verify phases, reusing the existing `MAX_RETRY_COUNT = 3` ceiling and `retry_pending`/`failed` terminal states unmodified — no new retry infrastructure. The one genuine retry-safety defect found in this design (a confirmed database defect silently escaping re-verification on retry) is the CRITICAL finding the Phase 4B review chain's first round found and Revision 1 fixed — independently re-derived and confirmed closed across three further review rounds. |
| **Deterministic execution**  | `STATE_ORDER` is unmodified. No new concurrency surface: the verify RPC is read-only, strictly lower lock-contention risk than the delete RPC it sits beside, and Phase 3.1's existing optimistic-concurrency-control mechanism (CAS on `updated_at`) is untouched and unbypassed by this design.                                                                                                                                                                                                                       |
| **Deterministic completion** | `success: result.status === "completed"` is unmodified code whose _meaning_ is now correctly strengthened: `"completed"` is reachable only if deletion and all three verify phases produced zero `"failed"` entries, traced end-to-end (including the retry-then-resolve path) across TDD §9 scenarios 15–17.                                                                                                                                                                                                           |

**No workflow-correctness defect open.** This is the single most heavily adversarially-tested property in the entire design package — four independent review rounds each specifically re-derived the marker/guard/retry mechanics from scratch against unmodified code, and none found `success`/`status` capable of being wrong after Revision 3.

## Review Area 3 — Verification Architecture

| Requirement (from this review's brief)                                  | Finding                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Verification authority is correctly defined**                         | Yes — ADR-0005 Chosen Architecture item 5 (binding) mandates it at WHY-altitude; TDD §5 implements it at HOW-altitude by widening `CRITICAL_STEPS`, reusing the existing `"failed"`-triggers-escalation branch unmodified.                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Verification can change workflow outcome**                            | Yes, demonstrated concretely, not just asserted: TDD §9 scenario 2/15/16/17 each show a confirmed verification defect converting `status: "completed"` into `status: "retry_pending"`/`"failed"`.                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Evidence is reproducible**                                            | Yes for the steady-state case (a clean re-run of `verify_user_owned_data_deleted` against unchanged data always returns the same counts — it is a deterministic `SELECT`). The per-table failure-isolation design (§3.5) means a schema-drift condition on one table doesn't erase evidence for the other 31.                                                                                                                                                                                                                                                                                                                                      |
| **Checkpoints remain authoritative**                                    | Yes — the final `status`/`success` derivation is still, and only, a function of `checkpoint` entries via the unmodified `entries.filter(e => e.resourceStatus === "failed")` / `CRITICAL_STEPS.includes(status)` branch. No verification result is ever consulted through any path other than `checkpoint`.                                                                                                                                                                                                                                                                                                                                        |
| **Retry semantics are well defined**                                    | Yes — §5.3 states the mechanism (reuse `MAX_RETRY_COUNT`, resume at the last checkpoint entry's phase), and §9 scenarios 15–17 trace it against three genuinely distinct guard shapes (aggregate-marker × 2, per-resource × 1).                                                                                                                                                                                                                                                                                                                                                                                                                    |
| **Placeholder checkpoint behavior cannot create false positives**       | **This is the exact property the CRITICAL finding violated, and the exact property Revisions 1 and 3 were built to restore.** Confirmed closed for both mechanisms in this design: the aggregate marker (Revision 1 — conditional status formula) and the per-resource guard (Revision 3 — guard fires produce no placeholder at all, closing the defect at its structural source rather than adding a status-aware exception to the dedup rule). The Third Final Re-Review's inductive proof (not just a scenario trace) establishes this holds for _any_ retry count or bucket-failure interleaving, not only the specific cases walked through. |
| **Historical retry data cannot influence current verification results** | **This is the exact property the Major finding (Revision 2) and its narrower recurrence (Revision 3) violated, and were built to restore.** The `verification` API rollup now deduplicates by keeping only the latest-`recordedAt` checkpoint entry per `(phase, resourceId)` before counting (§7.1) — confirmed to reduce to exactly one entry per verifiable resource per phase regardless of retry count, for both the aggregate-marker phases (by construction of the dedup rule) and the per-resource-guard phase (by construction of never producing more than one real entry per resource in the first place, after Revision 3).            |

**No verification-architecture defect open.** Every one of this review area's eight named properties maps directly onto a property the Phase 4B review chain already adversarially tested to convergence — this section is a confirmation, not a new derivation, and is stronger for having four independent prior attempts to break it fail to find anything beyond documentation-completeness gaps in the final round.

## Review Area 6 — Security Review

| Dimension                       | Finding                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Encryption boundaries**       | Unaffected. `checkpoint jsonb` was already unencrypted before this design (matching precedent — it stores `resourceId`/status/detail strings, not user content), and this design adds only a numeric `remainingCount` and a status enum value to that same, already-established trust boundary. No new field carries user PII: `detail`/`error` strings are operational metadata (table names, counts, Postgres error text), not user data.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Least-privilege access**      | The new SQL function is specified to inherit the _identical_ grant posture as the existing delete function: `SECURITY DEFINER`, `SET search_path = ''`, `REVOKE ALL ... FROM PUBLIC, anon, authenticated`, `GRANT EXECUTE ... TO service_role` only (TDD §3.6). No looser trust boundary is introduced for "just a read."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Auditability**                | Preserved and strengthened — this is Phase 4's entire purpose. Evidence is written into the same durable, RLS-protected substrate (`deletion_requests.checkpoint`, "Users read own deletion requests" policy, confirmed present per the Phase 4A discovery report) that already survives the deletion it documents (confirmed: `deletion_requests` is not one of the 32 Registry deletion targets; `auth.users` is never deleted by this flow).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Evidence integrity**          | This is the property the entire four-round review chain existed to establish. Confirmed at design level: a placeholder entry can no longer masquerade as real evidence (Revisions 1, 3), and historical retry generations can no longer pollute the current, reported evidence (Revisions 2, 3).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Immutable workflow history**  | Checkpoint entries are append-only by construction (`[...row.checkpoint, ...entries]` — confirmed by direct trace of `workflow.ts`'s `persist()` call sites; no code path mutates or removes an existing entry). One consequence worth naming, not previously flagged in the review chain: because every retry appends a full new generation of entries rather than replacing the prior one, `checkpoint` grows without an enforced upper bound across the _history_ even though the _current, deduplicated_ rollup stays correctly sized. Bounded today by `MAX_RETRY_COUNT = 3` (at most ~4 generations × ~34 entries ≈ 136 entries for a pathological all-phases-retry-every-time case) — not a present risk, but worth naming for whoever eventually builds Phase 6 (Reconciliation) or Phase 7 (Production Operations), since neither this design nor any prior one specifies an archival or pruning strategy for `checkpoint`'s raw history. | **GDPR compliance assumptions** | The evidentiary-standard question — whether an existence check ("0 rows remain") satisfies "auditable evidence," or whether compliance requires something stronger (e.g., retained cryptographic proof) — **remains genuinely, explicitly unresolved**, owned by Product/Legal per ADR-0005's own Open Questions table. This is not a Phase 4B design gap (the TDD is explicitly designed to be correct regardless of how this question is eventually answered, per §5.4's stated reasoning), but it is a real compliance question that implementation readiness does not, and cannot, resolve. Restated here so it is not lost in a purely engineering-framed sign-off.                                                                                                                                                                                                                                                                                              |

## Decision

### GO WITH MINOR FOLLOW-UPS

The design package — ADR-0004, ADR-0005, and the accepted Phase 4B Technical Design Document — is sound. It was adversarially tested across four independent review rounds, each with no memory of the prior round's findings, converging to zero unresolved CRITICAL or MAJOR findings against the property that actually matters: `success`/`status` cannot be made to lie about whether a user's data was really deleted, even under retry, resume, or interruption. This review's own eight review areas confirm that conclusion still holds, and the [Independent Production Audit](2026-07-20-account-deletion-phase4c-independent-audit.md) — a fifth, differently-scoped adversarial pass, specifically commissioned to challenge this report's own synthesis rather than the TDD's internal logic — found no new correctness defect and no case where this document or its sources misdescribe the code they cite.

**Implementation (Phase 4's Engineering Workflow step 8) may begin immediately** against the accepted TDD. All eight Phase 5 Entry Criteria are satisfied; no Implementation Readiness Matrix item is Blocked.

**Follow-ups — required before Phase 4C's implementation PR merges to `main`, not before implementation begins:**

1. Confirm TDD §14.2's two most safety-critical regression tests are present and passing: the "resume with marker present and a prior failed per-resource entry" test (guards against reintroducing the original CRITICAL defect), and the scenario-17 Storage retry test asserting `verification.storage.verified === 2`, not `1` (guards against reintroducing the Storage-undercount Major defect). Risk Register items 10–11.
2. Resolve or consciously accept the absence of a deployment-pipeline check for the TDD's own binding migration-before-code sequencing requirement, and decide whether a support runbook is needed for the newly-reachable permanent `status: "failed"` dead end once `verify_storage` joins `CRITICAL_STEPS` (no retry/cancel endpoint exists today). Risk Register item 12.

**Follow-ups — must be closed before this feature is relied on for real user deletions (not blocking for merge):**

3. Ops confirmation of production schema state for `document_chunks.user_id` and the three related tables (Risk Register item 1 — inherited from Phase 3.1, the single highest-impact open risk in this entire package, and not caused by this design).
4. The mandatory live-database validation pass for the new SQL function and the `maxDuration: 60` timing estimate (Risk Register items 2, 9 — expected and already scheduled by the TDD's own §14.2/§14.3, mirroring the Phase 2→2.1 and Phase 3→3.1 precedent).

**Explicitly not blocking, carried forward as documented, accepted risk or future-phase input:** the Product/Legal evidentiary-standard question, the unassigned OAuth/background-job verification scope, `recordedAt` monotonicity, the `"inconclusive"`-forecloses-recheck behavior, checkpoint's unbounded historical growth, checkpoint corruption tolerance, and the architectural observation that `checkpoint`'s append-only-no-natural-key shape is the structural root cause behind three of the four review rounds' findings and remains a live consideration for Phase 6 (Reconciliation). Risk Register items 3–8, 13.

One process note, stated plainly rather than left implicit: this entire governance chain — the ADR, the TDD, all four independent reviews, this PRR, and the Independent Production Audit — was produced by independently-seeded AI review passes, not by four different human engineers. That is a reason to trust the design's internal consistency (which was genuinely, adversarially tested); it is not a substitute for a human owning the production decision to ship real users' GDPR erasure requests through it. Whoever merges Phase 4C's implementation PR is that owner.
