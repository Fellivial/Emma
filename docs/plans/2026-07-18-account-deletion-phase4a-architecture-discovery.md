## 4. Architecture Review

The accepted baseline, restated at the altitude Phase 4 needs to build on:

- **Registry (`registry.ts`)** is the sole inventory. Every `DeletionResourceEntry` already carries a `verificationAdapter: string | null` field, populated `null` on all 37 entries today — this is not an omission Phase 4 discovers; it is deliberate over-provisioning ADR-0004 named explicitly: _"this is deliberate over-provisioning for the one property that matters most for a registry... Phase 1 populated fields those phases will need, even though only two are read by anything today."_ Verification is one of the phases that field was provisioned for.
- **The `DeletionAdapter` interface (`adapter.ts`)** already includes `verify()` as a first-class lifecycle stage, for the same reason: _"exists in the interface now so adding real verification later... doesn't require a breaking interface change."_ `StorageBucketAdapter.verify()` already implements it for real (Phase 3) — re-listing the user's Storage folder independently of what `delete()` reported, rather than trusting `delete()`'s own success flag.
- **The workflow state machine (`workflow.ts`)** already reserves three verification states in `STATE_ORDER` — `verify_database`, `verify_storage`, `verify_external` — positioned after all deletion phases and before `completed`. `stepVerifyStorage` already calls the real `StorageBucketAdapter.verify()`. `stepVerifyDatabase`/`stepVerifyExternal` are Registry-driven pass-throughs today, each producing a `"skipped"` checkpoint entry with `detail: "no verificationAdapter configured in the Registry"` per resource — i.e., they already do exactly what an unpopulated Registry should cause them to do, and are structurally ready to do something else the moment `verificationAdapter` stops being `null`.
- **`deletion_requests`/`checkpoint`** is the durable, RLS-protected, already-live-validated evidence substrate. Critically: `deletion_requests` is **not** one of the 32 tables in `DELETION_RESOURCE_REGISTRY`'s `DATABASE_RESOURCES` — it is not deleted by the workflow it records. Combined with the TDD's confirmation that `auth.users` (the FK target `deletion_requests.user_id` cascades from) is never deleted by this endpoint either, a `deletion_requests` row — and everything in its `checkpoint` array — survives the very deletion it documents. This is the load-bearing property that makes extending `checkpoint` a viable evidence store, and it was verified directly in code this session, not assumed.
- **The transactional SQL pattern** (`delete_user_owned_data_ordered`) is a proven template: Registry-parameterized (`p_tables jsonb`), identifier-validated, per-column-type-aware, `SECURITY DEFINER`/`service_role`-only. Phase 2.1's live validation already found and fixed the two defects (a `text`/`uuid` mismatch, an ambiguous column reference) that this exact pattern is prone to — a verification-side function following the same shape inherits that lesson for free.

## 6. Dependency Analysis

**Existing Dependencies** (Phase 4 consumes directly, unmodified):

- Registry (`registry.ts`) — table/column/resourceId inventory, the `verificationAdapter` extension point
- Workflow Orchestrator (`workflow.ts`) — `STATE_ORDER`, `runStep`, `checkpointEntry`, `persist` (with its already-hardened optimistic concurrency control)
- `DeletionAdapter` interface (`adapter.ts`) — the `verify()` stage contract, already real for Storage
- `deletion_requests` table + `checkpoint jsonb` — durable, RLS-protected, proven-to-survive-its-own-subject evidence substrate
- The transactional-SQL pattern (`delete_user_owned_data_ordered`) — template for a verification-side counterpart

**Required Extensions** (additions to existing components, not new components):

- A read-only, Registry-parameterized SQL function verifying zero remaining rows per table for a user (mirrors `delete_user_owned_data_ordered`'s shape and safety pattern; additive migration, no change to the existing delete function)
- Non-null `verificationAdapter` values on Registry entries where verification is meaningful today (32 database resources at minimum; Storage's is arguably already satisfied by `DeletionAdapter.verify()` and may not need a separate Registry-level value)
- Real logic in `stepVerifyDatabase`/`stepVerifyExternal` (modifying existing functions in `workflow.ts`, not adding a parallel module)
- A verification-adapter-resolution helper analogous to `getStorageDeletionAdapters()` in `registry-adapters.ts`, if database verification needs its own resolution step distinct from the Registry's raw entries (a Phase 4B/TDD-level call)
- Possibly an extended `CheckpointResourceStatus`/`CheckpointEntry` vocabulary to satisfy "distinguish deletion success from verification success" cleanly — additive to a `jsonb` column, no migration required even if extended

**Future Phase Dependencies** (explicitly not Phase 4, must not be built now):

- OAuth (`oauth.client_integrations`) and background-job (`background.document_process`) _deletion_ adapters — both still `deletionAdapter: null`. Verifying resources that nothing deletes yet is not meaningful; these remain blocked on deletion-adapter work not currently assigned to any specific named phase in the roadmap (see Open Questions).
- Grace-period scheduling (Phase 5) — explicitly named as a Phase 4 dependency-provider, not a Phase 4 consumer.
- Reconciliation / drift remediation automation (Phase 6) — Phase 4 detects and evidences discrepancies; Phase 6 is where the roadmap places acting on them.
- Operator/monitoring dashboards, production observability (Phase 7).
- Resolving the production schema-drift question for the four tables (`document_chunks`, `personas`, `push_subscriptions`, `proactive_daily`) — an environment/provisioning task, orthogonal to Phase 4's architecture, but a hard blocker for any live end-to-end validation of Phase 4's own eventual implementation, exactly as it already blocks Phase 3.1's.

## 9. Architecture Alternatives Evaluation

**Alternative A — Registry-and-Workflow-integrated verification (extend the existing checkpoint/state-machine substrate).**
Populate `verificationAdapter` on Registry entries; add one new read-only, Registry-parameterized SQL function for database resources (mirroring `delete_user_owned_data_ordered`'s safety pattern); implement real logic in the already-reserved `stepVerifyDatabase`/`stepVerifyExternal` workflow steps; write real verification evidence into the existing `checkpoint jsonb` array using the existing RLS-protected `deletion_requests` row.

- Complexity: Low — every extension point already exists; no new persistence, no new state-machine states, no new RLS surface.
- Maintainability: High — one inventory (Registry), one evidence store (`checkpoint`), one execution model (the existing workflow), consistent with every prior phase's stated design principle.
- Extensibility: Good — the same `verificationAdapter` field extends cleanly to OAuth/background-job resources once those get real deletion adapters.
- Operational cost: Low — no new infrastructure, no new monitoring surface, additive migration only.
- Compatibility: High — additive to the response shape and the checkpoint vocabulary; no existing consumer breaks.
- Production risk: Low-Medium — the one new SQL function inherits the same class of risk Phase 2.1 found real bugs in (type/casting edge cases), which argues for live validation before production reliance, not against the pattern itself (see Production Impact Assessment, §11).

**Alternative B — New, separate `verification_results` table and independent verification module, decoupled from `deletion_requests`/`checkpoint`.**
A dedicated table (`verification_results` or similar) with its own schema, written by a standalone verification module that can be invoked separately from `runDeletionWorkflow`.

- Complexity: Medium-High — a new table, new RLS policies, a new lifecycle question for how/when it's populated relative to the workflow.
- Maintainability: Lower — two persistence models now describe overlapping concerns (deletion progress vs. verification results) that the existing `checkpoint` design already anticipated holding together (the `verify_*` states were reserved in the _same_ enum as the deletion states from Phase 1 onward).
- Extensibility: Comparable to A for future resource types, but at the cost of a second schema to keep in sync with the Registry.
- Operational cost: Higher — a new migration, new backfill/lifecycle question for existing `deletion_requests` rows that predate verification.
- Compatibility: Neutral.
- Production risk: Higher — more new surface area than Alternative A for the same roadmap-defined scope.
- **Rejected.** Directly conflicts with the roadmap's own constraint ("Avoid parallel registries, duplicate workflow states, redundant adapters, or persistence models that duplicate responsibilities") and with the roadmap's declared dependency list (Phase 4 "Depends on: ... deletion_requests" — implying verification is meant to build on it, not beside it).

**Alternative C — Fully decoupled, out-of-band verification (independent of `runDeletionWorkflow`, triggered separately per user or on demand).**
A standalone verification pass that queries the Registry and the live database/Storage independently of the workflow's own state machine, on the reasoning that verification should not share fate with the code path that performed the deletion.

- Complexity: Medium — no new persistence necessarily required (could still write to `checkpoint`), but needs its own invocation mechanism outside the workflow's resumable-by-reinvocation model.
- Maintainability: Lower — two independent execution paths (the workflow, and this) both need to agree on Registry semantics and both need their own testing/hardening discipline, duplicating review surface for no scope gain within Phase 4 (scheduling triggers are explicitly Phase 5, not 4).
- Extensibility: Comparable.
- Operational cost: Higher within Phase 4's own bounds — "on demand" still needs _something_ to demand it, and the roadmap excludes scheduling/dashboards from Phase 4.
- Compatibility: Neutral.
- Production risk: Comparable to A for the verification logic itself, plus added risk from a second, less-exercised invocation path.
- **Rejected as a full alternative, but its core principle is preserved inside Alternative A**: verification must re-check real state independently of trusting the deletion step's own report — exactly what `StorageBucketAdapter.verify()` already does (re-lists rather than trusting `delete()`'s return value) and exactly what the new database-verification SQL function should do (an independent `SELECT`, not a re-read of the delete function's own row-count output). "Independent" here means _independent of the deletion step's self-report_, not _independent of the workflow's execution model_ — a narrower, already-proven pattern.

**Selected: Alternative A**, with Alternative C's re-verify-don't-re-trust principle carried in as a binding design constraint for Phase 4B.

## 10. Architecture Validation

- **Satisfies Phase 4 objectives and success criteria:** Yes — every roadmap-listed deliverable maps to an existing extension point or a small, justified addition (§5, §7); the success criteria (verify every resource, produce auditable evidence, distinguish deletion from verification success, generate final status from verification results) are all achievable within Alternative A without new infrastructure beyond one additive SQL function.
- **Does not duplicate existing capabilities:** Confirmed — Alternative A was selected specifically because it reuses the Registry, the adapter `verify()` contract, the reserved workflow states, and the `checkpoint` evidence store rather than building parallel versions of any of them (§7, §9).
- **Does not conflict with previous ADRs:** Confirmed against ADR-0001 (behavior flags — unrelated, no interaction), ADR-0002 (companion state — unrelated; note `db.companion_state` is itself one of the 32 Registry-listed deletion targets, verification of it follows the same generic database-resource path as any other table, no special case needed), ADR-0003 (Brain Gateway — unrelated), and ADR-0004 (directly extended, not contradicted — every "untouched" constraint ADR-0004 and subsequent phases have honored for the Registry/RPC/adapter interface remains honored by this architecture).
- **Preserves separation of concerns:** Confirmed — Registry stays inventory-only, the new SQL function stays narrowly read-only-verification-only (mirroring, not merging with, the delete function), the workflow stays orchestration-only.
- **Minimizes complexity:** Confirmed by the alternatives comparison (§9) — Alternative A has the smallest new-surface-area footprint of the three evaluated.
- **Follows extension-over-replacement:** Confirmed — nothing existing is replaced; every reused component listed in §7 keeps its current responsibility unchanged.

## 11. Production Impact Assessment

This is a projection of the _chosen architecture's_ eventual impact once built (Phase 4B/implementation), not a current-state assessment — no code changes were made in this discovery stage.

- **Runtime behavior:** `verify_database`/`verify_storage`/`verify_external` currently execute near-instantly (pass-throughs). Populating real verification adds real DB/Storage round-trips to every deletion request's total execution time. Recommendation carried to Phase 4B: batch database verification into one additional RPC call (mirroring the delete function's own batching) rather than one query per table, to avoid N+1 latency growth across 28-32 tables.
- **Persistence model:** Additive only — one new SQL function (new migration), a populated Registry field (no schema change, TypeScript-only), and possibly an extended `CheckpointResourceStatus` vocabulary (no migration required; `checkpoint` is unconstrained `jsonb`).
- **API contract:** `POST /api/emma/gdpr`'s response can gain new, additive fields (e.g., a structured verification rollup alongside the existing flat `summary` array). No existing field needs to change shape or meaning.
- **Backward compatibility:** No existing behavior changes. `settings/privacy/page.tsx` already reads `status`/`success` (hardened during Phase 3, per the SDD ledger's "2 Important findings... both fixed in c4292ea" note) rather than just `res.ok`, so it already tolerates additional response detail without changes.
- **Deployment:** One new, additive Supabase migration, following the exact same defensive pattern (`SECURITY DEFINER`, `SET search_path = ''`, `service_role`-only grant, identifier validation) already proven for the delete function — same deployment risk class as Phase 2's migration, which Phase 2.1 already established a live-validation methodology for.
- **Operational procedures:** None anticipated to change for Phase 4 itself — no scheduler, no dashboard, no new operator-facing surface (both explicitly out of Phase 4's roadmap scope; dashboards begin at Phase 7).

## 15. Recommendation

**Proceed to Phase 4B (Technical Design), subject to ADR-0005's approval per the roadmap's Engineering Workflow step 6 ("ADR Approval").**

Technical justification: every roadmap-listed Phase 4 deliverable traces to either an already-existing, already-hardened extension point (Registry's `verificationAdapter` field, the `DeletionAdapter.verify()` contract, the workflow's reserved `verify_*` states, the `deletion_requests`/`checkpoint` evidence substrate) or one narrowly-scoped, justified new component (a single read-only SQL function for database-resource verification). No component in this discovery required a "Create New" decision without direct technical justification (§7); no alternative that would have introduced parallel persistence or duplicated existing responsibilities was selected (§9). The chosen architecture was validated against every prior ADR without conflict (§10) and its projected production impact is additive-only across every dimension assessed (§11). Two genuine open questions remain (§13) — the evidentiary standard for "auditable evidence," and which phase (if any) will ever implement OAuth/background-job deletion adapters — neither blocks Phase 4B from beginning, since both are scoped as questions for product/legal and for the roadmap owner respectively, not as gaps in this discovery's own analysis.
