**Severity: MAJOR.**

**Trace (concrete, using the Registry's actual 2 Storage resources â€” `storage.document-ingestion`, `storage.task-documents`):**

- **Call 1** reaches `verify_storage` (because `verify_database` already succeeded). Per-adapter loop, first time for both resources â€” neither guard fires (no prior entries exist yet):

**Why this is not a manufactured edge case:** two Storage buckets is the entire, fixed cardinality of this resource type in the Registry (confirmed Â§1 above) â€” a "one bucket already clean, the other still dirty" split on the very first verification attempt is not a rare timing coincidence, it is one of only three possible outcomes of a 2-resource check (both clean, both dirty, split) and the _only_ one of those three that triggers this defect, but it is entirely ordinary, not a contrived multi-in-a-row race.

1. **Fix the dedup rule to be status-aware, not purely timestamp-based:** when the latest entry for a `(phase, resourceId)` key is `"skipped"` and an earlier entry for the same key is not, keep the earlier, substantive entry instead â€” a `"skipped"` resume-guard placeholder should never be allowed to suppress real evidence for the same resource within the same rollup.

Either is a small, local, one-paragraph fix; neither requires new architecture, a new checkpoint status, or a new field.

## 5. Task-Specified Stress Tests â€” Explicit Answers

- **(a) Can two entries for the same `(phase, resourceId)` collide on `recordedAt` at millisecond resolution?** Not within a single execution â€” each execution of `stepVerifyDatabase`/`stepVerifyStorage` produces at most one entry per distinct `resourceId` (32 distinct tables, 2 distinct buckets), so no two entries _from the same call_ ever share a dedup key. Across retries (separate `runDeletionWorkflow()` invocations, each triggered by a separate client HTTP call), a same-millisecond collision is theoretically possible but not practically reachable â€” retries are gated by `MAX_RETRY_COUNT` and require a new request to arrive, not a tight synchronous loop. **Not a real risk; correctly not addressed by the design, and does not need to be.**
- **(b) Could the marker and a real entry share the latest timestamp and get conflated?** No â€” step 1 excludes every marker unconditionally, by `resourceId`, before step 2's timestamp comparison ever runs; a marker is never a _candidate_ in the per-resource dedup regardless of its `recordedAt`. **Confirmed clean, independently re-derived in Â§3.**
- **(c) Is the marker's own placement in the dedup â€” before or after â€” actually stated, and does the order matter?** Â§7.1 states the order explicitly now ("among the entries surviving step 1"): exclude first, dedup second. Order does not change the _result_ here (a marker's `resourceId` never collides with a real one either way), but the TDD stating it explicitly, rather than leaving it implicit, is itself the correct fix for exactly the ambiguity round 1's original Finding #3 was about. **Confirmed adequately specified.**
- **(d) Is `recordedAt` sufficiently monotonic for "latest" to be meaningful, given `new Date().toISOString()` and possible clock adjustments (e.g., NTP step-back)?** In the general case, no absolute guarantee exists â€” a backward clock step between two calls could in principle make an objectively later verification attempt sort as "earlier." This is a real, if extremely low-probability, gap the TDD does not acknowledge. It is not blocking on its own (the window is production NTP-drift-sized, not attacker-controlled, and no other clock-dependent correctness property in this codebase is held to a stricter bar), but it is worth a documented **Observation** rather than silence, especially given the document's own habit (Â§3.5, Â§5.4) of stating "known, accepted, narrow risk" explicitly rather than leaving it for a future reader to wonder whether it was considered.
