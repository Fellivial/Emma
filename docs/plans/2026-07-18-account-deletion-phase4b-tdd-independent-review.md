## 2. ADR Compliance Review

Each of ADR-0005's seven Chosen Architecture items, checked against the TDD's translation:

1. **Populate `verificationAdapter` (label, no new fields).** §1 populates `"database-row-count-verify"` on all 32 database entries, leaves Storage/OAuth/background-job/excluded entries `null` with individually stated rationale, and confirms no new field is added to `ResourceEntryBase`/`DatabaseResourceEntry`/`OtherResourceEntry`. **Correctly translated.**
2. **New read-only, Registry-parameterized SQL function mirroring the delete function's safety pattern.** §3 specifies name, parameters, return shape, batching, identifier-validation/casting discipline (reused), and a deliberate, disclosed divergence in error-propagation (per-table catch instead of whole-call abort). The divergence is justified against the read-only nature of the function and is consistent with — not a reinterpretation of — the ADR's "mirrors ... safety pattern" language, which the ADR itself scoped to safety, not error propagation. **Correctly translated, and the one genuine fork is resolved with stated rationale, as the ADR anticipated.**
3. **`stepVerifyDatabase`/`stepVerifyExternal` get real bodies; no new `STATE_ORDER` states.** §4 gives both real bodies (§4.1, §4.3) and §5.2 confirms `STATE_ORDER` is untouched. **Correctly translated**, but see §6 and §7 below for a defect inside how §4.1's real body interacts with §5's outcome-authority mechanism.
4. **Evidence written into `checkpoint jsonb`.** §4, §6 write real per-resource entries replacing the `"skipped"` placeholders. **Correctly translated at the level of "where evidence lives"** — but see §7 (Checkpoint Review) for a defect in what one specific evidence entry's status value is allowed to mean.
5. **Workflow outcome authority — verification failure must be capable of changing final status.** §5 widens `CRITICAL_STEPS` to include all three verify phases, reusing the existing `"failed"`-only branch unmodified. This is the correct **shape** of mechanism the ADR requires. However, direct tracing of this mechanism against §4's own checkpoint design (§7, §9 below) shows the mechanism **does not reliably hold on retry** for the specific case ADR-0005 exists to fix (a confirmed database-verification defect). This is the review's central finding — see §7.
6. **Runtime duration — one batched call; evaluate `vercel.json` headroom.** §3.4 specifies one RPC call for all verifiable resources; §8.2 evaluates and recommends `maxDuration: 60`. **Correctly translated**, with a defensible (if unbenchmarked) rationale.
7. **Resume safety — verify steps use the `isPhaseCompleted()` skip-guard pattern.** §4.1, §4.2, §4.3 all add guards. §4.2 (Storage, per-adapter guard) is correctly modeled on the existing, proven pattern and is sound. §4.1 (database, aggregate marker) introduces a **new** pattern not previously proven in this codebase, and that new pattern is where the defect in item 5 originates (§7). §4.3 (external) specifies a guard whose corresponding marker-write step is never specified, making the guard inert as written (§7.4, §14).

**Conclusion:** items 1, 2, 6 are cleanly and correctly translated. Items 3, 4, 5, 7 are correctly translated in shape but contain one interacting defect (the aggregate marker's status semantics) that undermines item 5's binding requirement specifically for the scenario ADR-0005 was written to close. No item is reinterpreted against its own text, and no item is silently dropped — this is a completeness/correctness gap within a faithfully-attempted translation, not a compliance violation of the ADR's letter.

## 6. Verification Design Review

- **Database verification (§2, §3):** the read-only/write separation from the mutating delete function is well-reasoned and consistent with ADR-0005's Design Goal 3 (no new adapter contract). The per-table failure-isolation design (§3.5) is a genuine, well-justified improvement over the delete function's whole-transaction-abort model, correctly justified by the different risk profile of a read vs. a write.
- **Storage verification (§2.2, §4.2):** correctly identified as requiring no new logic — `StorageBucketAdapter.verify()` is already real. The only change (a resume-skip guard) is modeled correctly on the existing, proven per-adapter pattern used by `stepDeletingStorage`. **However**, §4.2 does not reconcile that widening `CRITICAL_STEPS` to include `verify_storage` (§5.1) means a Storage **outage** during verification (`list()` errors — §9 scenario 6) is now indistinguishable, for outcome purposes, from **confirmed leftover Storage objects** (§9 scenario 5) — both produce a `"failed"` checkpoint entry and both now block completion identically. The TDD explicitly built a three-way distinction for the analogous database case specifically to avoid this exact conflation (`"inconclusive"` vs `"failed"`, §3.5, §5.4) but declined to apply the same reasoning to Storage, without explaining why the two resource types warrant different treatment here. - **External verification (§2.2, §4.3):** correctly identified as unchanged logic (no adapter exists). The added resume guard is specified but its precondition (a marker entry must exist to guard against) is never established — see §7.4.
- **Evidence generation, batching, security/permission assumptions (§3.6):** all correctly reused from the delete function's proven pattern, with an accurate, specific description of what is and is not reused.

## 7. Checkpoint Review — Central Finding

This is the review's most severe finding, found by tracing the interaction between §4.1 (the aggregate marker), §4.4 (why the marker exists), and §5.1-§5.3 (the retry mechanism) against the actual code (`isPhaseCompleted()`, `resumeStartStatus()`, the `CRITICAL_STEPS` branch), not by reading any one section in isolation.

**The defect:** §4.1 step 3 specifies that after `verifyUserOwnedDataDeleted()` returns successfully (does not throw), `stepVerifyDatabase` appends **one synthetic marker entry, unconditionally**:

> `resourceId: "db.verification-batch"`, `phase: "verify_database"`, `resourceStatus: "completed"`, `detail: "batch verification executed"`

This status value is **not conditioned on whether any of the per-resource entries emitted in step 2 were themselves `"failed"`**. It is written as `"completed"` even when a table's `remaining_count > 0` was found — i.e., even when real, confirmed leftover data was just detected.

Trace the consequence through the actual retry path §5.3 describes:

1. **First call.** `verifyUserOwnedDataDeleted()` succeeds; one table has `remaining_count: 3`. `stepVerifyDatabase` returns 31 `"completed"` entries, 1 `"failed"` entry, and the marker entry (`"completed"`). The main loop (`workflow.ts:471-484`) computes `failed = entries.filter(e => e.resourceStatus === "failed")` → length 1. Since `verify_database` is now in `CRITICAL_STEPS` (§5.1), the workflow persists `checkpoint: nextCheckpoint` (which **includes** the marker), sets `status: "retry_pending"`, and returns. So far, correct — the caller is told `success: false`.
2. **Second call (the retry §5.3 describes).** `findActiveDeletionRequest` finds the row (`retry_pending` is non-terminal). `resumeStartStatus()` (`workflow.ts:389-395`) returns `row.checkpoint[row.checkpoint.length - 1].phase`. Because the marker was appended **last** in step 1 (deliberately, per §4.4's own stated reasoning), the last entry's phase is `"verify_database"` — so the cursor correctly resumes at `verify_database`, not `deleting_database`. So far this matches §5.3's claim.
3. `runStep` calls `stepVerifyDatabase`'s skip guard: `isPhaseCompleted(row, "verify_database", "db.verification-batch")`. `isPhaseCompleted()` (`workflow.ts:172-180`) tests `e.phase === phase && e.resourceId === resourceId && e.resourceStatus !== "failed"`. The marker entry from step 1 has `resourceId: "db.verification-batch"` and `resourceStatus: "completed"` — **it satisfies this test**, because `"completed" !== "failed"` is `true`. **The skip guard fires.**
4. `stepVerifyDatabase` therefore returns **one `"skipped"` entry** and does **not** call `verifyUserOwnedDataDeleted()` again. `verify_database`'s real logic never re-runs.
5. Back in the main loop, `entries = [one "skipped" entry]`. `failed = entries.filter(...)` → **empty**. `CRITICAL_STEPS.includes("verify_database")` is true, but `failed.length > 0` is false, so the retry-escalation branch is **not** entered. The workflow proceeds past `verify_database` to `verify_storage`, `verify_external`, and — assuming those are clean — to `"completed"`.
6. The response to this second call: `status: "completed"`, `success: true`.

**Consequence:** a deletion request that verification confirmed still has real leftover data in the database, on its very next re-invocation (the exact retry mechanism §5.3 claims exists specifically so "a verification-triggered retry ... will reproduce the same `"failed"` result deterministically"), silently converges to `success: true` without verification ever running a second time. This is precisely the failure mode the roadmap's Phase 4 objective and ADR-0005's Design Goal 4 exist to prevent — a user is told their data was deleted and verified, when in fact the one thing that was confirmed is that it was **not**.

This is not a hypothetical edge case requiring an unlikely input: it is the direct, mechanical consequence of §4.1 step 3's literal text (an unconditional `"completed"` status on the marker) combined with §5.1-§5.3's own stated retry design, on the exact scenario (§9 scenario 2) the TDD itself walks through as the "genuine defect" case. The TDD's own Failure Model table (§9) documents scenario 2's _first-call_ outcome correctly but never traces a _second_ call for that same scenario — the retry it explicitly claims will happen (§5.3) is never itself modeled as a failure-model row, and that is exactly the gap that hid this defect.

**Why this differs from the pattern it claims to mirror:** the existing, proven `stepDeletingDatabase` pattern (`workflow.ts:219-238`) that §4.1/§4.4 explicitly models this on has **no** split between "aggregate marker" and "per-resource results" — there is exactly one entry (`db.batch`), and its status is directly the batch's real outcome (`"completed"` only if `deleteUserOwnedData()` did not throw, `"failed"` otherwise). Because deletion is transactional (any failure aborts the whole call), a single entry can honestly carry both meanings ("the batch ran" and "the batch succeeded") at once. Verification's own design deliberately abandoned that transactional all-or-nothing property (§3.5, correctly, for good reasons specific to a read) — but in doing so, it created a batch that can execute successfully (no thrown error) while still finding a real defect, and never adjusted the marker's status semantics to account for that new possibility. The marker answers "did the batch execute" when the skip guard needs it to answer "did the batch execute **and find everything clean**" — the TDD conflates these two meanings under one status value in exactly the same category of error §5.4 goes out of its way to explain must not happen for `"inconclusive"` vs. `"failed"` ("could not determine" vs. "confirmed a problem") — except here it is "confirmed a problem" being silently absorbed into the meaning of "ran to completion."

**Severity: Critical.** This is a logic defect, not a documentation gap — it would ship a working-looking (unit-testable, code-reviewable, passes a single-invocation test) verification feature whose entire retry story is inert for the one scenario that matters most, and it directly reintroduces — one layer downstream, inside the TDD's own new mechanism — the exact class of "verification result doesn't actually govern final status" bug ADR-0005's Independent Review already found and got fixed at the ADR level (see that review's §5). No test proposed in §14.2 would catch it, because §14.2 proposes testing "resume with marker present" and "resume with marker absent" as if they were the only two states, without proposing the one test that would surface this — "resume with marker present **and** a prior `"failed"` per-resource entry in the same batch."

## 8. API Review

- **Response model (§7.1):** additive, correctly reasoned not to break `privacy/page.tsx`'s existing branch structure (confirmed directly against the file — no code changes required, matching the TDD's claim).
- **Defect found — verification rollup counting is not reconciled against the marker design (§4.1/§4.4):** §7.1 states the `verification` rollup is computed "by grouping the ... final `checkpoint` array by phase ... and counting `resourceStatus` occurrences within each group." The synthetic aggregate marker entries (`db.verification-batch`, and — if written per §4.3, which is itself unclear, see §7.4 below — `external.verification-batch`) share the same `phase` value (`verify_database` / `verify_external`) as the 32/N real per-resource entries. Nothing in §7.1 excludes these synthetic entries from the count. A literal implementation of §7.1's description would therefore report `verification.database.verified: 33` (32 real tables plus the marker) on a clean run, or an inflated `skipped` count on a resumed/skip-guarded run (the marker's own `"skipped"` re-emission, §4.1 step 1, would itself be counted as if it were a 33rd resource). This contradicts §2.3's own promise ("every table gets a result... one entry per verifiable resource") and is never reconciled against §7.1's counting logic — the two sections were evidently designed independently.
- **Naming inconsistency:** §7.1's rollup buckets are named `{verified, failed, inconclusive, skipped}`, but the checkpoint-level vocabulary defined in §6.1 is `{completed, failed, skipped, inconclusive}`. The document never states that API-level `verified` corresponds to checkpoint-level `resourceStatus === "completed"` — a plausible but unstated mapping, and the only place in this fairly rigorous document where a naming translation between two layers is left implicit rather than defined.
- **Backward compatibility (§7.3):** claim confirmed correct against the actual file — `privacy/page.tsx` does not read `summary` and its three-branch structure on `success`/`status` requires no change.
- **Error propagation (§7.5):** the asymmetry described (verification failures absorbed one layer below the route handler; deletion failures may still throw to the route) is consistent with the rest of the design and correctly reasoned.

## 10. Failure Model Review

The 14-scenario table (§9) is unusually thorough and each individually-described scenario was checked against the actual code paths it claims to exercise. Two gaps found beyond what's already covered in §7:

- **Missing scenario:** as detailed in §7 above, no row models "a real per-resource database failure occurs (scenario 2), the workflow returns `retry_pending`, and the user (or an automated retry) re-invokes delete" — the single most important scenario for validating that ADR-0005 item 5's binding requirement actually holds across a retry, and the exact scenario where the design breaks.

## 11. Compatibility Review

- **API compatibility:** confirmed additive and non-breaking, as described.
- **Migration compatibility:** confirmed additive; no change to the existing delete function or table shapes.
- **Registry compatibility:** confirmed — one field-value change across 32 entries, two new pure derivation functions, no existing consumer's behavior changes.
- **Checkpoint compatibility:** the additive-field/additive-union-value reasoning (§10 row "Checkpoint compatibility") is correct for _old rows being read by new code_. It does not address the _new_ compatibility hazard this review surfaces in §7 (new rows, produced by the new code itself, becoming permanently under-verified across retries) — that hazard is internal to the new design, not a legacy-data compatibility question, so it falls outside what §10's table was scoped to check, but it is a compatibility-shaped defect nonetheless (the row's evidence "looks complete" to any future reader while being silently stale).
- **Workflow compatibility — regression against the existing test suite, not disclosed:** §10's row states `CRITICAL_STEPS` widening is "an additive change to a `const` array, not a structural change to the state machine's shape," implying low risk. Direct trace against two currently-passing tests shows this understates the actual behavioral impact:
  - `tests/unit/deletion-workflow.test.ts`, test _"creates a new row and reaches completed for a fresh user, tolerating unconfigured storage as best-effort"_ (lines 199-222): asserts `result.status === "completed"` for a fresh user when Storage is unconfigured (the test's `rpc` mock never configures `NEXT_PUBLIC_SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY`, so `getStorageClient()` returns `null` and `deleting_storage` entries come back `"failed"` — tolerated today because `deleting_storage` is not in `CRITICAL_STEPS`).
  - `tests/unit/gdpr-workflow-integration.test.ts`, test _"delegates to the deletion workflow and returns its status instead of deleting inline"_ (lines 44-117): asserts `body.status === "completed"` and `body.success === true` via a mocked `createClient` that returns an object with only `rpc`/`from` (no `.storage` key) — meaning any call into `StorageBucketAdapter`'s `getStorageClient()` (which reuses the same globally-mocked `createClient`) would hit `supabase.storage.from(...)`, a `TypeError` on `undefined`, caught by the existing try/catch as a `"failed"` result.

  Once `verify_storage` is added to `CRITICAL_STEPS` (§5.1) — which is necessary in general, since the whole point of Phase 4 is to make Storage verification failures load-bearing too — **both of these currently-passing tests would flip their expected outcome** from `"completed"`/`success: true` to `"retry_pending"`/`success: false`, purely because of an _environmental_ Storage misconfiguration in the test harness, not because of any confirmed leftover data. Neither test is listed in §14.2's testing requirements as needing to change, and the TDD's Compatibility section does not disclose that this is a real, concrete, currently-reproducible regression against the existing suite — it is presented as a purely additive, low-risk change.

- **Deployment/rollback compatibility (§10, §14.4, §14.5):** correctly reasoned and consistent with the established Phase 2/2.1 precedent.

**One traceability weakness:** item 5's row states the TDD's implementation is "the `CRITICAL_STEPS` widening mechanism, its scoping to `"failed"` only, and the resulting `route.ts` semantics" — this is traced to §5 in full, which is accurate as a _description_ of what §5 contains, but the traceability table itself makes no claim (and cannot, since the table is a mapping exercise, not a verification pass) about whether §5's mechanism actually _works_ end-to-end once composed with §4's checkpoint design. The same structural risk recurred here, one document later, and was not caught by this document's own self-review.

## 13. Internal Consistency Review

- **The central finding (§7):** an internal contradiction between §4.1's unconditional marker status and §5.3's claim that retries "reproduce the same failed result deterministically." These two sections cannot both be true as written.
- **§7.1 vs. §4.1/§4.4 (API rollup counting vs. marker entries):** the marker entries are never excluded from the API's phase-grouped count, contradicting §2.3's "one entry per verifiable resource" premise (§8 above).
- **§7.1's `verified` vs. §6.1's `completed`:** unexplained terminology drift between the checkpoint-level and API-level vocabularies (§8 above).
- **§4.3's guard vs. its precondition:** §4.3 specifies a skip guard (`isPhaseCompleted(row, "verify_external", "external.verification-batch")`) but never specifies that a corresponding `"external.verification-batch"` marker entry is written after the step runs — unlike §4.1's explicit step 3 for the database case. As specified, the guard's condition can never become true (nothing produces a matching entry), making it dead code — which contradicts §4.3's own stated purpose ("keeps all three verify steps structurally uniform"): the three steps are not, in fact, structurally uniform as specified, since only one of the three (database) has both a guard and a marker-write; storage uses a different, correct pattern (per-adapter real status); external has a guard with no corresponding write.
- **Failure Model §9 row 6's dangling cross-reference** ("see rationale below" — no rationale follows), covered in §10 above.

## 14. Implementation Readiness Assessment

The document's own Finalization section claims: "every new component ... is specified at a level ... that requires no further architectural judgment calls to implement." Checked against the findings above, this claim does not hold in full:

- An implementer following §4.1 literally would ship the retry-defeating defect in §7 — this **is** an architectural judgment call the TDD should have made (what should the marker's status be conditioned on?) and did not.
- An implementer would need to independently notice and decide how to change `stepVerifyDatabase`/`stepVerifyExternal`'s call signatures (currently zero-argument) — a small but real gap the TDD does not flag (§4 above).
- An implementer would need to independently decide whether `stepVerifyExternal` needs a marker-write step for its guard to do anything (§13 above) — the TDD is silent on this, and either answer requires judgment the document doesn't supply.
- An implementer would need to independently decide how to exclude synthetic marker entries from the API's verification rollup counts (§8 above) — again, a judgment call, not a mechanical transcription.
- An implementer would need to independently decide whether Storage's `list()`-outage case should really be equivalent to a confirmed-leftover-objects case for blocking-completion purposes (§6, §11 above) — the TDD asserts this is "left unchanged" without acknowledging it now has new consequences under the widened `CRITICAL_STEPS`.

**Conclusion: the document is not yet implementation-ready.** The gaps above are not documentation nits; at least one (§7) is a logic defect that would ship a broken retry path for the central compliance scenario this entire phase exists to close, and would very likely pass code review and a naive test pass, since it only manifests on a specific two-invocation sequence not exercised by any test the TDD itself proposes.

## 21. Final Verdict

**MAJOR REVISIONS REQUIRED.**

The Phase 4B TDD correctly translates the majority of ADR-0005's Chosen Architecture — the Registry population, the new SQL function's design, the reuse of Storage's existing verification, the general shape of the workflow-outcome-authority mechanism, and the runtime/batching requirements are all soundly reasoned and, where checked against the actual repository, accurate. Scope compliance is clean: no Phase 5/6/7 scope, no implementation code, no re-litigation of ADR-0005's fixed inputs.

However, direct, end-to-end tracing of the interaction between the new aggregate skip-guard marker (§4.1, §4.4) and the retry/outcome-authority mechanism it is meant to support (§5.1-§5.3) surfaces a **critical logic defect**: the marker's status is written unconditionally as `"completed"` regardless of whether real, confirmed leftover data was found, which means the retry mechanism this document itself describes as the way a confirmed database-verification failure gets resolved (§5.3) never actually re-runs verification. A deletion request that correctly reported `success: false` on its first call will — on the very next call, with zero further server-side changes and no fix to the underlying data — report `success: true`. This is the same category of defect ADR-0005's own Independent Review already found and fixed once, at the ADR level (that review's central finding, its §5); an equivalent defect has reappeared one document later, inside this TDD's own new mechanism, and was not caught by this document's extensive self-review, its traceability tables, or its 14-scenario failure model.

Three further findings (the `CRITICAL_STEPS`/Storage-outage conflation and its concrete regression against two currently-passing tests; the API rollup's failure to exclude synthetic marker entries from its counts; the unspecified marker-write precondition for `stepVerifyExternal`'s guard) compound the case that this document, while substantially complete and well-reasoned in its broad strokes, is not yet safe to hand to Phase 4C as an implementation-ready specification.

**Phase 4C (Implementation) must remain blocked until all blocking technical design issues have been resolved.**
