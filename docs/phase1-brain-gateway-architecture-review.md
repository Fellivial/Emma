## 1. Architecture Review Report

### 1.1 Executive Summary

The Brain Gateway (`src/core/brain/`) is a small, cleanly bounded module: 3 files, 551 lines (`gateway.ts` 71, `types.ts` 169, `providers/openrouter.ts` 311), sitting between 14 non-test caller files (16 invocation sites, per Phase 0 §4/§7 as independently corrected) and a single provider implementation. Judged purely on the boundary it was built to enforce — "application code never constructs a provider request, holds a provider URL or API key, or parses a provider-shaped response" (`gateway.ts` header comment, mirroring ADR-0003's Decision section: "Application code SHALL NOT construct a provider request, hold a provider URL or API key, or parse a provider-shaped response directly") — it succeeds without qualification: Phase 0's grep sweep found zero provider-wire leaks outside `src/core/brain/providers/openrouter.ts`, and this review's direct reading of `gateway.ts`, `types.ts`, and `openrouter.ts` confirms the boundary is structural, not conventional. The gateway also already carries real weight beyond one-shot chat: streaming (async-generator events, normalized trailing `done`), tool-calling (`agent-loop.ts`'s two call sites, `maxRetries: 2` main loop / `maxRetries: 0, timeoutMs: 8_000` tool-evaluation), vision, and embeddings all flow through the same three functions.

Architectural quality diverges sharply depending on which layer is examined. **At the inference boundary itself** (Gateway ↔ Provider), the architecture is mature: single entry point, normalized contracts, provider-specific vocabulary (OpenRouter's `models: [...]` fallback array, the 529 status code, SSE chunk framing) contained entirely inside one file, and an explicitly documented extension path (implement `BrainProvider`, extend `gateway.ts`'s provider selection, zero caller changes) that this review finds procedurally credible — every mechanism it depends on (interface, selection point, normalized contract) already exists and is already proven against a real, non-trivial caller (the agent loop's tool-calling path). **Above the boundary**, in the Application Layer ADR-0003 explicitly declines to redesign, architectural quality is materially weaker: prompt construction has one named owner (`personas.ts`) but five unowned competitors (Phase 0 §1.7); behavior derivation has one named owner (`behavior-flags.ts`) but three independent invocations with differing input completeness; two context-summarization pipelines and two usage-limit implementations coexist with no shared config; and the two most identity-defining behavior flags (`warmth`, `initiative`) have no closed-loop verification at all. None of this is new information — Phase 0 catalogued all of it as technical debt — but viewed through an architectural-quality lens rather than an inventory lens, these are not isolated debts, they are a **consistent pattern**: every subsystem that has exactly one designated owner in documentation has, in practice, at least one undocumented competitor doing the same job.

**Operationally**, the architecture has essentially no dedicated infrastructure: no correlation IDs, no distributed tracing, no metrics collection, and logging that is a mix of `console.warn`/`console.error` with prefixes that vary by file (`[EMMA]`, `[EMMA API]`, `[CostGate]`, `[Memory DB]`) plus ad hoc `Sentry.captureException`/`captureMessage` calls at exactly three call sites in the entire AI pipeline (`route.ts:509`, `:637`, `:665`). For a system the roadmap itself will later route "hybrid" and "self-hosted" inference through — configurations with materially higher and more variable failure rates than a single managed API — this is a gap in the foundation the roadmap's later phases will build on, not a cosmetic one.

**Extensibility toward the roadmap's stated future** (multiple concurrently-available providers, local/hybrid inference, capability routing, tool execution, agent workflows) is uneven: the hardest two items on that list — tool execution and agent workflows — are not aspirational, they already exist and already route through the Gateway (`agent-loop.ts`). The easiest-sounding item — adding a second provider — has a well-documented extension point but is **unverified in practice**: the entire normalized contract (streaming shape, error taxonomy, finish-reason vocabulary) has exactly one implementation to prove it against, and this review found no test, mock, or second implementation anywhere in the repository that exercises the `BrainProvider` interface with a differently-shaped backend. Routing/capability-registry readiness is honestly scoped as "not yet" by ADR-0003 itself and this review concurs there is no partial or premature routing logic to evaluate.

### 1.2 Review Methodology

This review was performed by direct reading of the Brain Gateway's implementation (`gateway.ts`, `types.ts`, `providers/openrouter.ts` — read in full) and its immediate operational surroundings (`src/lib/errors.ts`, `src/core/cost-gate.ts`, `src/core/security/sanitise.ts`, `src/core/env-validation.ts`, `src/core/models.ts`, `src/core/personas.ts`, `src/core/response-validator.ts`, `src/core/context-manager.ts`, `src/core/memory-db.ts`'s retrieval function, `src/core/agent-loop.ts`'s gateway call sites, `src/app/api/emma/route.ts`'s Sentry call sites and rate-limit block — read in full or in targeted excerpt), cross-referenced against Phase 0's exhaustive inventory (§1–§9) and its Independent Review, plus the governing documents (Roadmap v1.0, ADR-0003, ADR-0001, ADR-0002, `phase7a-brain-architecture-readiness.md`, `phase7b-brain-gateway-implementation-report.md`). Repo-wide greps were run for operational-architecture evidence (Sentry usage, correlation/trace/request IDs, metrics libraries) to avoid relying on Phase 0's narrower per-file citations for a question Phase 0 was not scoped to answer. Per the roadmap's own instruction, Phase 0 facts already established (call-site counts, dependency directions, boundary ownership) are cited, not re-verified.

This is a **quality** review, not a **fact** review: where Phase 0 answers "what exists," this document answers "how good is what exists, judged as architecture." Findings are evidence-based (file/line cited) but the judgments — cohesion, coupling, extensibility ratings, risk severities — are this review's assessment, not restatements of Phase 0's neutral inventory.

### 1.4 Architecture Assessment

#### 1.4.1 Architectural Responsibility Review

Responsibility separation is **strong at the Gateway/Provider seam and weak above it**. `gateway.ts` does exactly one thing — hold the provider instance and re-export four functions — and its own header comment is explicit about what it does _not_ own (cost gating, persona/prompt content, memory ranking, behavior flags, emotion/routine tag conventions, Sentry policy, SSE envelope shape). That negative-space declaration, written into the file itself rather than only in ADR-0003, is unusual discipline and this review treats it as evidence the boundary was designed with responsibility separation as a first-class concern, not an afterthought.

Above the Gateway, responsibility is nominally clear (Phase 0 §8's ownership table names a single owner for every one of the six ownership questions it posed) but **mixed in practice** in three places this review confirms are architectural, not incidental:

- **Prompt construction** is owned by `personas.ts` in name, but this review directly confirms (`personas.ts:35-125`) that persona system prompts, response-length rules, avatar-expression instructions, and routine instructions are all interleaved as string literals inside one file with no internal module boundary between "persona voice" and "protocol instructions" (the `[EMMA_ROUTINE]`/`[emotion:]` tag contracts) — two concerns ADR-0003 itself treats as separable (the Gateway explicitly "does not know what `[emotion:...]` or `[EMMA_ROUTINE]` mean" — they are Application Layer conventions that merely travel through it). Mixing persona content and machine-protocol instructions in the same undifferentiated prompt string is a responsibility-mixing pattern independent of the five-competing-prompts issue Phase 0 already flagged.
- **Response validation** (`response-validator.ts`) is architecturally honest about a narrower responsibility than its name suggests: the file's own docstring states it is "deliberately lightweight and lenient... it NEVER rewrites the response," and this review's direct read confirms the implementation checks exactly 3 heuristics (emoji presence, prose sentence count, teasing-marker regex) against 3 of the 5 behavior flags. This is not a mixed responsibility — it is a **narrower one than its position in the architecture implies**: it sits where a real validation/repair gate would sit (post-generation, pre-delivery) but performs only observability, a distinction easy to miss without reading the implementation.
- **Cost tracking** (`cost-gate.ts`) cleanly owns budget/rate decisions and delegates identity resolution and persistence to injected dependencies (a genuinely testable, DI-shaped module — see §2), but this review confirms it is invoked as a **second**, independently-configured check on top of a **separate** distributed rate limiter already applied earlier in `route.ts` (Phase 0 §1.7) — the same request is subject to two different rate-limiting responsibility-holders with no single authority over "is this request allowed."

**Judgment:** responsibility separation is a design principle the Gateway itself honors structurally. The Application Layer honors it as a _documentation_ pattern (one named owner per responsibility) that the implementation does not consistently enforce (multiple invocation points, competing prompt sources, split rate-limiting authority).

#### 1.4.2 Boundary Review

Component boundaries are evaluated in full in §3 (a required Phase 1 deliverable). Summary judgment here: the Application/Gateway/Provider boundary (ADR-0003's three-tier split) is **well-defined, enforceable by direct inspection, and consistently upheld** — this is the one boundary in the system with an explicit, written "must not do" contract (ADR-0003's Architectural Boundaries section, and the finer-grained version Phase 0 §7 derived from it) and confirmed grep-clean adherence. The finer-grained boundaries inside the Application Layer (Behavior vs. Prompt Builder, Memory vs. Behavior, Emotion vs. everything downstream of it) are **defined but not enforceable by any mechanism** — they hold because each module happens not to import across the boundary today, not because anything (a lint rule, a module-boundary tool, a runtime assertion) would catch a violation if one were introduced. The TTS/STT boundary is **well-defined in scope (explicitly excluded by ADR-0003) but structurally inconsistent with the rest of the system**: it is a second, unrelated "provider integration" pattern (direct HTTP, per-client encrypted-Supabase-stored keys) that a reader encountering "Brain Gateway" as the inference boundary would not expect to coexist, unremarked, one directory over.

#### 1.4.3 Dependency Architecture Review

Full assessment in §4. Summary: dependency direction is **consistently downward and violation-free** at the layer that matters most — Phase 0 confirmed zero UI-layer imports of `src/core/brain/*`, and this review's direct read of `gateway.ts`/`openrouter.ts` confirms the reverse direction also holds (the provider file imports only `models.ts` and shared error utilities, never anything upstream of it). No cyclic dependencies were found or suspected in the reviewed files. One **hidden dependency** is confirmed directly by this review: `providers/openrouter.ts:14` imports `BRAIN_MODELS`/`VISION_MODELS`/`UTILITY_MODELS` from `src/core/models.ts`, meaning the _task-tier → concrete-model_ mapping — arguably an application-level product decision (which model serves "brain" requests) — physically lives in a file two hops from any application caller, reachable only by knowing to look inside the provider. A caller cannot discover which model actually serves `task: "brain"` without reading the provider's internals, even though task selection is the caller's own decision.

#### 1.4.4 Provider Isolation Review

**Isolation is total for the one provider that exists.** Every piece of OpenRouter-specific knowledge this review can identify — the two hardcoded URLs (`gateway.ts` does not know them; only `openrouter.ts:31-32` does), the auth header shape (`openRouterHeaders()`, `openrouter.ts:43-54`), the `models: [...]` fallback-array request field (`buildChatBody`, `openrouter.ts:133-153`), SSE chunk parsing (`OpenRouterStream.events()`, `openrouter.ts:186-230`), and the 529→`OVERLOADED` status mapping (`openrouter.ts:107-129`) — lives inside one 312-line file. Nothing in `gateway.ts` or `types.ts` references OpenRouter by name, URL, or wire shape.

Isolation for **future** providers (Ollama, vLLM, LM Studio) is **structurally plausible but empirically unproven**. The `BrainProvider` interface (`types.ts:163-169`) is minimal — four methods, no OpenRouter-shaped parameters — which is a good sign for portability. But this review found no second implementation, no mock provider used in tests to exercise the interface's polymorphism, and no evidence any contributor has attempted to satisfy `BrainProvider` against a genuinely different transport (e.g., a provider without SSE, or one whose fallback semantics differ from an array-of-model-IDs). The `BrainRequestError.code` enum (`BAD_REQUEST | AUTH_ERROR | RATE_LIMIT | OVERLOADED | TIMEOUT | UPSTREAM_ERROR`) is OpenRouter's own failure taxonomy relabeled — reasonable as a first cut, but whether it is actually provider-_neutral_ (as opposed to "the one taxonomy this exercise has been checked against") cannot be confirmed without a second provider to test it against. This is a normal state for a single-provider system and not itself a defect, but it means the isolation claim — accurate today — carries more uncertainty than its confident, settled prose in ADR-0003 and the Phase 7B report might suggest to a reader who has not read the code.

#### 1.4.5 Routing Architecture Review

There is no routing architecture, by design and by ADR-0003's explicit scope exclusion, and this review confirms the code matches that claim — `TASK_MODELS` (`openrouter.ts:37-41`) is a static compile-time map from the closed `BrainTask` union to a fallback array; there is no runtime selection logic, no scoring, no capability matching. **Routing ownership**, when it eventually exists, has an ambiguous future home: ADR-0003's own diagram places "routing/registry logic" inside the Gateway box, but the _current_ task-tier-to-model mapping — the closest thing to routing that exists — lives inside the _provider_ file, not the gateway. Whether future routing (which the roadmap's Phase 3 will design) should live in `gateway.ts` (above provider selection, choosing _which_ provider) or remain provider-internal (each provider choosing _which model within itself_) is genuinely two different architectural shapes, and this review notes the current code offers a precedent for the second shape (provider-internal fallback) without having ever exercised the first shape (cross-provider selection) even once. **Routing extensibility**: the `BrainTask` union is closed (`"brain" | "vision" | "utility"`) and capability-tier-shaped rather than model-ID-shaped, which is the correct abstraction level for future capability-driven routing to extend without a caller-facing breaking change — this is a genuine extensibility strength, assessed independently of the fact that no routing logic yet consumes it.

#### 1.4.6 Context Pipeline Review

**Context ownership is split, not layered.** Two independent pipelines exist with no shared configuration, no shared trigger condition, and no shared persistence model (Phase 0 §1.4, confirmed directly in this review by reading `context-manager.ts:8-24`): a client-side pipeline keyed on a 100k-token budget model with `estimateTokens()` doing 4-char-per-token approximation (`context-manager.ts:33-38`), triggering at 75% utilization, versus a server-side pipeline (`route.ts`) keyed on a flat 20-message hard cap with **no token accounting of any kind**. These are not two views of one context-budget concept; they are two different concepts (token-budget-aware vs. message-count-aware) that happen to both be called "context management." **Context composition** (what goes into a request) is owned end-to-end by `route.ts` as the composition root — a defensible single-owner pattern — but the _pipeline_ feeding it (client trim/summarize → server truncate) has no single component that could answer "what is the actual context budget for this request" without reading both files. **Lifecycle**: client-side summaries live only in the browser's in-memory message array (never persisted); server-side summaries persist to `conversations.summary`, encrypted. A single request can carry both simultaneously (Phase 0, confirmed) — this is not a lifecycle inconsistency in the sense of data loss, but it is an architecture with no single source of truth for "what does the model already know from earlier in this conversation."

#### 1.4.7 Memory Architecture Review

**Retrieval architecture**, confirmed by direct read of `getRelevantMemoriesForUser` (`memory-db.ts:120-160+`): every call fetches **all** active memory rows for the user from Supabase (`.eq("status", "active").order("created_at", ...)`, no DB-side `LIMIT`), then only applies keyword-overlap × confidence scoring in application memory, and only if the row count exceeds the requested `limit` (default 15). This is architecturally sound _given_ the 200-active-memory-per-user cap Phase 0 documented — full scan of ≤200 rows scored in-process is not a performance concern at that ceiling — but the architecture has **no ranking capability that would survive raising that cap**: there is no database index or column supporting relevance ranking (no embedding column, confirmed absent by Phase 0), so "raise the memory cap" and "add semantic ranking" are coupled changes, not independent ones, despite looking like two unrelated product decisions from outside the architecture. **Injection architecture**: memories flow into the prompt exclusively through `personas.ts`'s `serializeMemories()` call (Phase 0 §1.3) — a single, correctly centralized injection point, in contrast to the fragmented prompt-construction pattern found elsewhere. **Persistence boundary**: encryption (`security/encryption.ts`) and Supabase access are both fully contained inside `memory-db.ts` — no other module performs memory I/O directly (confirmed by Phase 0 §8, "None — single owner, no split," the one responsibility Phase 0 found with zero ownership fragmentation). **Scalability**: sound to the current cap; architecturally flat past it, because retrieval sophistication and storage capacity are silently coupled by the absence of any indexed alternative.

#### 1.4.8 Behavior Architecture Review

**Personality ownership** is architecturally exemplary in its _derivation_ half and fragmented in its _invocation_ half — a distinction this review considers important because they are usually conflated. `deriveBehaviorFlags()` (ADR-0001, `behavior-flags.ts`) is a pure function with a documented, closed precedence chain (persona baseline → CustomPersona → memory preferences → emotion/time modulation) and closed enum output — genuinely good architecture: deterministic, unit-testable without I/O, single derivation logic. But Phase 0 confirms (§1.7, §8) this pure function is **called from three independent locations** (server `route.ts`, client `page.tsx`, `agent-loop.ts`'s notification path) with differing input completeness, and this review's read of `agent-loop.ts` context confirms the third call site hardcodes `personaId: "mommy"` regardless of the user's actual persona (Independent Review §1.6, claim 13, verified). A pure function called from three places with three different input sets is not "one owner, three call sites" — it is three independently-drifting _effective_ derivations that happen to share code. **Tone generation** is owned downstream by `personas.ts` rendering `renderBehaviorDirectives()`'s diff-against-baseline output — a genuinely elegant mechanism (only non-default flags render, keeping prompt bloat to ADR-0001's stated ≤8-line budget) — but tone generation for the WhatsApp channel bypasses this entire mechanism (Phase 0 §1.4), meaning "how Emma's tone is generated" has two disconnected answers depending on channel. **Composition**: behavior flags compose cleanly with prompt directives (one flag → one directive line, additive) but do not compose at all with response generation once the response exists — `warmth` and `initiative` are rendered as directives and then structurally unverified (§1.4.9 below), meaning two of five flags participate in composition only going in, never coming back out as a closed loop.

#### 1.4.9 Emotion Architecture Review

**Ownership**: emotion fusion (`emotion-engine.ts`) is a client-only React hook — an architectural choice, not an oversight, since it fuses live signals (voice RMS/ZCR, a server-side vision call, text keywords) that are naturally client-local except for the vision leg, which itself round-trips through the Gateway. **Lifecycle**: `EmotionState` is re-derived per interaction from live signal capture with no persistence of its own — ADR-0002 (Companion State) explicitly chose to persist only a _snapshot_ of the last fused state (`last_mood`, `last_emotion`), not a running history, precisely to keep emotion state architecturally ephemeral by design rather than accumulate an unbounded time series. This is a coherent, intentional lifecycle decision, not an accidental gap. **Downstream integration** is the one place this review finds emotion architecture genuinely disciplined: Phase 0 §7 confirms raw `EmotionState` has exactly two legitimate consumers (Behavior, Prompt Builder) and every other downstream system (avatar, voice, greeting, proactive speech) consumes only the _derived_ `BehaviorFlags`, never raw emotion — a real, enforced (by absence of alternate import paths, per Phase 0's grep) architectural constraint that prevents emotion-interpretation logic from being reimplemented at each consumer. **Architectural consistency**: undermined by one fact this review flags independently — `emotionState` in the chat request body is entirely client-asserted (Phase 0 §1.7) and feeds directly into a companion-state row that is later described in ADR-0002 as containing "among the most intimate data Emma holds." An architecture that treats a value as trustworthy enough to drive behavior derivation and persist to an encrypted intimate-data table, while that same value has no server-side corroboration, has an internal consistency gap between how seriously the _storage_ layer treats the data (AES-256-GCM, RLS, GDPR export) and how seriously the _ingestion_ layer verifies it (not at all).

#### 1.4.10 Prompt Architecture Review

**Ownership**: nominally `personas.ts`, actually six competing owners counting the five Phase 0 identified (`vision/route.ts`, `summarize/route.ts`, `ingest/whatsapp/route.ts`, and two in `history/route.ts`) plus the internal mixing this review identified in §1.4.1 (persona voice and protocol-tag instructions interleaved in one string). **Composition**: `buildSystemPromptBlocks()`'s stable/dynamic split (Phase 0 §1.3) is a genuinely useful architectural seam — it anticipates prompt-caching (explicitly named as a future extension point in the Phase 7B report) by already separating what changes per-turn from what doesn't — but `buildSystemPrompt()`, the _only_ function any caller actually invokes, flattens that seam back into one string, meaning the seam currently provides zero runtime value; it is dormant infrastructure for a future capability the Gateway would also need to expose (per the Phase 7B report's own note) before it could be used. **Reuse**: near-zero across channels — the standard chat flow, vision, summarization, and WhatsApp each construct prompt text independently with no shared template, shared tag-guard helper (the `[EXTERNAL DATA]` prompt-injection guard is implemented three separate times, Phase 0 §1.7), or shared persona-voice fragment. **Isolation**: good in one specific respect — prompt text never crosses into the Gateway or provider layer as anything but opaque message content (ADR-0003's boundary holds); poor in the cross-channel sense just described. **Maintainability**: a change to Emma's core tone rules (e.g., the "baby"/"mmm"/"ahh" vocabulary in `personas.ts:41-57`) has no mechanism to propagate to the five other hardcoded prompts, meaning tone consistency across channels is maintained by remembering to update six files, not by architecture.

#### 1.4.11 Validation Architecture Review

**Response validation** (`response-validator.ts`) is architecturally coherent for what it actually is — a narrow, log-only, 3-of-5-flag observability check, confirmed by direct reading (§1.4.1) — but is positioned in the request lifecycle exactly where a repair/gating mechanism would sit (`route.ts` step 14, post-parse, pre-`done`-event), which this review considers a naming/positioning risk more than an implementation flaw: nothing about its file name, its call site, or its return shape (`{ consistent, violations }`) signals "observability only" to a reader who has not read ADR-0001's explicit "never rewrites" design note. **Safety validation** is a different, better-separated concern: `sanitiseInput()` (`security/sanitise.ts`) is a genuinely self-contained four-layer pipeline (length → control-char strip → repeated-char collapse → 19 regex injection patterns) with its own closed threat taxonomy and a hard block on any single high-severity match, confirmed by direct read to run _before_ the LLM is ever called (short-circuits at `route.ts` step 9, per Phase 0 §1.4) — this is real gating, not observability, and it is the one validation mechanism in the reviewed system that actually blocks. **Repair pipeline**: does not exist anywhere in the reviewed architecture — there is no mechanism, at any layer, that takes a non-conforming model output and attempts to correct it rather than merely log or discard it. **Validation ownership** is therefore split three ways with three different postures: `sanitiseInput` (pre-inference, blocking, security-motivated), `response-validator.ts` (post-inference, observability-only, behavior-motivated), and `request-validation.ts` (schema-shape, blocking where used — `history/route.ts`, `agent/route.ts` — but, confirmed by Phase 0 and unchanged as of this review, **not wired into the main chat route at all**, meaning the highest-traffic endpoint in the entire system has no request-body schema validation of any kind, only the downstream cost-gate/rate-limit checks and the sanitiser's plain-text heuristics).

#### 1.4.12 Configuration Architecture Review

**Configuration ownership** is split along a defensible line (model IDs in `models.ts`, per-client product config in `client-config.ts`, environment shape in `env-validation.ts`) with one confirmed leak: the embedding model ID lives in `providers/openrouter.ts:35`, not `models.ts`, justified in-file as "provider configuration, not caller configuration" — a defensible architectural position (the embedding model choice genuinely is a provider-internal decision, unlike brain/vision/utility which are task-tier decisions the caller cares about), but it means "the model registry" is not, in fact, one file; it is one file plus one documented exception, and a reader consulting only `models.ts` would not find every model ID Emma uses. **Feature flags**: none exist as a distinct mechanism anywhere in the reviewed AI pipeline — behavior varies by persona, plan tier, and behavior flags, none of which are feature-flag-shaped (boolean or staged rollout); this is a neutral observation (the system does not need feature flags for what it does today), not a gap relative to any stated requirement. **Provider configuration**: `env-validation.ts`'s `PRODUCTION_REQUIRED_ENV` (confirmed by direct read, lines 15-29) lists `OPENROUTER_API_KEY` as unconditionally required in production — a single-provider assumption hardcoded into the _validation_ layer, one level more structural than a normal application-code assumption, because it means production boot itself fails without OpenRouter specifically, not merely "without a configured provider." This is already-documented, deliberately-deferred debt (Phase 0 §1.7, Phase 7B report "Remaining technical debt") — this review's contribution is the observation that it sits in configuration architecture specifically, meaning multi-provider readiness is blocked one layer earlier than the Gateway itself: before a second provider's _code_ could even be reached, production environment validation would need to stop assuming OpenRouter is the only valid provider. **Runtime configuration**: task-tier→model mapping is compile-time static (`TASK_MODELS`, `openrouter.ts:37-41`) with no runtime override mechanism (no env var, no database row, no admin toggle) — changing which model serves `"brain"` requests requires a code change and redeploy, which is architecturally consistent with the "DEV: free tier, LAUNCH: claude-sonnet-4-5" comments in `models.ts` signaling this is an accepted, temporary state rather than a permanent design choice.

#### 1.4.13 Extension Architecture Review

Evaluated per the roadmap's named examples, readiness only, no redesign:

- **Multiple providers**: extension point exists and is documented (`BrainProvider` interface + `gateway.ts` selection point); mechanism is sound; **proof is single-sample** (§1.4.4) — this review rates readiness _procedurally high, empirically unverified_.
- **Local inference / hybrid inference**: no code-level blocker was found in the Gateway/Provider layer itself (a local provider is "just another `BrainProvider`" by the same mechanism as any other new provider), but **configuration architecture is not ready** — the unconditional `OPENROUTER_API_KEY` requirement (§1.4.12) would need to change before a hybrid deployment (some requests local, some OpenRouter) could even boot in production under current validation rules, and there is no provider-selection-per-request mechanism today (the Gateway holds exactly one module-level provider instance, `gateway.ts:51`, chosen once at import time, not per-request).
- **Multi-model routing / capability routing**: no partial implementation exists to build on or be misled by (confirmed §1.4.5) — this is a clean-slate readiness question for Phase 3, not an extension-of-existing-code question. The task-tier abstraction (`BrainTask`) is, independent of routing itself, already the right shape for capability-driven routing to extend.
- **Tool execution**: **already implemented and already routes through the Gateway.** `agent-loop.ts` passes `tools: BrainToolDefinition[]` and consumes `toolCalls`/`tool_calls` through `brainChat()` (confirmed by direct read, `agent-loop.ts:18,358-403`). This is not a future capability the architecture must accommodate — it is a present capability already proven against the real contract, and the strongest piece of extensibility evidence in this review.
- **Agent workflows**: likewise already implemented (`agent-loop.ts`'s main loop + `evaluateTool` two-call-site pattern, Phase 7B report). The Gateway's contract has already been exercised by the single most demanding caller pattern in the system (multi-turn, tool-calling, resumable via `tasks.step_transcript`) without requiring a Gateway-shape change — direct evidence the normalized contract is not merely simple-chat-shaped.

**Judgment**: extension readiness is inverted relative to how the roadmap lists these examples — the items that sound most advanced (tool execution, agent workflows) are done; the item that sounds simplest (a second provider) is architecturally supported but has never been exercised.

#### 1.4.14 Operational Architecture Review

This is the weakest-evidenced domain in the reviewed architecture, confirmed by direct grep and file reads rather than inference from Phase 0. **Logging**: no shared structured-logging utility exists for the AI pipeline; every module prefixes its own `console.warn`/`console.error` calls with its own ad hoc tag (`[EMMA]` in `errors.ts`, `[EMMA API]` in `route.ts`, `[CostGate]` in `cost-gate.ts`, `[Memory DB]` in `memory-db.ts`), confirmed by direct reading of all four files — this is already named as accepted debt in the Phase 7B report ("Console-log prefixes across routes remain heterogeneous... preserved to keep observability text stable"). **Tracing**: a repo-wide grep for `correlationId|traceId|requestId|OpenTelemetry|opentelemetry` returns matches only in `src/core/account-deletion/` — a wholly unrelated subsystem — meaning **the AI request pipeline has no request-correlation mechanism at all**. A single chat turn crosses at minimum: sanitisation, two rate/cost checks, memory fetch, behavior derivation, prompt build, the Gateway, the provider, response parsing, validation, and (asynchronously) TTS — with no shared identifier a log aggregator or Sentry issue could use to reconstruct that these entries belong to one user-visible turn. **Metrics**: the same grep for `metrics\.|statsd|posthog` returns zero AI-pipeline matches; no counters, histograms, or gauges exist for gateway call volume, latency, retry rate, or per-provider error rate — the only quantitative signal captured anywhere is `cost-gate.ts`'s own token/request accounting, which serves billing, not operational health. **Error visibility**: Sentry is wired but sparse — confirmed exactly three call sites in `route.ts` (a "Brain provider error" message at :509, and two `captureException` calls at :637/:665) plus independent usage in `agent-loop.ts` and `admin-diagnostics.ts`; no Sentry instrumentation exists inside the Gateway or provider files themselves, meaning a provider-level failure is only visible to Sentry if the specific calling route happens to capture it — the Gateway's own header comment confirms this is intentional ("Sentry capture policy" is explicitly listed as _not_ the Gateway's responsibility), which is a defensible boundary decision but currently has no compensating owner: no call site this review checked treats "capture on gateway failure" as _its_ job either, it is simply omitted at most of the 16 invocation sites (only the primary chat route captures).

**Judgment**: operational concerns are **not architecturally integrated** — they exist as scattered, per-file conventions rather than as a designed cross-cutting layer, and the one place a cross-cutting concern (Sentry) is explicitly scoped out of the Gateway's responsibility, no other component has explicitly picked it up.

#### 1.4.15 Error Handling Architecture Review

Added as a dedicated subsection per the Phase 1 Independent Review's finding that the roadmap's named scope items "error handling" and "retry strategy" had previously been addressed only incidentally within other subsections rather than analyzed in their own right.

Error representation in the reviewed system is **three-shaped, not one**. (1) At the Gateway/Provider boundary, `BrainRequestError` (`types.ts:97-105`) is a normalized value object with a closed `code` enum (`BAD_REQUEST | AUTH_ERROR | RATE_LIMIT | OVERLOADED | TIMEOUT | UPSTREAM_ERROR`), a `status`, and a `retryable` boolean, constructed in exactly one place (`normalizeHttpError`, `openrouter.ts:109-129`) — this is the reviewed system's cleanest error-modeling and the direct product of ADR-0003's normalization principle. (2) Independently, `src/lib/errors.ts` defines a thrown `EmmaError` class hierarchy (`ApiError`, `AuthError`, `RateLimitError`, `RateLimitUnavailableError`), each carrying its own `code`/`status`/`retryable` triple — structurally similar to `BrainRequestError` but a separate type family with no shared base type or conversion function connecting the two. (3) `cost-gate.ts`'s `CostGateDecision` (`cost-gate.ts:101-116`) is a third, ad hoc discriminated union (`reason: "identity_unavailable" | "budget_exceeded" | "rate_limited" | "metering_unavailable"`) that overlaps in purpose (rate/budget failure) with `RateLimitError`/`RateLimitUnavailableError` from (2) without reusing them. **User-facing translation** is a fourth, disconnected layer: `getPersonaErrorMessage()` (`errors.ts:132-151`) switches on a raw numeric HTTP status (400/401/429/500/502/503/504/529) to produce in-persona copy, keyed independently of the `BrainRequestError.code` taxonomy the Gateway already computed for the same failure — a caller holding a `BrainRequestError` has both `.status` and a normalized `.code`, and the persona-message function consults only the former, silently bypassing the latter's normalization work. This is not a correctness defect today (status and code are derived from the same source and stay in sync) but it is a second, parallel error vocabulary doing work the first one already did.

**Judgment**: error _normalization_ at the Gateway boundary is a genuine architectural strength (§6, Strength #4). Error _representation_ system-wide is fragmented into three independent shapes plus a fourth translation layer that bypasses the most normalized of the three — the same "one named owner, one undocumented competitor" pattern found elsewhere in the Application Layer (§1.1), here applied to failure handling rather than prompt/behavior/context.

#### 1.4.16 Retry Strategy Architecture Review

Retry **mechanism** is properly centralized: `fetchWithRetry()` (`src/lib/errors.ts:61-128`) is the single implementation of exponential-backoff retry in the reviewed codebase, and the OpenRouter provider's `sendChatRequest()` (`openrouter.ts:155-168`) and `embed()` (`openrouter.ts:288-296`) both call it rather than reimplementing retry logic — no duplicate retry loop was found anywhere in the reviewed files. Retry **policy**, by contrast, is set per call site with no documented rationale distinguishing the choices: the shared default (`DEFAULT_RETRY`, `errors.ts:51-56`) is `maxRetries: 3`, but the Gateway's own request translation overrides it downward per request (`request.maxRetries ?? 0` at `openrouter.ts:164`) — meaning **zero retries is the Gateway's effective default** unless a caller explicitly opts in. Phase 0 confirms the main chat route opts in at `maxRetries: 2`, and this review's direct reading of `agent-loop.ts` confirms it opts in at `maxRetries: 2` for its main tool-calling loop but stays at the zero default (`maxRetries: 0, timeoutMs: 8_000`) for tool-evaluation (`agent-loop.ts:358,1087`). Each of these is a silent, per-call-site choice with no accompanying rationale (no comment anywhere explains why tool-evaluation should retry less than the main chat turn). **Retry-eligibility boundary**: `DEFAULT_RETRY.retryOn` (`429, 500, 502, 503, 529`) is fixed and itself carries provider-specific vocabulary — 529 is an OpenRouter/Anthropic-specific status, defined independently in both `errors.ts`'s shared retry list and `openrouter.ts`'s own normalization — meaning a status code ADR-0003 treats as something "the Gateway normalizes so callers never see it" is simultaneously hardcoded one layer up, in a utility that predates and sits outside the Gateway's boundary. **Timeout coupling**: `connectionTimeoutMs` (`errors.ts:73-76`) aborts only when no response _headers_ arrive within the window — a request that starts streaming and then stalls mid-body is not covered by this mechanism at all, a boundary condition the Phase 7B report itself accepted as a known behavior delta ("connection timeout, not whole-request abort") but one this review found not re-examined anywhere else in this document.

**Judgment**: the retry _mechanism_ is a real, single-implementation, DRY strength. Retry _policy_ is architecturally under-specified — each of the Gateway's 16 invocation sites makes its own silent retry-count decision with no shared policy or stated per-task-tier default, and the one shared retry-eligibility list already contains provider-specific vocabulary (529) at a layer that sits above the Gateway boundary meant to contain exactly that kind of knowledge.

## 4. Dependency Assessment

Extends Phase 0 §4 (Dependency Map, factual) with a quality judgment.

**Dependency direction**: consistently downward through the reviewed layers — UI → API routes → Application modules → Gateway → Provider → external API — with no upward or cross-layer violation found in any file this review read directly. This is the architecture's most consistently strong property across every domain in §1.4.

**Cyclic dependencies**: none found or suspected. The reviewed files form a strict DAG; `gateway.ts` depends on `types.ts` and `providers/openrouter.ts`, which depend on `models.ts` and `lib/errors.ts`, neither of which depend back on anything in `src/core/brain/`.

**Dependency inversion**: present and well-used at exactly one point — `gateway.ts` depends on the `BrainProvider` _interface_ (`types.ts`), not on `createOpenRouterProvider` as a concrete type, satisfying the Dependency Inversion Principle in the one place the architecture most needs it (provider substitutability). `cost-gate.ts`'s `CostGateDependencies` injection pattern (§1.4.1) independently achieves the same inversion for its own testability. Neither pattern is replicated elsewhere in the reviewed Application Layer modules (`personas.ts`, `behavior-flags.ts`, `memory-db.ts` all depend on concrete Supabase/encryption modules directly), meaning dependency inversion is a pattern the codebase knows how to use but applies inconsistently — present exactly where two different engineers independently needed it for two different reasons (provider swap, unit-testability), absent elsewhere.

**Unnecessary dependencies**: none identified in the Gateway/Provider files. One **near-miss** is noted: `providers/openrouter.ts` imports all three model-array constants from `models.ts` even though a single request only ever needs one (`TASK_MODELS[request.task]`) — architecturally harmless (it's a static import of small arrays, not a runtime cost) but worth noting as the one place the provider's import surface is broader than its per-call need.

**Hidden dependencies**: one confirmed (§1.4.3) — the task-tier→model mapping's true location inside the provider rather than the nominal model registry. A second, softer instance: `env-validation.ts`'s unconditional `OPENROUTER_API_KEY` requirement (§1.4.12) is a hidden dependency from _deployment/boot_ on a specific provider that the Gateway's own contract claims not to have.

**Layering violations**: none found within the reviewed files. The one layering question this review could not fully resolve from code alone is whether TTS/STT's direct-provider pattern constitutes a _violation_ of the Gateway's "single inference entry point" principle or a _deliberately separate_ layer — ADR-0003's text supports the latter reading explicitly ("Out of Scope"), so this review records it as a scope boundary, not a violation, consistent with treating ADR-0003 as authoritative for what it claims to govern.

## 6. Architecture Strengths

Documented per the phase's explicit instruction that strengths must be recorded alongside weaknesses.

1. **The Gateway boundary is structurally, not conventionally, clean.** Zero provider-wire leaks confirmed by two independent grep sweeps (Phase 0's and this review's) and by this review's direct reading of every file inside `src/core/brain/`. This is the single strongest architectural property in the reviewed system.
2. **The Gateway's own header comment declares what it does _not_ own.** A negative-space contract written into the implementation itself (not only the ADR) is unusual engineering discipline and makes the boundary legible to a reader who never opens ADR-0003.
3. **Task-tier abstraction (`BrainTask: "brain" | "vision" | "utility"`) rather than a model-ID parameter.** Callers never choose a model; they choose an intent. This is exactly the right abstraction level for the capability-driven routing the roadmap's later phases intend to build, and it already exists.
4. **The error contract's dual mode (values for HTTP failures, throws for transport failures) deliberately mirrors the pattern every pre-existing call site already used**, documented as a conscious migration-risk-reduction decision (Phase 7B report) rather than an arbitrary API design choice — evidence of architecture decisions grounded in the codebase's actual usage patterns, not abstract preference.
5. **Streaming is already a genuinely normalized capability, not a pass-through.** The provider synthesizes a uniform trailing `done` event with usage/finish-reason regardless of whether the underlying provider reports usage inline (OpenRouter does) or would need it synthesized (a hypothetical future provider might not) — this is forward-looking design already implemented, not merely planned.
6. **Tool-calling and agent workflows — the hardest capabilities on the roadmap's own extension checklist — are not aspirational.** They exist today (`agent-loop.ts`) and already prove the Gateway's contract against a materially harder caller pattern (multi-turn, resumable, tool-invoking) than simple one-shot chat.
7. **Dependency direction is clean and violation-free everywhere this review checked.** No cycles, no upward dependencies, and the one place dependency inversion was needed most (provider substitutability) uses it correctly.
8. **`cost-gate.ts`'s dependency-injection shape** (`CostGateDependencies`, all I/O injected) makes it independently unit-testable without network or database access — a pattern this review found nowhere else in the reviewed Application Layer, but a genuine example of good architecture existing in the codebase to potentially learn from.
9. **The behavior-flags _derivation_ function itself** (as distinct from its multiple call sites) is a well-designed, deterministic, pure, closed-enum component — exactly the kind of unit ADR-0001 intended, and this review's direct reading confirms it matches its own design document closely.
10. **`sanitiseInput()` is a real, self-contained, blocking security gate** with a documented, closed threat taxonomy — the one validation mechanism in the system that fully matches its apparent purpose with no gap between name and behavior.

## 7. Technical Debt Assessment

Categorized by architectural nature, not proposed remediation. Item detail already established in Phase 0 §1.7 is cross-referenced, not repeated; new categorization judgment is this review's contribution.

### 7.1 Structural debt (the architecture itself has more than one shape for one concept)

- Two context-summarization pipelines, two usage-limit implementations, two stacked rate-limit checks, six prompt-construction owners, three behavior-flag-derivation call sites (§1.4.1, §1.4.6, §1.4.8, §1.4.10; Phase 0 §1.7). **Common shape**: in every instance, one component is documented/named as "the" owner and at least one other component independently reimplements the same responsibility. This is the review's central structural-debt finding (§1.1).

### 7.2 Boundary debt (a boundary exists in documentation but has no enforcement mechanism)

- Behavior ↔ Prompt Builder, Emotion ↔ downstream consumers (§3 table) — both hold today by the absence of a violation, not by a structural barrier (lint rule, module boundary, interface) that would catch one.
- TTS/STT's parallel, internally-inconsistent-with-itself provider-integration pattern (§1.4.2, §3) — explicitly out of ADR-0003's scope, so not a _violation_ of the Gateway boundary, but an unenforced inconsistency in how "talk to an external inference provider" is done at all in this codebase.

### 7.3 Operational debt (no dedicated infrastructure for a cross-cutting concern)

- No tracing/correlation IDs, no metrics, heterogeneous ad hoc logging, sparse Sentry coverage (§1.4.14). This is the most uniform and least-owned debt category in the review — no single file or module is "responsible" for having not built this, because nothing in the current architecture assigns operational-concerns ownership to anyone.

### 7.4 Configuration debt (an assumption is baked in at a lower/more-structural layer than it needs to be)

- `env-validation.ts`'s unconditional `OPENROUTER_API_KEY` requirement (§1.4.12) — single-provider assumption at the boot-validation layer, one level more rigid than an application-code assumption would be.
- Embedding model ID split from the rest of the model registry (§1.4.12) — defensible in isolation, but means "the model configuration" is not actually centralized in the one file documentation implies it is.

### 7.5 Documentation/governance debt (architecture-adjacent documents describe a state the code has already outgrown)

- `ADR-0003`'s "Implementation: None yet" header (§5 risk #12; Phase 0 §1.7, §5) — carried forward from Phase 0 as a still-unresolved item as of this review; this review adds only the observation that it is specifically a risk to _this roadmap's own process_, since the roadmap treats ADR-0003 as required Phase-1 input.
- Marketing copy asserting a Claude-based model that does not match `models.ts`'s actually-wired `openai/gpt-oss-120b:free` (Phase 0 §1.7) — outside this review's architectural scope to assess further, noted only for completeness against Phase 0's original finding.

### 7.6 Unverified-assumption debt (an architectural claim is true today but has never been tested against the condition that would falsify it)

- Provider-neutrality of the normalized `BrainProvider` contract, never exercised against a second implementation (§1.4.4, §1.4.13, risk #3). This is a distinct category from the others: not a known defect, but a claim resting on n=1 evidence in an architecture explicitly designed to be judged on n≥2 provider support.

## 8. Phase 1 Conclusion

**Overall architectural maturity**: bimodal. The Brain Gateway itself — the component this roadmap is named for — is a mature, disciplined piece of architecture: single entry point, total provider isolation (proven), clean dependency direction, a minimal and already-battle-tested extension surface (proven against tool-calling and agent workflows, not just simple chat). The Application Layer surrounding it, which ADR-0003 explicitly and correctly declined to redesign, exhibits a consistent and recurring quality pattern across nearly every subsystem reviewed: a single named owner in documentation, and at least one undocumented competitor performing the same responsibility in practice (prompt construction, behavior derivation, context management, rate/usage limiting). Operational architecture — tracing, metrics, structured logging, consistent error visibility — is essentially absent as a designed layer anywhere in the system, Gateway included.

**Overall strengths**: a genuinely clean, structurally-enforced inference boundary; a task-tier abstraction well-suited to the roadmap's own future routing plans; a streaming/tool-calling contract already proven against real, hard callers rather than toy examples; correct and consistent dependency direction throughout every file this review examined; isolated but real examples of good architectural practice (dependency injection in `cost-gate.ts`, a pure deterministic core in `behavior-flags.ts`, a genuinely self-contained security gate in `sanitiseInput()`) that the codebase does not yet apply consistently elsewhere.

**Overall weaknesses**: responsibility fragmentation is the dominant pattern outside the Gateway — not isolated incidents but a recurring shape across prompt construction, behavior derivation, context management, and rate/usage limiting. Boundaries above the Gateway are defined in prose but unenforced by any mechanism. The provider-neutrality claim central to this roadmap's own premise rests on a single implementation and has never been tested against a second one. Operational architecture is not merely thin — it is architecturally unassigned, with the Gateway explicitly declining ownership of Sentry policy and no other component consistently picking it up. A governance-level inconsistency (ADR-0003's stale header) already demonstrates that this project's own ADRs can drift from the code they describe, a pattern this roadmap should be alert to in its own future phases.

**Readiness for Gap Analysis (Phase 2)**: the current architecture is understood in sufficient depth — both factually (Phase 0) and now qualitatively (this document) — to support a meaningful comparison against a target architecture. The specific gaps a Phase 2 Gap Analysis would need to reconcile are already surfaced here with evidence: single-provider-proven abstractions that need multi-provider validation, unenforced boundaries that a target architecture would need to decide whether to formalize, and an operational layer that does not currently exist to compare against a target at all (Phase 2 will need to treat "add operational architecture" as a gap with no current-state baseline to gap-analyze against, only an absence).

**Recommendation: Ready for Phase 2, with Minor Concerns.**

Justification: no finding in this review rises to a severity that would block proceeding — every risk identified (§5) is Medium or lower except two High-severity operational gaps (tracing, metrics), and both of those are gaps of _absence_ (nothing to fix, only something to design in Phase 2/3) rather than defects in what exists. The "Major Review Required" threshold is not met: this review found no architectural decision that is actively wrong, only decisions that are incomplete, unenforced, or proven only once. The "Minor Concerns" qualifier reflects specifically: (1) the unverified single-provider-proof risk, which Phase 2's own gap analysis against a multi-provider target architecture is the natural place to surface concretely rather than a reason to halt beforehand; and (2) the absence of any operational architecture, which this review recommends Phase 2 treat as an explicit, named gap category rather than let it be implicitly absorbed into "provider isolation" or "extensibility" gap analysis, where it does not naturally belong.

## Explicit Non-Goals Confirmation

Per the Phase 1 spec, this document does not redesign the Brain Gateway, does not create a target architecture, does not introduce ADRs, does not propose implementation, does not create implementation plans, does not refactor code, and does not modify runtime behavior. Every statement above is a judgment about the architecture as it exists in the repository at the time of this review, grounded in direct reading of the cited files and in Phase 0's independently-reviewed factual baseline.

The paragraph above states what this document does not **change**. It does not, by itself, state what this document does not **evaluate** — added here per the Phase 1 Independent Review's finding that this distinction was previously left implicit. This review does not evaluate: performance or load characteristics (no benchmarking, latency measurement, or throughput testing was performed — the operational-gap findings in §1.4.14/§5 describe an _absence of instrumentation_, not a measured quantity); implementation quality below the architectural level (code style, naming, test-coverage adequacy, or line-level correctness of business logic — e.g., whether `countProseSentences`'s regex counts correctly in every case — are out of scope; this review evaluates _where_ responsibilities sit and _whether_ boundaries hold, not whether the code inside them is well-written); operational validation in production (§1.4.14 is static analysis of what instrumentation exists in the code, not a review of actual incident history, Sentry issue volume, or real-world latency/error data); the effectiveness of `sanitiseInput()`'s injection-pattern regexes against adversarial input (discussed only as a boundary/gating example, §1.4.11); the cost-control adequacy of `cost-gate.ts`'s specific per-operation limits (evaluated only as a dependency-injection pattern, not a business policy); and future architecture design (correctly deferred to Phase 2/3 per the roadmap's own gating).
