## 1. Architecture Freeze Report

### 1.1 Executive Summary

This Freeze selects one architectural direction — a single candidate or an explicitly documented synthesis of multiple candidates — for every one of the nine domains named in the roadmap's Phase 3.1 scope, drawing exclusively from the 33 candidate architectures Phase 3 catalogued and validated by its Independent Review. No new candidate is introduced anywhere in this document; every decision below is a selection among, or a disclosed combination of, architectures Phase 3 already described, consistent with this phase's explicit instruction not to repeat Discovery.

Three domains resolve to a single candidate with no synthesis required (Context → Centralized Pipeline; Memory → Database-Side Ranking; Workflow → Agent-loop extension). Four domains resolve to a documented synthesis Phase 3 or its Independent Review already flagged as a natural composition rather than a novel invention (Provider → Registry + Capabilities Descriptor + Adapter Layer; Routing → the already-compositional Layered Routing candidate; Operational → Gateway-centralized instrumentation narrowed to a correlation-ID contract, built on the existing Sentry dependency; Prompt → centralization as the end-state, reached via the already-incremental layered migration path, using composable-fragment technique for each channel adapter). Two domains (Configuration; Extension/Governance) resolve to a sequenced set of decisions where not every named candidate competes for the same slot — Configuration's three candidates address three different sub-gaps rather than one shared question (a structural fact the Phase 3 Independent Review's Finding Ed-2 already identified), so this Freeze adopts one now and formally accepts-but-defers the other two; Extension/Governance's four sub-domains each resolve independently.

No Critical inconsistency was found between any two selected architectures (§1.4). Technical Design Readiness: **Ready with Minor Clarifications** (§7) — every domain has enough decided to begin Technical Design, but three specific design questions (the Capabilities Descriptor schema shape, the exact Context-Pipeline token-budget reconciliation semantics, and the correlation-ID's precise propagation contract) are explicitly left to Technical Design rather than pre-decided here, consistent with this phase's Explicit Non-Goals (no implementation specification).

### 1.2 Decision Methodology

This Freeze was performed by: (1) treating every candidate architecture in Phase 3 (§4–§11) and every finding in its Independent Review (candidate-independence relationships, trade-off ratings, risk categories, the Architecture Traceability Matrix) as the complete and only input — no repository re-inspection beyond what Phase 0–3 already performed was needed or done, per this phase's Explicit Non-Goals; (2) applying the roadmap's eight Decision Principles (preserve validated strengths, resolve one or more validated gaps, remain provider-agnostic, minimize coupling, improve long-term maintainability, support future extensibility, remain internally consistent, be evidence-justified) as the explicit test every selection below must pass, with the specific principle(s) driving each decision named in its rationale; (3) for domains where Phase 3's Independent Review had already identified a candidate as a disclosed composition of its siblings (Routing Candidate D; the Provider B/C relationship), adopting that composition directly rather than re-deriving it; (4) for domains with a genuine either/or choice (Context, Memory, Extension/Governance's sub-domains), selecting the single candidate best supported by the accumulated evidence and stating explicitly why every other candidate in that domain was not selected; (5) for domains where a synthesis was constructed rather than simply adopted from an existing Phase 3 disclosure (Operational, Prompt), documenting the synthesis explicitly as required by this phase's "no undocumented hybrid solutions" constraint — naming exactly which element of which candidate is retained and why.

### 1.4 Architecture Consistency

Every pairwise interaction between selected architectures was checked for conflict. Three load-bearing consistency relationships were confirmed, and one surface-level tension was checked and resolved:

- **Routing's Capability layer (§5B, adopted as Layer 2 of the selected Layered Routing) depends on Provider's Registry and Capabilities Descriptor (§4B/§4C) existing first.** This is not a conflict but the exact sequencing dependency Phase 3's Independent Review already identified (§2 of that review: "GAP-01 → GAP-02" prerequisite relationship). Both selections are consistent with, and mutually require, each other in the same order Phase 3 already established.
- **Configuration's Provider-Conditional Boot Validation (§10A) depends on "which provider(s) are configured" being a queryable fact.** The selected Provider architecture (Registry, §4B) supplies exactly this fact. No conflict; §10A's selection presumes §4B's selection, and both are frozen together.
- **Operational's Gateway-centralized instrumentation does not repeat Context Candidate C's rejected coupling violation.** Context Candidate C was rejected (§3) specifically because it would have taught the Gateway a business-logic fact (what to summarize, when) that ADR-0003 reserves for the Application Layer. Operational's selection teaches the Gateway nothing about _why_ a request was made — it only times, counts, and correlates requests it was already going to serve, which is infrastructure metadata about a call the Gateway already owns per ADR-0003 Principle 1 (Single Inference Entry Point). This was explicitly checked as the sharpest surface-level tension in this Freeze — two decisions that both "put more into the Gateway" — and resolved: telemetry about a request the Gateway is single-entry-point for is categorically different from a business decision (what to summarize) the Gateway was never meant to make. No conflict.
- **Prompt's centralized end-state (§8A) and Extension's tooling-enforced DI (§11.2C) are mutually reinforcing, not merely compatible.** A centralized Prompt Pipeline is exactly the kind of new, substitutable component DI as a systemic convention (§11.2C) is meant to apply to going forward — the Prompt Pipeline should be built DI-shaped from its first line, per the same pattern `cost-gate.ts` already demonstrates. No conflict; this is recorded as a positive interaction the Selected Architecture Catalog (§2) notes under Prompt's rationale.

**No conflict requiring resolution was found between any two selected architectures.**

## 2. Selected Architecture Catalog

For every domain: selected architecture, rationale, supporting evidence, accepted trade-offs. Decision Principles invoked are named explicitly per domain.

### 2.1 Provider

**Selected:** Provider Registry (§4B) as the selection mechanism, with Capabilities Descriptor (§4C) attached to each registry entry, and a distinct Adapter Layer (§4D) for wire-format/error normalization — composed exactly as Phase 3 itself described their relationship ("§4C composes naturally with §4B... rather than competing with it"; §4D addresses an orthogonal question, "how a provider's wire format is normalized," not how one is selected).

**Rationale:** GAP-01 (unverified provider-neutrality), GAP-03 (boot-time single-provider lock-in), and GAP-06/GAP-07 (fragmented error representation, provider-specific vocabulary in shared code) are four distinct problems that the status-quo-extended Candidate A cannot resolve simultaneously — A leaves provider selection static (does not address GAP-03's need for a queryable "which providers are configured" fact) and does nothing for GAP-06/GAP-07 (error/retry normalization). The Registry supplies the queryable selection surface every other domain's dependent decisions need (Routing's capability layer, Configuration's boot validation); the Capabilities Descriptor makes provider differences explicit rather than assumed away, directly targeting GAP-01's core problem (a contract "proven" only by inspection, never by a second implementation); the Adapter Layer gives every future provider one place to translate its own error vocabulary into the Gateway's normalized taxonomy, directly closing GAP-06 and removing the hardcoded `529` from `errors.ts`'s shared, provider-agnostic-by-design retry list (GAP-07). Decision Principles invoked: _resolve one or more validated gaps_ (all four Provider-domain gaps addressed, none left partially open), _support future extensibility_ (ADR-0003's own "Future Providers" section anticipates a registry-shaped mechanism without naming one), _preserve validated strengths_ (the `BrainProvider` four-method interface itself is untouched — Registry/Descriptor/Adapter all sit around it, not through it).

**Supporting evidence:** Phase 3 §4 (all four candidates); Phase 2 §6 ("the Gateway holds exactly one module-level provider instance, chosen once at import time, not per-request"); ADR-0003 "Future Providers"; Phase 3 Independent Review §4 (confirms B→C composition is disclosed, not hidden duplication).

**Accepted trade-offs:** Medium complexity for the Registry, Medium-High for the Descriptor (schema-design risk, per Phase 3 §4C weaknesses) — accepted because the alternative (Candidate A) does not close GAP-03 or GAP-06/07 at all, only GAP-01 partially. The "n=1" evidentiary risk (none of B/C/D can be _proven_ correct until a second provider exists) is not resolved by this selection and cannot be — it is accepted explicitly (§6).

### 2.2 Routing

**Selected:** Layered Routing (§5D) — task routing (§5A, already the correct abstraction level per Phase 1/2) narrows first, capability routing (§5B) narrows further once Provider's Registry/Descriptor exist, policy routing (§5C) breaks remaining ties once it is activated.

**Rationale:** §5D is not a fourth alternative competing with §5A–C; Phase 3 itself presents it as their composition, and the Phase 3 Independent Review's Candidate Independence Assessment confirms this is transparently disclosed, not hidden duplication. Selecting it therefore does not "reject" §5A–C individually — it adopts all three as sequenced layers, each activated when its precondition is met: Layer 1 (task routing) is active today, requires nothing new. Layer 2 (capability routing) is frozen as the target but not yet activatable — it depends on Provider's Registry/Descriptor (§4B/§4C, selected above) existing first, exactly the "GAP-01 → GAP-02" sequencing Phase 3's Independent Review already named. Layer 3 (policy routing) is frozen as the eventual target but remains, per ADR-0003's own explicit "Out of Scope" classification, not authorized for design work until a second provider exists to make cost/latency policy meaningful — this Freeze does not reopen or override ADR-0003's scope decision. Decision Principles invoked: _remain internally consistent_ (Chain-of-Responsibility layering lets each layer be added without touching the ones below it, matching the Gateway's own "pure addition" migration precedent from ADR-0003); _support future extensibility_; _evidence-justified_ (this is the only Routing candidate the Phase 3 Independent Review found to have zero standalone independence problem, precisely because it doesn't compete with the domain's other candidates — it subsumes them).

**Supporting evidence:** Phase 3 §5 (all four candidates); Phase 2 §7 (`BrainTask` "exactly the right abstraction level"); Phase 3 Independent Review §4 (Routing D's disclosed compositional relationship) and §2 (GAP-01→GAP-02 prerequisite).

**Accepted trade-offs:** Medium complexity, rising with each layer added — accepted because the alternative (freezing only Task Routing, §5A alone) would leave GAP-02's capability/policy/fallback/hybrid sub-types permanently unaddressed, contradicting the "resolve validated gaps" principle. The Layer-2/Layer-3 activation-ordering risk (a partially-implemented router must clearly signal which layers are live) is accepted and flagged for Technical Design, not resolved here.

### 2.3 Context

**Selected:** Centralized Context Pipeline (§6A).

**Rationale:** Candidate C (Gateway-Adjacent Service) is rejected outright (§3) for direct, unresolvable tension with ADR-0003's Separation of Concerns principle — an already-approved architectural baseline this Freeze does not reopen. Between the two remaining candidates, Candidate B (Federated Pipeline) is lower-risk but, by Phase 3's own admission, "does not fully close GAP-09" and risks recreating the exact ownership-fragmentation pattern (GAP-05) that produced the two-pipeline problem in the first place — Phase 3 explicitly warns a third context mechanism could be added later without implementing B's shared contract, "exactly as `route.ts`'s server-side cap was added without coordinating with `context-manager.ts` in the first place." Selecting a candidate whose own documented weakness is "could repeat the root cause it's meant to fix" fails the _resolve one or more validated gaps_ principle more seriously than Candidate A's higher migration cost fails _avoid unnecessary complexity_ (a Discovery-phase principle, not one of this phase's eight Decision Principles). Candidate A fully closes GAP-09 by construction — one lifecycle, one owner, one place to answer "what does the model already know from earlier in this conversation." Decision Principles invoked: _resolve one or more validated gaps_ (fully, not partially); _improve long-term maintainability_ (a single reconciled owner vs. two owners coordinated by convention).

**Supporting evidence:** Phase 3 §6 (all three candidates, especially §6B's self-identified recurrence risk and §6C's self-identified boundary-violation risk); Phase 2 §8 (the token-budget-vs-message-count semantic mismatch); Phase 3 Independent Review §14 (Coupling risk assessment, confirming §6C carries the highest coupling risk in the entire document).

**Accepted trade-offs:** Medium-High complexity, concentrated in reconciling the token-budget-aware and message-count-based summarization concepts into one semantics — this reconciliation cost is explicitly accepted (§6) as a one-time investment rather than deferred, because deferring it (via Candidate B) was found to risk recreating the underlying problem rather than closing it.

### 2.4 Memory

**Selected:** Database-Side Ranking Infrastructure (§7B).

**Rationale:** GAP-08's core finding is that raising the memory cap and improving ranking quality are _architecturally coupled_ today, "because full in-process scoring is only viable below that cap" — Candidate A (in-place ranking extension) improves scoring quality without touching this coupling at all; Candidate C (Tiered Retrieval) loosens but does not remove it. Only Candidate B removes the coupling structurally, via a database-side `LIMIT` with genuine ranking that scales independently of row count — the specific capability the roadmap's own Objective names ("scale... without requiring major application refactoring"). A coupled cap-and-ranking change is exactly the kind of "major refactoring" that Objective disclaims; resolving it now, architecturally, is more consistent with that Objective than deferring it behind a cheaper interim measure. Decision Principles invoked: _resolve one or more validated gaps_ (fully, structurally, not partially); _support future extensibility_ (removes a coupling that would otherwise force a joint decision later); _evidence-justified_ (Phase 2 §9 and Phase 3 §7 both independently identify B as the only candidate that removes rather than loosens the coupling).

**Supporting evidence:** Phase 2 §9 (GAP-08, "architecturally coupled... only viable below that cap"); Phase 3 §7 (all three candidates); roadmap Objective ("scale... without requiring major application refactoring").

**Accepted trade-offs:** High complexity and a new infrastructure dependency (a vector-capable column/index) — the highest-cost candidate in the Memory domain by a wide margin, per Phase 3's own trade-off matrix. This cost is explicitly accepted (§6), not deferred, because Technical Design and Implementation Planning (Phase 4/5, not this phase) are where migration sizing and sequencing belong — this Freeze selects the architecture, not the migration order. Candidate C (Tiered Retrieval) is rejected as the frozen target but recorded (§3) as a viable interim deployment step Technical Design may choose to sequence before B, without that sequencing choice constituting a different frozen architecture.

### 2.5 Prompt

**Selected:** Centralized Prompt Composition (§8A) as the end-state, reached via the Layered migration path (§8C: internal `personas.ts` persona/protocol separation first, channel adapters second), with each channel adapter internally built as composable fragments (§8B's technique) rather than a monolithic per-channel function.

**Rationale:** The Phase 3 Independent Review's Candidate Independence Assessment found that §8C's first phase (persona/protocol separation within `personas.ts`) is a subset of what §8A's centralization already accomplishes — §8C is "more accurately characterized as an incremental, phased path toward a similar end-state than a fundamentally distinct target architecture" (Phase 3 Independent Review, Finding Min-1). This Freeze treats that finding as decision-relevant rather than a mere editorial note: it means §8A (the end-state) and §8C (the path to it) are not competing selections at all, but the same selection described at two different grains — selecting §8A as the destination and §8C as its incremental route is not a synthesis of two different architectures, it is one architecture described completely. §8B is added as a technique, not a competing destination, because it is the only candidate that activates the dormant `buildSystemPromptBlocks()` prompt-caching seam Phase 3 found "provides zero runtime value today" — building each of §8C's channel adapters as composable fragments (§8B's approach) rather than a single monolithic per-channel function preserves that caching seam's future usability while still converging on §8A's single-owner end-state. Decision Principles invoked: _resolve one or more validated gaps_ (ownership, reuse, and consistency sub-gaps of GAP-05's Prompt instance, most completely per Phase 3 §8A); _improve long-term maintainability_ (a tone-rule change propagates once, not six times); _minimize architectural coupling_ (§8C's phased sequencing avoids a single large five-call-site migration, reducing coupling risk during the transition itself).

**Supporting evidence:** Phase 3 §8 (all three candidates); Phase 3 Independent Review §4 (Finding Min-1, the A/C relationship); Phase 1 §1.4.10 (the "interleaved as undifferentiated string literals" finding §8C's first phase directly resolves); Phase 2 §10 (reuse/consistency gaps).

**Accepted trade-offs:** The largest total migration surface of any Prompt candidate (five external call sites plus internal restructuring) — accepted because it is back-loaded via §8C's phasing rather than front-loaded, and because every alternative (freezing §8B or §8C alone as the destination) leaves the six-owner fragmentation only partially resolved, per Phase 3's own comparison ("consistency less completely than Candidate A" for §8B alone).

### 2.6 Operational

**Selected:** Centralized Gateway-Level Instrumentation (§9A) as the foundation, extended with a correlation-ID contract (drawn from §9B, narrowed in scope to correlation-ID propagation only — not full per-layer structured-event emission) so that Application-Layer callers can be joined to Gateway-boundary telemetry, built on the already-present `@sentry/nextjs` dependency (§9C's lower-adoption-cost option) rather than a new observability platform.

**Rationale:** §9A alone cannot answer _why_ a request was made (which behavior flags, which persona, which channel) because that context lives in the Application Layer and, per ADR-0003, must not leak into the Gateway — §9A's own documented weakness names exactly this limitation and states the candidate "must therefore also design for" a correlation identifier. §9B alone closes GAP-04 most completely but at "the highest implementation and coordination cost" in the document, requiring every layer to independently emit structured events with no existing shared library to agree a shape against — a cost this Freeze judges disproportionate to what GAP-04 actually requires (a way to join Gateway telemetry to its cause, not full per-layer structured logging everywhere at once). The selected synthesis takes exactly the piece of §9B needed to close §9A's specific blind spot (a correlation ID, generated once per request and threaded down to the Gateway) without adopting §9B's full "every layer emits its own structured events" scope. §9C's platform-adoption principle is folded in at its lowest-cost variant: extend the Sentry dependency Emma already has (Phase 2/3 confirm it is the only observability-adjacent dependency present) rather than introduce OpenTelemetry, deferring that heavier adoption decision rather than making it here (per this phase's non-goals — no technology recommendation is finalized, only the adoption _category_ is selected). Decision Principles invoked: _minimize architectural coupling_ (Gateway-boundary instrumentation stays inside the Gateway; only a correlation ID, not business context, crosses the boundary); _evidence-justified_ (targets exactly the gap §9A's own prose identifies as its limitation, using the mechanism §9B's own prose identifies as the fix).

**Supporting evidence:** Phase 3 §9 (all three candidates, especially §9A's self-identified "why" blind spot and §9B's self-identified coordination cost); Phase 2 §11 (`package.json` confirms `@sentry/nextjs` present, no `pino`/`winston`/`@opentelemetry/*`); Phase 3 Independent Review §5 (confirms the Operational domain's trade-off ratings are internally consistent).

**Accepted trade-offs:** The correlation-ID contract still requires touching every layer that calls the Gateway (route handlers, the agent loop, utility routes) to generate and thread the identifier — a smaller version of §9B's "touches every layer" cost, accepted because it is the minimum viable version of that cost needed to close §9A's blind spot, not the maximum version §9B describes.

### 2.7 Configuration

**Selected now:** Provider-Conditional Boot Validation (§10A). **Accepted-in-principle, deferred:** Runtime Configuration Store (§10B) and Feature-Flag Layer (§10C).

**Rationale:** The Phase 3 Independent Review's Candidate Independence Assessment (Finding Ed-2) established that Configuration's three candidates are not three competing answers to one question — they address three different sub-gaps (§10A → GAP-03; §10B → the runtime-override half of GAP-13; §10C → the staged-rollout half of GAP-13). This Freeze adopts that finding directly: rather than picking one candidate and rejecting the other two, it selects §10A now — the one candidate with no dependency on any other domain's selection and the one addressing GAP-03, which Phase 2 rated "the single most structurally consequential item" in its Configuration analysis — while formally accepting §10B and §10C as correct future directions whose _activation_ is contingent on other domains actually needing them: §10B once Routing's capability/policy layers (§5B/§5C) are activated and need a place to live beyond a compile-time constant; §10C once a second provider or routing-policy rollout actually needs staging, consistent with Phase 2's own severity framing of GAP-13 ("Low, current; Medium, relative to stated future"). Decision Principles invoked: _resolve one or more validated gaps_ (GAP-03 closed now, without waiting on GAP-13's contingent relevance); _evidence-justified_ (this sequencing is the one Phase 3's Independent Review already identified as the domain's actual structure, not an invented one).

**Supporting evidence:** Phase 2 §13, §14 (GAP-03 "the most structurally consequential single item"); Phase 3 §10 (all three candidates); Phase 3 Independent Review §4 (Finding Ed-2, the domain's structural difference from every other domain's candidate set) and §8 (Finding Min-3, noting GAP-03/GAP-13's shared "n=1" root cause with Provider/Routing — carried forward into §6 of this document as an accepted risk).

**Accepted trade-offs:** §10A requires "which provider(s) are configured" to be a queryable fact, supplied by the selected Provider Registry (§4B, §2.1 above) — this dependency is accepted, not treated as a gap in §10A's own readiness, because the Provider domain's selection already satisfies it.

### 2.8 Extension

**Selected:** Static boundary enforcement (§11.1B, a lint/module-boundary rule) + Tooling-enforced dependency inversion (§11.2C, a lint rule against direct concrete-module imports where an interface exists) + Workflows as an extension of the existing agent loop, no new abstraction (§11.4A).

**Rationale (§11.1, Boundary Enforcement):** §11.1A (convention only, status quo) is rejected — Phase 3's own Risk Assessment (as corrected by this initiative's Independent Review, Finding M-2) names it "the only candidate in this document whose risk is not offset by any compensating property." §11.1C (runtime assertion) is not selected as the primary mechanism because it fires later in the feedback loop than a CI-time lint failure and adds runtime overhead to every guarded boundary; it is recorded (§3) as a defense-in-depth addition Technical Design may layer on top of §11.1B without conflict, not a rejected direction. §11.1B is selected as primary: CI-time, zero runtime cost, and the exact mechanism Phase 2 itself named as a suggested example for closing GAP-10.

**Rationale (§11.2, Dependency Inversion):** §11.2A (module-by-module, no forcing function) is rejected on direct evidence — GAP-11 itself demonstrates this exact approach has been available since at least `cost-gate.ts`'s introduction and has been adopted in only two places, meaning "no forcing function" is not a hypothetical risk but an already-observed failure mode. §11.2B (documented convention alone) reduces but does not eliminate that risk, per Phase 3's own text ("a documentation-only convention is subject to the identical 'holds by convention, not a barrier' pattern"). §11.2C (tooling-enforced) is selected because it is the only candidate that closes the exact failure mode GAP-11 already demonstrated, and because selecting the same enforcement philosophy (tooling, not convention) as §11.1 satisfies the _remain internally consistent_ Decision Principle directly — this domain should not enforce boundaries with tooling while enforcing DI with a document alone.

**Rationale (§11.4, Workflow Concept):** §11.4B (a distinct orchestration layer) is rejected — Phase 2 and Phase 3 both found zero evidence of a distinct "workflow" need beyond the already-shipped, already-Gateway-integrated agent loop, and building for an unevidenced need directly contradicts _preserve validated architectural strengths_ (the agent loop is one of the roadmap's own two hardest-sounding extension items, already done — Phase 2 §14). §11.4A is selected: it preserves the proven mechanism and defers any new abstraction until a concrete multi-agent/multi-session need is actually evidenced.

**Supporting evidence:** Phase 2 §16 (GAP-10, GAP-11 descriptions); Phase 3 §11.1–§11.2, §11.4 (all candidates); Phase 3's Risk Assessment as corrected by the Independent Review (Finding M-2, the added risk bullets for §11.1/§11.2/§11.4); Phase 2 §14 (agent loop/tool-calling already proven).

**Accepted trade-offs:** §11.1B/§11.2C's coverage is "only as complete as the rules authored" (per Phase 3's own §11.1B weakness) — a new boundary (e.g., the future Context or Prompt Pipelines selected above) needs its own rule written; this is accepted as an ongoing governance responsibility, not a one-time cost, and is named explicitly in the ADR Inventory (§5) and Risk Register (§6).

### 2.9 Governance

**Selected:** A documented, versioned Extension Model artifact (§11.3A), with §11.2C's lint rule (selected above, §2.8) serving as its forcing function against drift.

**Rationale:** §11.3B (scaffolding/template mechanism) is rejected as the primary artifact — it is higher complexity and addresses only the mechanical/boilerplate parts of extension, not the judgment calls (e.g., "should this provider's quirk be normalized in the adapter or exposed to callers") a governed policy document is meant to guide, per Phase 3's own comparison. §11.3A's self-identified risk — a document can drift from the code it describes without a forcing function, precisely the failure mode GAP-14 already demonstrates for ADR-0003's own stale header — is directly answered by pairing it with §11.2C's tooling: where the Extension Model document states a DI or boundary expectation, §11.2C's lint rule makes that expectation checkable rather than merely written down, closing the specific gap between "documented" and "enforced" that this initiative has now observed twice (once in ADR-0003's header, once in GAP-11's own DI-adoption history). Decision Principles invoked: _resolve one or more validated gaps_ (GAP-12 directly); _evidence-justified_ (the pairing directly answers a risk this same initiative has already demonstrated twice, not a hypothetical one).

**Supporting evidence:** Phase 2 §16 (GAP-12, GAP-14); Phase 3 §11.3 (both candidates); ADR-0003 line 6 ("Implementation: None yet," the GAP-14 drift this pairing is designed to avoid repeating).

**Accepted trade-offs:** A lint rule can only enforce what it is written to check — it cannot verify a judgment call (e.g., "was this the _right_ place to normalize this provider quirk") the way a human reviewer consulting the Extension Model document can. This is accepted: the two mechanisms are complementary, not substitutes, and neither is expected to fully replace the other's coverage.

**Governance action item (not an architectural decision):** ADR-0003's header ("Implementation: None yet") should be corrected to reflect Phase 7B's completed state as part of the ADR work this Freeze's Decision Inventory (§5) triggers. This Freeze does not edit ADR-0003 itself — GAP-14 is a documentation-accuracy issue on a file outside this phase's own deliverable, and correcting it is recorded here as an action item for the ADR-creation phase, not performed as a side effect of this document.

## 6. Risk Acceptance Register

Every risk named across Phase 3 and its Independent Review, classified against the selected architectures as Resolved, Accepted, or Deferred.

### Resolved (the selected architecture removes the risk)

- The absence of any provider-selection mechanism (part of GAP-01/GAP-03's underlying problem) is resolved by the Provider Registry (§2.1) — a queryable selection surface now exists where none did.
- The memory cap-ranking architectural coupling (GAP-08) is resolved by Database-Side Ranking (§2.4) — the two concerns become independently adjustable.

- The "no forcing function" DI-adoption risk demonstrated by GAP-11 is resolved by Tooling-Enforced DI (§2.8) — adoption is no longer optional-and-observed-to-stall.

### Accepted (the risk is real, cannot be architecturally eliminated, and this Freeze proceeds anyway)

- **The "n=1" evidentiary risk** (Phase 3 §14; Phase 3 Independent Review Finding Min-3, which additionally identified this risk as extending to Configuration's GAP-03/GAP-13-addressing candidates, not just Provider/Routing as Phase 3's own text stated). None of the Provider, Routing, or Configuration selections above can be _proven_ correct until a second provider actually exists — this is accepted explicitly as a property of the current single-provider system, not a defect in this Freeze's selections, and any future selection would inherit the identical constraint.
- **Memory's schema-migration and new infrastructure-dependency cost** (§2.4) — accepted as the price of structurally resolving GAP-08 rather than merely loosening it; sizing and sequencing that cost is Phase 4/5's job, not this Freeze's.
- **Context's reconciliation cost** (unifying token-budget-aware and message-count-based summarization semantics, §2.3) — accepted as a one-time investment in exchange for fully closing GAP-09 rather than partially closing it.
- **Operational's correlation-ID propagation coordination cost** (§2.6) — smaller than §9B's full scope, but still requires touching every Gateway-calling layer; accepted as the minimum viable cost to close §9A's "why" blind spot.
- **The Prompt migration's back-loaded gap-closure** (§2.5) — the six-owner fragmentation persists until every channel adapter is built; accepted because the alternative (a single large migration) carries higher transitional coupling risk, per the _minimize architectural coupling_ principle.
- **Boundary/DI enforcement's "only as complete as the rules authored" limitation** (§2.8, §2.9) — accepted as an ongoing governance responsibility rather than a one-time gap; every new component this Freeze introduces (Context Pipeline, Prompt Pipeline, Provider Registry) will need its own enforcement rule authored, not automatically inherited.

### Deferred (explicitly not resolved now, activation contingent on a stated future condition)

- **Routing Layer 3 (Policy Routing, §5C)** — deferred until a second provider exists and ADR-0003's own "Out of Scope" classification for cost/latency routing is formally revisited (not proposed here).
- **Configuration's Runtime Configuration Store (§10B)** — deferred until Routing's capability/policy layers are activated and need a runtime home.

- **Extension's runtime-assertion boundary check (§11.1C)** — deferred as an optional defense-in-depth addition to the selected lint-based enforcement, not a required one.
